### A05-1  `!=` is not reflected onto a right-hand instance's `_ne_` (only `_eq_` is reflected)
- **Severity:** Low
- **Kind:** correctness-bug
- **Location:** src/kirito/runtime.hpp:2087-2101 (applyBinaryOp Eq/Ne block) + kiEquals 1540-1549
- **What:** When an instance is the LEFT operand, `obj != x` dispatches to the class's `_ne_`
  (via `inst->findMethod(binOpMethod(Ne))`). But when the instance is the RIGHT operand
  (`x != obj`, x non-instance), the Ne branch's instance-check `l.kind()==Instance` is false, so it
  falls through to `kiEquals(lhs,rhs)` which reflects only via `_eq_` (viaEq), then negates. The
  right operand's `_ne_` is never consulted. `==` reflects correctly (kiEquals viaEq calls the
  right instance's `_eq_`), so `_eq_` is symmetric but `_ne_` is not.
- **Repro:** class C with `_eq_ -> True` and `_ne_ -> True` (deliberately inconsistent):
  `c != 5` => True (calls `_ne_`); `5 != c` => False (reflects to `_eq_`, negates). Non-commutative.
  (ne2.ki, confirmed on release/ki-linux-x64.)
- **Confidence:** confirmed (reproduced)
- **Fix sketch:** In applyBinaryOp's Eq/Ne block, when the LEFT operand is non-instance and the
  RIGHT operand is an instance defining `_ne_`, dispatch `rhs._ne_(lhs)` before falling back to the
  `!_eq_` path — mirror the left-operand `_ne_` handling. Or document that `_ne_` is intentionally
  derived from `_eq_` when the instance is on the right (but then the left-operand path shouldn't
  prefer a standalone `_ne_` either — pick one policy for symmetry).

### A05-2  `makeMethod` kwarg binder silently substitutes `None` for an omitted required arg (dict.get returns wrong value)
- **Severity:** Medium
- **Kind:** correctness-bug / missing-guard
- **Location:** src/kirito/native.hpp:168-194 (makeMethod NativeFnKw binder), esp. the gap-fill at
  line 189 `if (!set[i]) slots[i] = v.none();`
- **What:** The keyword binder for built-in type methods has no concept of *required* parameters. When
  a later parameter is supplied by keyword and an earlier one is omitted, every unset slot below the
  highest set index is filled with `None` and passed to the impl. For methods where the impl
  type-checks its args, this surfaces as a misleading error; for `Dict.get(key, default)` (and
  `Dict.pop(key, default)`, `Dict.setdefault`), `None` is a legal key, so `d.get(default = 999)`
  silently looks up the key `None`, misses, and returns the default — no error at all. The user
  omitted a required argument and got a wrong-but-plausible answer.
- **Repro:** `var d = {"a": 1}` then `d.get(default = 999)` => `999` (should raise "missing required
  argument 'key'"). `d.pop(default = 42)` => `42`. **Worse — `d.setdefault(default = 7)` silently
  MUTATES the dict, inserting `{None: 7}`** and returning 7. With a planted `None` key, get/pop return
  that key's value. (mm4.ki, mm7.ki, confirmed on release/ki-linux-x64.) Contrast signatured natives:
  `bindArgs` (function.hpp:95-100) correctly raises "missing required argument". The two kwarg binders
  disagree on required-arg handling.
- **Confidence:** confirmed (reproduced)
- **Fix sketch:** Give `makeMethod` per-param required/default info (like `NativeParam`), or at minimum
  raise "<name>() missing required argument '<param>'" when a slot below `outlen` is unset for a
  parameter the method requires. Simplest safe fix: when a gap is filled with `None` because a keyword
  supplied a *later* param, raise instead of silently defaulting — the method's positional protocol
  never permits a real hole. (Secondary: even the type-checked methods give misleading messages like
  "insert expects an Integer" instead of "missing required argument 'index'".)

### A05-3  Reflected comparison/arithmetic operators are not dispatched to a right-hand instance (only `==`/`_eq_` reflect)
- **Severity:** Low
- **Kind:** correctness-bug / doc
- **Location:** src/kirito/runtime.hpp:2074-2107 (applyBinaryOp), numericBinary guard 156-162
- **What:** `==` reflects onto a right-hand instance's `_eq_` (via kiEquals), but `<`/`<=`/`>`/`>=` and
  `+`/`-`/`*`/`/`/... do NOT reflect. `1 < obj`, `3 + obj` throw "unsupported operand type" even when
  `obj`'s class defines `_lt_`/`_gt_`/`_add_`. This is an inconsistency (one operator family reflects,
  the rest don't). The spec doesn't promise reflected operators, so this is borderline doc, but the
  partial implementation (only `==`) is surprising.
- **Repro:** class Money with `_add_`; `Money(5) + 3` => 8 but `3 + Money(5)` => "unsupported operand
  type 'Money' for arithmetic with 'Integer'". (ra.ki, confirmed.)
- **Confidence:** confirmed (reproduced)
- **Fix sketch:** Either document that reflected operators are unsupported except `_eq_`/`_ne_` (and
  drop `==` reflection for consistency), or add reflected dispatch: when the left operand's `binary`
  fails/left is a non-instance and the right is an instance defining the reflected dunder, try the
  right operand. Given the spec silence, documenting the limitation is lowest-risk.

### A05-4  `_bool_`/`truthy()` errors lose line:col span in a `while`/`if` condition (reports 0:0)
- **Severity:** Low
- **Kind:** doc / missing-guard (diagnostics)
- **Location:** src/kirito/runtime.hpp:1701-1717 (InstanceValue::truthy throws without span) +
  the JumpIfFalse/condition path in bytecode_vm.hpp (does not wrap truthy() in located()).
- **What:** A `_bool_` that returns a non-Bool throws `'X'._bool_ must return a Bool`. When the
  instance is a `switch`/set/comparison operand the error carries a proper span (e.g. `5:9`), but as
  a `while`/`if` condition it reports `sp2.ki:0:0` — line/col lost. Inconsistent, and unhelpful for
  locating the offending condition. (Same class of issue would affect any error escaping `truthy()`.)
- **Repro:** `class Bad: var _bool_ = Function(self): return 5` then `while Bad():` => `sp2.ki:0:0:
  error: ...`. Compare `_hash_` in a set literal, which reports the real line. (sp2.ki, confirmed.)
- **Confidence:** confirmed (reproduced)
- **Fix sketch:** Wrap the truthiness evaluation in the JumpIfFalse (and JumpIfTrue / and/or) opcode
  handlers in `located(in.span, …)` so an error from `_bool_` is tagged with the condition's span.
  (Likely belongs to A04 bytecode_vm, but the throw site is here.)

## Test gaps (C++ and .ki)
_(methods/attributes/arguments with no or weak coverage; adversarial/edge cases missing)_

- **`makeMethod` omitted-required-arg (A05-2):** no test exercises a built-in method call that omits
  a required leading arg while supplying a later one by keyword (`d.get(default=X)`,
  `d.pop(default=X)`, `s.replace(count=1)`, `xs.insert(item=X)`). Add `.ki` + `.experr` cases asserting
  a "missing required argument" error (once fixed) — today several silently substitute None.
- **`!=` reflection (A05-1):** no test for `x != obj` where obj is the right operand and defines a
  standalone `_ne_`. Add a `.ki` asserting commutativity of `!=` (or documenting the derived-from-`_eq_`
  policy).
- **Reflected operators (A05-3):** no test pins down that `3 + obj` / `1 < obj` are (currently)
  unsupported when only obj defines the dunder. Add either a `.experr` documenting the limitation or a
  positive test once reflection is added.
- **Privacy — subclass method accessing a *superclass-instance's* private:** `checkPrivateAccess`
  requires `isInstanceOf(obj, currentClass)`, so a B-method reading `other._x` (where `_x` is defined
  in the shared base A and `other` is a bare A instance) is DENIED (priv3.ki). This matches C++
  `protected`-through-base-ref semantics but the spec's "privacy is per class *chain*" wording could be
  read to allow it. No test covers this direction — add one and clarify the doc which way it's meant.
- **`_bool_`/`_hash_`/`_eq_`/`_len_` return-type violations:** `_hash_` non-Integer and `_bool_`
  non-Bool are handled; verify `_len_` negative/non-Integer, `_iter_` non-iterable, `_contains_`
  non-bool return, and `_eq_` returning a non-bool truthy value all have `.ki` regression tests.
- **Bool as an ordering/sort key:** `sorted([True, False])` throws "cannot order 'Bool' and 'Bool'"
  and `True < False` throws. Intentional (Bool is a distinct non-numeric type) but undocumented and
  untested — worth an explicit test so it isn't "fixed" accidentally.
- **Slice INT64_MIN/MAX step:** `xs[::INT64_MIN]` and `xs[::INT64_MAX]` work (verified ASan-clean) but
  have no regression test — add them (the count-driven loop in sliceIndices is the guard against the
  `i += step` overflow it documents).
- **`**` two's-complement wrap:** `2**64==0`, `2**63==INT64_MIN`, `10**19` wrap, `(-1)**9999999999`
  verified by probe but ensure a `.ki` golden covers Integer-pow wraparound (distinct from the
  add/sub/mul wrap tests).

## DRY / single-source-of-truth notes
_(logic duplicated across files that should be unified)_

- **Keyword-argument binding is implemented FOUR times with divergent semantics** — the root cause of
  A05-2. The "match named args to params by linear name search, reject unknown/duplicate, fill
  missing" logic is duplicated in:
  1. `NativeFunction::bindArgs` — function.hpp:78-102 (errors on missing required)
  2. `makeMethod`'s NativeFnKw closure — native.hpp:170-192 (**gap-fills missing with None — no
     required check**, the A05-2 bug)
  3. `KiFunction::callFull` — runtime.hpp:1977-2012 (errors on missing; also does default eval +
     annotation enforcement)
  4. `InstanceValue::callKw` — runtime.hpp:1793-1809 (delegates the merge to callFull)
  These three independent implementations disagree on the missing-required-arg policy. Extract a single
  `bindNamed(paramNames, hasDefault[], positional, named) -> {slots, errors}` helper (parameterized on
  whether missing-without-default is an error) so all callers share one contract. This both fixes
  A05-2 and removes ~40 lines of triplicated loop.
- **Numeric compare logic** exists both inline in `numericBinary`'s Integer×Integer switch
  (runtime.hpp:176-179) and in `numericCompare` (131-145). Deliberate (hot-path fast case) and
  behaviourally identical, so acceptable, but a comment cross-reference would help keep them in sync if
  the exact-compare rules ever change.
- **Negative-index resolution** (`if (i<0) i+=n; bounds-check`) appears in `sequenceIndex`
  (runtime.hpp:369-378) and again open-coded in `ListVal::pop` (484-491) and Bytes/String getItem.
  Minor; `sequenceIndex` could be reused by pop.
