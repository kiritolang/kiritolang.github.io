### A05-1  `!=` is not reflected onto a right-hand instance's `_ne_` (only `_eq_` is reflected)
- **Severity:** Low
- **Kind:** correctness-bug
- **Location:** src/kirito/runtime.hpp:2087-2101 (applyBinaryOp Eq/Ne block) + kiEquals 1540-1549
- **What:** When an instance is the LEFT operand, `obj != x` dispatches to the class's `_ne_`
  (via `inst->findMethod(binOpMethod(Ne))`). But when the instance is the RIGHT operand
  (`x != obj`, x non-instance), the Ne branch's instance-check `l.kind()==Instance` is false, so it
  falls through to `kiEquals(lhs,rhs)` which reflects only via `_eq_` (viaEq), then negates. The
  right operand's `_ne_` is never consulted. `==` reflects correctly (kiEquals viaEq calls the
  right instance's `_eq_`), so `_eq_` is symmetric but `_ne_` is not.
- **Repro:** class C with `_eq_ -> True` and `_ne_ -> True` (deliberately inconsistent):
  `c != 5` => True (calls `_ne_`); `5 != c` => False (reflects to `_eq_`, negates). Non-commutative.
  (ne2.ki, confirmed on release/ki-linux-x64.)
- **Confidence:** confirmed (reproduced)
- **Fix sketch:** In applyBinaryOp's Eq/Ne block, when the LEFT operand is non-instance and the
  RIGHT operand is an instance defining `_ne_`, dispatch `rhs._ne_(lhs)` before falling back to the
  `!_eq_` path — mirror the left-operand `_ne_` handling. Or document that `_ne_` is intentionally
  derived from `_eq_` when the instance is on the right (but then the left-operand path shouldn't
  prefer a standalone `_ne_` either — pick one policy for symmetry).

### A05-2  `makeMethod` kwarg binder silently substitutes `None` for an omitted required arg (dict.get returns wrong value)
- **Severity:** Medium
- **Kind:** correctness-bug / missing-guard
- **Location:** src/kirito/native.hpp:168-194 (makeMethod NativeFnKw binder), esp. the gap-fill at
  line 189 `if (!set[i]) slots[i] = v.none();`
- **What:** The keyword binder for built-in type methods has no concept of *required* parameters. When
  a later parameter is supplied by keyword and an earlier one is omitted, every unset slot below the
  highest set index is filled with `None` and passed to the impl. For methods where the impl
  type-checks its args, this surfaces as a misleading error; for `Dict.get(key, default)` (and
  `Dict.pop(key, default)`, `Dict.setdefault`), `None` is a legal key, so `d.get(default = 999)`
  silently looks up the key `None`, misses, and returns the default — no error at all. The user
  omitted a required argument and got a wrong-but-plausible answer.
- **Repro:** `var d = {"a": 1}` then `d.get(default = 999)` => `999` (should raise "missing required
  argument 'key'"). With a planted `None` key it returns that key's value. (mm4.ki, confirmed on
  release/ki-linux-x64.) Contrast signatured natives: `bindArgs` (function.hpp:95-100) correctly
  raises "missing required argument". The two kwarg binders disagree on required-arg handling.
- **Confidence:** confirmed (reproduced)
- **Fix sketch:** Give `makeMethod` per-param required/default info (like `NativeParam`), or at minimum
  raise "<name>() missing required argument '<param>'" when a slot below `outlen` is unset for a
  parameter the method requires. Simplest safe fix: when a gap is filled with `None` because a keyword
  supplied a *later* param, raise instead of silently defaulting — the method's positional protocol
  never permits a real hole. (Secondary: even the type-checked methods give misleading messages like
  "insert expects an Integer" instead of "missing required argument 'index'".)

### A05-3  Reflected comparison/arithmetic operators are not dispatched to a right-hand instance (only `==`/`_eq_` reflect)
- **Severity:** Low
- **Kind:** correctness-bug / doc
- **Location:** src/kirito/runtime.hpp:2074-2107 (applyBinaryOp), numericBinary guard 156-162
- **What:** `==` reflects onto a right-hand instance's `_eq_` (via kiEquals), but `<`/`<=`/`>`/`>=` and
  `+`/`-`/`*`/`/`/... do NOT reflect. `1 < obj`, `3 + obj` throw "unsupported operand type" even when
  `obj`'s class defines `_lt_`/`_gt_`/`_add_`. This is an inconsistency (one operator family reflects,
  the rest don't). The spec doesn't promise reflected operators, so this is borderline doc, but the
  partial implementation (only `==`) is surprising.
- **Repro:** class Money with `_add_`; `Money(5) + 3` => 8 but `3 + Money(5)` => "unsupported operand
  type 'Money' for arithmetic with 'Integer'". (ra.ki, confirmed.)
- **Confidence:** confirmed (reproduced)
- **Fix sketch:** Either document that reflected operators are unsupported except `_eq_`/`_ne_` (and
  drop `==` reflection for consistency), or add reflected dispatch: when the left operand's `binary`
  fails/left is a non-instance and the right is an instance defining the reflected dunder, try the
  right operand. Given the spec silence, documenting the limitation is lowest-risk.
