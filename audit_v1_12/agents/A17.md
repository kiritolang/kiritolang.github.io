## Findings

### A17-1  DateTime `_setstate_` is a public in-place mutator that breaks immutability + the hash-key contract
- **Severity:** Medium
- **Kind:** correctness-bug / missing-guard
- **Location:** src/kirito/stdlib_time.hpp:261-270 (`_setstate_` binding) — cf. class doc "immutable once built" (line 133) and the hashable/equality contract (lines 164-172)
- **What:** DateTime advertises itself as *immutable once built* and *hashable by instant* (usable
  as a Dict/Set key; equal instant ⇒ equal hash). But the serialization hook `_setstate_(epoch)` is
  exposed as an ordinary callable method on **any live DateTime** and mutates the instance in place
  (`setEpoch` on `self`). Calling it on a DateTime that is already a Set/Dict key silently corrupts
  the container: the element is now stored under a stale hash bucket, so membership/lookup fail for
  the object that is still physically inside the set. This violates the hashable-key invariant that
  the whole "DateTime as a Dict/Set key" feature rests on, and contradicts the documented immutability.
- **Repro / failure scenario:**
  ```
  var t = import("time")
  var d = t.make(2020, 1, 1)
  var s = {d}
  print(d in s)        # True
  d._setstate_(0)      # mutate a live Set key in place  -> now 1970-01-01
  print(d in s)        # False  (element is still the sole member: len(s)==1, iterating yields it)
  print(t.make(1970,1,1) in s)  # False — the new instant isn't findable either
  ```
  Set is corrupted: `len(s)==1`, iteration yields the mutated element, yet neither the object itself
  nor a fresh equal-instant DateTime can be found by `in`.
- **Confidence:** confirmed (reproduced on `release/ki-linux-x64`; ASan-clean, so it's a logic bug, not UB).
- **Fix sketch:** Make `_setstate_` one-shot / non-mutating-of-live-keys. Cleanest: only allow it to
  set the epoch while the instance is still in its freshly-deserialized default state (guard: throw/
  no-op if `epoch != 0` already, or track an `initialized_` flag set by any public constructor), so
  the deserializer's `alloc(empty)` → `_setstate_(e)` path still works but a user can't re-home an
  established value. (Same shape as any other native `_setstate_`, but DateTime is special because it
  is *hashable* — the mutation invalidates container placement.) Note the existing test comment in
  `tools/tests/unit/test_sys_time_deep.cpp:104-105` explicitly (and incorrectly) asserts "there is no
  Kirito-level path to hand a malformed state to a freshly-deserialized DateTime" — this repro is
  exactly that path and should become a regression test.

### A17-2  `time.sleep` has no upper bound — a huge finite value blocks indefinitely
- **Severity:** Nit
- **Kind:** resource-exhaustion (DoS-in-script)
- **Location:** src/kirito/stdlib_time.hpp:310-314
- **What:** `sleep(secs)` only guards `secs > 0`; `sleep(1e18)` sleeps for ~3e10 years (effectively a
  hang) rather than throwing. Not a memory/UB bug (ASan-clean; NaN/inf/negative/zero all return
  immediately and correctly), and Python's `time.sleep` is likewise unbounded, so this is a nit —
  but a defensive upper clamp (throw on an absurd duration) would match the "guards THROW not hang"
  theme and the bounded-`range`/repetition guards elsewhere.
- **Repro:** `time.sleep(1e18)` — never returns (killed by a 4s `timeout` in probing).
- **Confidence:** confirmed.
- **Fix sketch:** reject `secs` above a sane ceiling (e.g. a few years) with a clear error, or clamp.

### A17-3  `format()` grows buffer to 256 KB when a valid format legitimately expands to empty
- **Severity:** Nit
- **Kind:** perf / minor
- **Location:** src/kirito/stdlib_time.hpp:237-252
- **What:** `strftime` returns 0 both when the buffer is too small AND when the result is
  legitimately empty. The retry loop treats every `n == 0` as "didn't fit" and quadruples the buffer
  up to 6 times (256 → ~262 KB) before giving up and returning "". For a format that genuinely
  expands to empty (e.g. `%Z` with no TZ, or a locale where `%p` is empty), this allocates up to
  ~262 KB and does 6 wasted `strftime` calls to produce the correct `""`. Result is correct; only
  wasteful. The code comment already acknowledges the ambiguity.
- **Confidence:** plausible (reasoned; correct output confirmed, the waste is on the empty-legit path).
- **Fix sketch:** distinguish the two cases — if the buffer is already large (e.g. ≥ some threshold)
  and `strftime` still returns 0, accept `""` and stop; or probe with a sentinel.

## Test gaps (C++ and .ki)
- **`_setstate_` live-mutation (A17-1) is untested and mis-documented.** Existing tests only cover
  `d._setstate_()` (no-arg → throws, OOB guard) and non-Integer state. None call
  `d._setstate_(validEpoch)` on an established value, and `test_sys_time_deep.cpp:104-105` asserts
  the path doesn't exist. Add a regression: put a DateTime in a Set, call `_setstate_`, assert the
  container invariant (and, after the fix, assert `_setstate_` on an initialized value throws / no-ops).
- **`add`/`sub` float-argument truncation** is under-tested: `d.add(3600.9)` truncates to 3600s
  (confirmed via `toInt64Checked`). No test pins the truncation direction for positive/negative
  fractional deltas; add one so the float→seconds contract is explicit.
- **`sleep` boundary** (negative/NaN/inf/zero return immediately) has no `.ki` assertion; add cheap
  ones (`sleep(-1)`, `sleep(nan)` return None without delay).
- Otherwise coverage is strong: `make` rollover (month 13 / day 32 / hour 25) is tested in
  `r6_time.ki` + `audit_time.ki`; strptime strictness (out-of-range, trailing input, literal
  mismatch, unsupported directive, %% ) in `r10_datasys.ki`/`deep_system.ki`/`audit_time.ki`;
  equality+hash + Dict/Set key + serialize/dump round-trip in `audit_time.ki`.
