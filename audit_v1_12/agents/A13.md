# A13 — matrix / complex

**Files in scope:** stdlib_matrix.hpp, stdlib_complex.hpp
**Status:** DONE (read-only audit + behavioural probes on release/ki-linux-x64 + dist/debug-asan)
**Bottom line:** the wrapper layer is in good shape. Every explicit guard I probed fires as a
catchable `KiritoError` (never a crash): non-square det/inverse/trace throw, singular inverse throws
(not NaN), dimension mismatch in +,-,* throws clearly, `*` is matmul not element-wise, complex
ordering `< <= > >=` throws, all domain singularities (log/log10 of 0, atanh(±1), 0 to a
negative/complex power) throw while valid off-axis points (sqrt(-1)=i, log(-1)=iπ, asin(2)) do not,
huge-alloc guard (16M elems) throws `Matrix too large`, and the `== exact / .compare tolerant`
boundary rule holds bit-for-bit for Complex, Matrix, and ComplexMatrix (incl. NaN≠NaN, inf==inf,
Hermitian dot conjugation). No Critical/High. Findings are one Medium (reflection asymmetry, shared
with the Wave-1 runtime theme), plus DRY/consistency Lows and test gaps.

## Findings

### A13-1  Scalar-on-left is not dispatched: `2*M`, `2+z`, `i*C` throw while the mirror works
- **Severity:** Medium
- **Kind:** correctness-bug (usability / reflected-operator gap)
- **Location:** stdlib_matrix.hpp:117 (MatrixVal::binary), stdlib_complex.hpp:118
  (ComplexVal::binary) & :294 (ComplexMatrixVal::binary) — really the numeric dispatch in runtime.hpp
- **What:** These native numeric types only handle the operation when they are the LEFT operand.
  With a scalar/other type on the left the VM dispatches to the scalar's `binary`, which has no
  knowledge of Matrix/Complex, so a commutative op fails one-sided:
  - `M * 2` → ok, but `2 * M` → `unsupported operand type 'Matrix' for arithmetic with 'Integer'`
  - `z + 2` → ok, but `2 + z` → `unsupported operand type 'Complex' for arithmetic with 'Integer'`
  - `C * complex.i` → ok, but `complex.i * C` → `Complex arithmetic expects a Complex or a number`
  `2*M`, `2+z`, scalar·matrix and `Complex*ComplexMatrix` are all standard mathematical notation, so
  this is a real ergonomic wart specific to the numeric stack (people write `2*M` constantly).
- **Repro:** `(2 * matrix.Matrix([[1,2],[3,4]]))[0,0]` throws; `(matrix.Matrix([[1,2],[3,4]])*2)[0,0]`
  is `2.0`. Same for `2 + complex.of(1,1)` and `complex.i * complex.Matrix([[complex.of(1,1)]])`.
- **Confidence:** confirmed (reproduced on release binary)
- **Fix sketch:** this is the same reflected-operator asymmetry the Wave-1 report flagged as a
  cross-cutting theme (A05-1/A05-3). The right fix is in the runtime numeric-dispatch fallback: when
  the left operand's `binary` rejects an rhs it doesn't understand, retry a reflected dispatch on the
  rhs for commutative ops (Add/Mul) — or, minimally, teach `IntVal`/`FloatVal` binary to delegate to
  the rhs's `binary` when the rhs is a NativeClass. Coordinate with A05 so there is one rule.

### A13-2  MatrixVal and ComplexMatrixVal duplicate ~90% of their surface
- **Severity:** Low
- **Kind:** DRY
- **Location:** stdlib_matrix.hpp:30-385 vs stdlib_complex.hpp:201-658
- **What:** The real and complex matrix wrappers are near-identical, differing only in the element
  type (`double` vs `cdouble`) and a handful of element-formatting/coercion calls. Duplicated
  verbatim (modulo T): `rows/cols/data/at/isVector`, `str`, `equals` (exact loop), `inspectMembers`,
  `getItem` (1→row / 2→element), `setItem`, `indexOf` (Integer + non-negative bounds), the `getAttr`
  method set `rows/cols/shape/compare/get/set/row/apply/_getstate_/_setstate_`, the `make`/`fromTensor`
  size-guard helpers (identical `kMax*Elems = 16M`), and the module factories `zeros/ones/identity/
  vector` (identical non-negative checks). Two copies to keep in sync: any fix to bounds, guards, or
  the serialization format has to be applied twice (and can silently drift).
- **Repro / failure scenario:** N/A (maintenance hazard, not a runtime bug).
- **Confidence:** confirmed (side-by-side read)
- **Fix sketch:** extract a `MatrixBase<T>` (or a CRTP `NativeMatrix<Derived, T>`) carrying the
  shape/index/guard/serialize machinery; the real and complex wrappers supply only the element
  codec (float↔double vs `cpx::asComplex`/`complexToString`) and the type-name. The tensor engine is
  already generic in T, so the wrapper can be too.

### A13-3  transpose/trace: engine call for real Matrix, hand-rolled loop for ComplexMatrix
- **Severity:** Low
- **Kind:** DRY / inconsistency
- **Location:** stdlib_matrix.hpp:217 (`tensor::transpose`), :232 (`tensor::trace`) vs
  stdlib_complex.hpp:405-411 (manual transpose loop), :419-425 (manual hermitian), :428-434 (manual
  trace loop)
- **What:** The real Matrix routes transpose and trace through the shared tensor engine
  (`tensor::transpose`, `tensor::trace`); the ComplexMatrix reimplements the same two operations with
  its own nested loops even though `tensor::Tensor<cdouble>` already supports them (the engine is
  generic in T, as determinant/inverse already use via `tensor::determinant`/`tensor::inverse` for
  both). Two implementations of one operation is exactly the drift risk A13-2 describes, in miniature.
- **Confidence:** confirmed (read)
- **Fix sketch:** use `tensor::transpose(t)` / `tensor::trace(t)` in the complex wrapper too (keep the
  hand-rolled loops only for `conjugate`/`hermitian`, which the engine has no direct op for — or add
  a `conj`-transpose helper to the engine and share it). Reduces the complex wrapper and unifies
  behaviour/edge handling with the real path.

### A13-4  MatrixVal::apply does not RootScope its per-element fn argument (ComplexMatrix does)
- **Severity:** Low
- **Kind:** missing-guard (latent GC-rooting inconsistency)
- **Location:** stdlib_matrix.hpp:237-247 (`apply`) vs stdlib_complex.hpp:436-447
- **What:** `MatrixVal::apply` builds each callback argument as `std::array<Handle,1> args{vm.makeFloat(...)}`
  and calls the user fn without a `RootScope`, whereas `ComplexMatrixVal::apply` wraps the fresh arg
  in `RootScope rs(vm)`. Today the real path is safe (the freshly-alloc'd `out` matrix holds raw
  `double`s, not handles, so a mid-apply GC can't dangle it, and `applyCall` roots the arg span on
  entry), so this is not an observed bug — a 20×20 apply with a heavily-allocating callback ran clean
  under ASan. But it is an inconsistent rooting discipline between two copies of the same routine and
  a trap for a future refactor (e.g. if `apply` ever accumulates handles). Flagging so the DRY unify
  (A13-2) also normalizes the rooting.
- **Repro:** did not reproduce a fault (ASan-clean); reasoned inconsistency only.
- **Confidence:** plausible (reasoned; no crash observed)
- **Fix sketch:** add `RootScope rs(vm);` around the arg construction in the real `apply` to match the
  complex one, or resolve both when factoring out `MatrixBase<T>::apply`.

### A13-5  Generic operator-unsupported message for matrix ordering (vs Complex's precise one)
- **Severity:** Nit
- **Kind:** doc / diagnostics
- **Location:** stdlib_matrix.hpp:137, stdlib_complex.hpp:315
- **What:** `M < M` reports `Matrix does not support this operator` and `C < C`
  `ComplexMatrix does not support this operator` — the operator name isn't surfaced, and unlike
  `ComplexVal` (which says `complex numbers are not ordered (no <, <=, >, >=)`) it doesn't tell the
  user *why*. Minor, but the scalar Complex message is a nice template to match for matrices.
- **Confidence:** confirmed
- **Fix sketch:** intercept Lt/Le/Gt/Ge in both matrix `binary()`s with a `matrices are not ordered`
  message before the generic fallthrough.

### A13-6  dot/cross silently accept mixed row/column vector orientation
- **Severity:** Nit
- **Kind:** doc (undocumented lenience)
- **Location:** stdlib_matrix.hpp:259-281, stdlib_complex.hpp:449-471
- **What:** `dot` and `cross` only check `isVector()` + equal element count, ignoring orientation, so
  a 1×3 row vector `.dot(`/`cross(` a 3×1 column vector succeeds (flattened-data semantics). Convenient
  and arguably fine for a scripting language, but numpy would reject the 2-D shape mismatch, and the
  behaviour is undocumented. Not a bug; noting the semantic choice.
- **Confidence:** confirmed (`matrix.vector([1,2,3]).dot(matrix.Matrix([[1],[2],[3]]))` → `14.0`)
- **Fix sketch:** either document "vectors are compared by flattened contents, orientation-agnostic",
  or (stricter) require matching orientation. Low priority; document rather than change behaviour.

## Test gaps (C++ and .ki)
- **Reflection/asymmetry:** no test asserts the *current* one-sided behaviour of `2*M` / `2+z` /
  `i*C` (A13-1). Whichever way it's resolved, add `.ki` tests pinning it (either they work, or they
  throw a specific message) so it can't silently regress.
- **Empty / degenerate shapes:** `Matrix([]).determinant()` (→ 1.0, the empty-product convention),
  `Matrix([]).inverse()` (→ `[]`), `Matrix([]).trace()` (→ 0.0), `matrix.vector([]).norm()` (→ 0.0),
  `matrix.zeros(0,0)` — all behave sensibly but appear untested. Worth golden tests since the empty
  determinant = 1 convention is a surprising-but-correct edge.
- **Huge-alloc guard:** add `.ki` tests that `matrix.zeros(100000,100000)`, `matrix.identity(1e6)`,
  `matrix.Matrix(1e9,1e9)`, and the ComplexMatrix equivalents throw `Matrix too large` /
  `ComplexMatrix too large` (confirmed working; needs a regression test).
- **Complex analytic domain matrix:** the throwing set (log/log10 of 0, atanh(±1), 0**neg, 0**complex)
  and the *non-throwing* off-axis points (sqrt(-1), log(-1), asin(2), acosh(0)) should each have an
  assertion — the correctness of "throw only at genuine singularities" is subtle and easy to break.
- **is_zero threshold:** `complex.of(1e-11,0).is_zero()` (True) vs `complex.of(1e-8,0).is_zero()`
  (False) — the 1e-20 (|z|²) cutoff is a hardcoded magic number with no test pinning it.
- **Hermitian dot:** assert `u.dot(v) = Σ conj(u_i)·v_i` (conjugation on the FIRST operand) and that
  `u.dot(u)` is real ≥ 0 — the Hermitian-vs-plain inner product is a classic silent bug and currently
  rests only on manual inspection.
- **ComplexMatrix ordering:** no test that `C < C` etc. throw (they do, generically).
- **Serialization:** Matrix/ComplexMatrix/Complex round-trip through `dump` correctly (verified `==`),
  but the `_setstate_` malformed-state guards (short state list, mismatched data size, negative dims,
  short `[re,im]` pair) have no adversarial test — those are the guards most likely to crash on a
  hand-crafted/corrupt blob.

## DRY / single-source-of-truth notes
- **Biggest item — A13-2:** MatrixVal ≈ ComplexMatrixVal (element type aside). Prime candidate for a
  templated `MatrixBase<T>` / CRTP base. The 16M-element size guard, the Integer/non-negative
  `indexOf`, the `getItem`/`setItem` shape logic, the `rows/cols/shape/get/set/row/compare/apply/
  _getstate_/_setstate_` methods, and the `zeros/ones/identity/vector` factories are all duplicated.
- **A13-3:** transpose & trace should both go through the generic tensor engine (real path already
  does; complex path re-rolls them). determinant/inverse already correctly share `tensor::determinant`
  / `tensor::inverse` for both element types — follow that model for transpose/trace.
- The two square-check helpers (`if (rows()!=cols()) throw "... requires a square ..."`) repeat 4×
  across the two files; a shared `requireSquare(m, opname)` would centralize the message.
- `cClose` (stdlib_complex.hpp:35) and `floatClose` (runtime.hpp) are deliberately separate to avoid a
  header-order dependency — noted as an intentional, documented duplication, not a defect.
