### A20-1  semver range: whitespace between operator and version silently mis-parsed (kpm-critical)
- **Severity:** High
- **Kind:** correctness-bug
- **Location:** src/kirito/stdlib_kimodules.hpp:2727-2755 (`_parserange`), 2657-2724 (`_expand`)
- **What:** node-semver accepts optional whitespace between a comparator operator and the version,
  e.g. `">= 1.2.0"`, `"> 1.2.0"`, `"< 2.0.0"`. `_parserange` tokenizes an OR-part by **splitting on
  a single space** (`orpart.strip().split(" ")`), so `">= 1.2.0"` becomes two tokens `">="` and
  `"1.2.0"` that are ANDed. `_expand(">=")` sees an empty version → `xmj` true → yields `>=0.0.0`;
  `_expand("1.2.0")` yields `=1.2.0`. The AND is therefore `>=0.0.0 AND =1.2.0`, i.e. it matches
  ONLY the exact version 1.2.0 instead of the intended `>=1.2.0`. Worse: `validrange(">= 1.2.0")`
  returns **True**, so the malformed range is silently accepted and resolves wrong rather than
  erroring. kpm resolves `repo@<constraint>` through `satisfies`/`maxsatisfying`, so a manifest with
  a spaced constraint silently installs the wrong (or no) version.
- **Repro:**
  ```
  var sv = import("semver")
  sv.satisfies("1.5.0", ">= 1.2.0")   # => False   (node-semver: TRUE)
  sv.satisfies("2.0.0", "> 1.2.0")    # => False   (node-semver: TRUE)
  sv.validrange(">= 1.2.0")           # => True    (silently accepts the broken range)
  ```
- **Confidence:** confirmed (reproduced on release/ki-linux-x64)
- **Fix sketch:** before tokenizing, collapse operator-adjacent whitespace: regex/scan-glue a
  leading `>=|<=|>|<|=|~|^` token to the version token that follows it (node-semver's own tokenizer
  joins `(operator)\s*(version)`). Simplest: replace `"> "`/`">= "`/etc. runs, or re-tokenize so a
  bare-operator token absorbs the next token.

### A20-2  semver: leading zeros accepted in core and numeric prerelease (invalid per SemVer/node-semver)
- **Severity:** Medium
- **Kind:** correctness-bug / missing-guard
- **Location:** src/kirito/stdlib_kimodules.hpp:2486-2490 (`_isnum`), 2501-2524 (`parse`)
- **What:** `_isnum` accepts any digit run, and `parse`/`valid` convert core parts via `Integer()`,
  so `"01.2.3"`, `"1.02.3"`, and a numeric prerelease id with a leading zero (`"1.2.3-01"`) are all
  accepted. SemVer §2/§9 and node-semver reject leading zeros in MAJOR/MINOR/PATCH and in numeric
  prerelease identifiers; `semver.valid("01.2.3")` is `null` in node-semver. `valid` here returns the
  string unchanged. Because `compare` normalizes via `Integer()`, `"01.2.3" == "1.2.3"` in precedence,
  so `valid` disagreeing with node-semver is the main issue (a tag `v01.2.3` would validate).
- **Repro:**
  ```
  var sv = import("semver")
  sv.valid("01.2.3")     # => "01.2.3"  (node-semver: None)
  sv.valid("1.2.3-01")   # => "1.2.3-01" (node-semver: None; numeric id leading zero)
  ```
- **Confidence:** confirmed
- **Fix sketch:** in `_isnum`/core validation reject a length>1 run starting with `"0"`; likewise for
  a numeric prerelease identifier (a pre id that `_isnum` and has leading zero and len>1 → throw).

### A20-3  statistics.quantiles clamps endpoints instead of extrapolating (diverges from Python)
- **Severity:** Low
- **Kind:** correctness-bug
- **Location:** src/kirito/stdlib_kimodules.hpp:415-436 (`quantiles`)
- **What:** For the exclusive method, when a cut point's interpolation position `pos = i*(ld+1)/n`
  falls below 1 or at/above `ld`, this impl **clamps** to the min/max data value (`s[0]`/`s[ld-1]`).
  CPython's `statistics.quantiles` (exclusive, the default) instead **extrapolates linearly** past the
  data ends, producing values outside `[min,max]`. They agree whenever the data is large relative to
  `n` (all cut points land in the interior), but diverge for small samples.
- **Repro:**
  ```
  import("statistics").quantiles([1,2])   # Kirito: [1.0, 1.5, 2.0]   Python: [0.75, 1.5, 2.25]
  ```
- **Confidence:** confirmed (cross-checked python3 statistics)
- **Fix sketch:** follow CPython's algorithm: `m=ld+1; j,delta=divmod(i*m,n); j=clamp(j,1,ld-1);
  out = (s[j-1]*(n-delta)+s[j]*delta)/n` — clamp the *index* j, not the result, so the interpolation
  extrapolates. (Low priority: only affects tiny samples.)

### A20-4  functools.reduce conflates "no initial" with an explicit `initial=None`
- **Severity:** Medium
- **Kind:** correctness-bug
- **Location:** src/kirito/stdlib_kimodules.hpp:232-245 (`reduce`)
- **What:** `have = initial != None` treats an explicitly-passed `initial=None` as "no initial value
  supplied". So `reduce(f, seq, None)` folds `seq` from its own first element (Python folds `None`
  with the first element), and `reduce(f, [], None)` **throws** "reduce of empty sequence" where Python
  returns `None`. A None accumulator seed is a legitimate use (e.g. building up an optional).
- **Repro:**
  ```
  var ft = import("functools")
  ft.reduce(Function(a,b): return a+b, [1,2,3], None)   # Kirito: 6   Python: TypeError (None+1)
  ft.reduce(Function(a,b): return a+b, [], None)        # Kirito: throws   Python: None
  ```
- **Confidence:** confirmed
- **Fix sketch:** Kirito lacks a unique sentinel default, but one can be minted per-call:
  `var _NO = []` (fresh list, unique identity) as the default, then `have = initial is-not _NO`
  via `id(initial) != id(_NO)`. (Same sentinel pattern would fix any "None means unset" default.)

### A20-5  tabular sortvalues throws on a column with missing (None) values
- **Severity:** Medium
- **Kind:** correctness-bug / missing-guard
- **Location:** src/kirito/stdlib_kimodules.hpp:1830-1834 (DataFrame.sortvalues), 1532-1544 (Series.sortvalues)
- **What:** `sortvalues` sorts the column via `positions.sort(key: col[p])`; if the column holds a
  `None` (a missing value, the norm for a real dataset) the underlying comparison `None < Integer`
  throws `cannot order 'None' and 'Integer'`. pandas sorts NaN/None to the end (`na_position='last'`)
  and never raises. So sorting any frame with missing data crashes.
- **Repro:**
  ```
  var tb = import("tabular")
  tb.DataFrame({"a":[3,None,1]}).sortvalues("a")   # throws "cannot order 'None' and 'Integer'"
  ```
- **Confidence:** confirmed
- **Fix sketch:** partition out missing (`_isnan`) rows, sort the rest, and append the missing rows at
  the end (both Series and DataFrame paths — share a helper).

### A20-6  tabular groupby keeps None/NaN groups (pandas drops them)
- **Severity:** Low
- **Kind:** doc / correctness-divergence
- **Location:** src/kirito/stdlib_kimodules.hpp:1936-1950 (GroupBy._init_)
- **What:** grouping by a column with `None` keys creates a `None` group; pandas `groupby` drops
  NaN/None keys by default (`dropna=True`). Reasonable as-is (None is a valid key) but divergent and
  undocumented. Additionally NaN Float keys (`x != x`) would each fall in distinct groups (exact `==`),
  fragmenting a group — worth documenting.
- **Repro:** `tb.DataFrame({"g":["a",None,"a"],"v":[1,2,3]}).groupby("g").sum()` → includes a None group.
- **Confidence:** confirmed
- **Fix sketch:** document the difference, or add a `dropna` option that filters `_isnan(k)` keys.

### A20-7  textwrap.wrap never breaks a word longer than the width
- **Severity:** Low
- **Kind:** correctness-divergence / doc
- **Location:** src/kirito/stdlib_kimodules.hpp:509-523 (`wrap`)
- **What:** A single word longer than `width` is emitted whole on its own line. Python's
  `textwrap.wrap` (default `break_long_words=True`) chops it into width-sized pieces. So a line can
  exceed `width` here where Python guarantees `<= width`.
- **Repro:** `import("textwrap").wrap("supercalifragilistic word", 5)` → `['supercalifragilistic','word']`
  (Python: `['super','calif','ragil','istic','word']`).
- **Confidence:** confirmed
- **Fix sketch:** when `len(word) > width`, slice it into width chunks before appending; or document
  the no-break behaviour.

### A20-8  base64.decode is stricter than Python — rejects non-canonical trailing bits
- **Severity:** Low
- **Kind:** correctness-divergence
- **Location:** src/kirito/stdlib_kimodules.hpp:601-624 (`decode`)
- **What:** `decode` throws `invalid base64: truncated or corrupted input` when the final group carries
  non-zero unused bits (e.g. `"QI=="`). Python's `base64.b64decode` is lenient and silently drops
  those bits (`b64decode("QI==") == b'@'`). Canonical output from this or any correct encoder always
  has zero trailing bits, so round-trips are unaffected; only non-canonical third-party base64 is
  rejected. Deliberate (per the comment) but a divergence worth noting.
- **Repro:** `import("base64").decode("QI==")` throws; Python returns `[64]`.
- **Confidence:** confirmed
- **Fix sketch:** if strict-canonical is intended, keep as-is + document; otherwise mask off and ignore
  the trailing bits like Python.

## DRY / single-source-of-truth notes
_(logic duplicated across files that should be unified)_

- **semver version-string validation duplicated:** the strictness rules live only in `parse`/`_isnum`
  while `_expand` re-parses version components independently (`Integer(parts[i])`, its own x-part
  logic). A leading-zero / range-vs-valid consistency fix (A20-2) must touch both, or `valid` and
  `validrange` will keep disagreeing on the same string. Consider one shared `_parsecore(str)` used by
  both `parse` and `_expand`.
- **Series-reduction glue duplicated across tabular:** `DataFrame._aggcols`, `GroupBy._reduce`, and
  `GroupBy.agg` each rebuild `Series(vals)` and call the same reducer set; `agg`'s reducer Dict
  (sum/mean/min/max/std/count/median) is a second copy of the reduction surface already on `Series`.
  A single `{name -> Series-method}` table shared by `_reduce`/`agg`/`describe` would DRY it.
- **`_isnan`/`_numeric`/`_isnumericcol`** (tabular) and the median/variance logic are duplicated
  between `Series` methods and `statistics` module — not identical enough to merge, but the
  "sample variance = sum((x-m)^2)/(n-1)" formula appears in both (statistics.variance and
  Series.variance); a shared numeric-core would remove the drift risk.
