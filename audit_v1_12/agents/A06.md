### A06-1  List.apply / List.sort snapshot elements are not GC-rooted (dangling-handle error, not the documented snapshot semantics)
- **Severity:** Medium
- **Kind:** correctness-bug / missing-guard (GC root)
- **Location:** src/kirito/runtime.hpp:506 (`List.apply`, `std::vector<Handle> src = self_list(...).elems`), :539 (`List.sort`, `std::vector<Handle> src = ...`)
- **What:** `apply` and `sort` snapshot the receiver's element handles into a plain `std::vector<Handle> src` (deliberately, so user code can't observe the partial result / to survive a mid-op realloc of the list). But that snapshot is NOT a GC root (the GC only scans tempRoots_/pins/aux operand stacks — see vm.hpp:127 `collectGarbage`). If the user `fn`/`key` invoked during the loop drops every OTHER reference to a snapshotted element (e.g. `xs.clear()`) and then allocates enough to cross the GC threshold (100000 allocs), those elements are swept. The next iteration passes a dangling handle to `fn`/`key`, and the arena's generation check throws `dangling handle (stale generation)` — a confusing internal error surfacing to the user instead of the documented snapshot behaviour (fn sees the original elements). Small interned ints hide it; non-interned elements (inner lists/strings/large ints) expose it.
- **Repro / failure scenario:** (reproduced, catchable throw — NOT memory-unsafe, the generation check catches it)
  ```
  var xs = [["a"], ["b"], ["c"], ["d"], ["e"]]
  var first = True
  var f = Function(x):
      if first:
          first = False
          xs.clear()
          var junk = []
          var i = 0
          while i < 300000:
              junk.append(i * 1000000 + 12345)
              i = i + 1
      return x
  var out = xs.apply(f)   # -> error: dangling handle (stale generation)
  ```
  Same with `xs.sort(key = f)` (error at the subsequent `key` call). ASan reports NO memory error (arena generation guard catches it first).
- **Confidence:** confirmed (reproduced on dist/debug-asan)
- **Fix sketch:** GC-root the snapshot for the duration of the op. e.g. `RootScope rs(vm); for (Handle h : self_list(...).elems) rs.add(h);` capturing into a rooted `src`, or add the snapshot vector as an aux-root region (`pushAuxRoots(&src)`/`popAuxRoots()`). Apply to all four snapshot sites below.

### A06-2  Dict.apply / Set.apply snapshots (pairs()/items()) are not GC-rooted — same UAF-avoided-by-generation-check bug as A06-1
- **Severity:** Medium
- **Kind:** correctness-bug / missing-guard (GC root)
- **Location:** src/kirito/runtime.hpp:667 (`Dict.apply`, `auto pairs = dict(...).pairs()`), :801 (`Set.apply`, `std::vector<Handle> src = ...items()`)
- **What:** identical pattern to A06-1: the snapshot of handles taken before the per-element `fn` loop is a plain local (not a GC root). If `fn` drops the container's other references to a snapshotted key/value/element and forces a GC, the handle dangles and the op throws `dangling handle (stale generation)`. (String.apply and Bytes.apply are IMMUNE — they snapshot the raw `std::string`, not handles, and build fresh values per element.)
- **Repro / failure scenario:** (reproduced) `d.apply(f)` with `f` doing `d.clear()`+300k allocs on the first value: `apply` completes but stores dangling value handles; `out.values()` then throws `dangling handle (stale generation)`. `s.apply(f)` with `s.clear()` and **non-pinned** elements (build them via `base + "1"` etc — NB: bare string/int *literals* are pinned as bytecode constants and mask the bug, so a naive `{"a","b",...}` test appears to pass) throws on the subsequent iteration. Note List/Dict list-VALUE literals are NOT pinned, only scalar literals are.
- **Confidence:** confirmed (reproduced on release + dist/debug-asan; Dict corruption surfaces on result read, Set on the next `fn` call)
- **Fix sketch:** root the snapshot (see A06-1). Broader note: this snapshot-then-call-user-code pattern (`std::vector<Handle> snap = ...; for (h : snap) fn(h)`) recurs; auditing every native that does it (extend/update/join loops read from a live-rooted source so are OK; only the ones snapshotting into an unrooted local are affected).

### A06-3  Integer/Float `.compare()` loses precision for large int64 (routes through double)
- **Severity:** Low
- **Kind:** correctness-bug
- **Location:** src/kirito/runtime.hpp:310 (`makeNumericCompare` → `floatClose(asDouble(...), asDouble(other), ...)`)
- **What:** `.compare()` converts both operands with `asDouble` before calling `floatClose`, collapsing every int64 magnitude beyond 2^53. So two *distinct* large integers are reported "close" even with zero tolerance, in direct contrast to `==`, which the codebase deliberately made EXACT (via `compareIntFloat`, avoiding the lossy round-trip). `(2^63-1).compare(2^63-2, rel_tol=0.0, abs_tol=0.0)` -> True while `==` -> False.
- **Repro / failure scenario:** (reproduced)
  ```
  (9223372036854775807).compare(9223372036854775806, rel_tol = 0.0, abs_tol = 0.0)   # True (wrong)
  9223372036854775807 == 9223372036854775806                                          # False (right)
  ```
- **Confidence:** confirmed (reproduced on release)
- **Fix sketch:** When both operands are Integer, do the tolerance test in integer space (or special-case rel_tol==0 && abs_tol==0 to exact `==`). Low priority since `.compare` is explicitly the *approximate* method, but zero-tolerance should still agree with `==`.

### A06-4  String find/index/rfind/rindex/count: an empty substring with `start` past the end reports a match instead of "not found"
- **Severity:** Low
- **Kind:** correctness-bug (edge semantics)
- **Location:** src/kirito/runtime.hpp:1041 (`cpIndexToByte` clamps an out-of-range cp to `s.size()`), consumed by find/index/rfind/count (:1254/:1352/:1361/:1234)
- **What:** Because `cpIndexToByte` clamps a `start` beyond the string to `s.size()`, `s.find(sub, start)` for an empty `sub` returns that clamped byte offset and the `bp + sub.size() <= hi` check passes, so an empty substring is "found" at position len. Python returns -1 (find/rfind) / raises (index/rindex) / 0 (count) when start > len. Non-empty substrings are unaffected (`find("x", 5)` correctly -> -1).
- **Repro / failure scenario:** (reproduced) `"abc".find("", 5)` -> 3 (want -1); `"abc".index("", 5)` -> 3 (want throw); `"abc".rfind("", 5)` -> 3 (want -1); `"abc".count("", 10)` -> 1 (want 0); `"abc".count("", 5, 2)` -> 1 (want 0).
- **Confidence:** confirmed (reproduced on release)
- **Fix sketch:** in the search methods, when `sub` is empty, first check the resolved code-point `start` against the code-point length and return the not-found result if `start > len` (and clamp `count`'s window boundary set to `b >= lo && b <= hi && b <= size`). Minor; only affects the empty-needle past-end corner.

### A06-5  `x in list` / list.index/count use pure value-equality with no identity short-circuit: `nan in [nan]` is False even for the SAME object
- **Severity:** Low
- **Kind:** correctness-bug (edge semantics) / doc
- **Location:** src/kirito/runtime.hpp:1586 (`ListVal::contains` -> `kiEquals`), :1540 (`kiEquals` only short-circuits identity for List/Dict, not scalars)
- **What:** `kiEquals` deliberately does NOT identity-short-circuit scalars (so `NaN != NaN` holds for `==`). But that also means an object is not "in" a list that literally contains that very object when its value-equality is false — `n = nan; n in [n]` is False, and `[n].index(n)` throws "value not in List". Python's containment/index use `is or ==`, so a NaN is always found in a list holding it. Surprising but arguably intentional given Kirito's exact-IEEE-754 stance.
- **Repro / failure scenario:** (reproduced)
  ```
  var n = import("math").nan
  var xs = [n]
  n in xs        # False   (Python: True)
  xs.count(n)    # 0
  xs.index(n)    # throws "index: value not in List"
  ```
- **Confidence:** confirmed (reproduced on release)
- **Fix sketch:** Optional: add an identity (same-handle) short-circuit inside `ListVal::contains`/`index`/`count`/`remove` before the value compare, matching Python's `is or ==`. Or document the divergence explicitly. Low.

### A06-7  Set/Dict store the SAME NaN object twice — Set uniqueness invariant violated; `contains`/`get` can't find an element that is in the container
- **Severity:** Medium
- **Kind:** correctness-bug (invariant violation)
- **Location:** src/kirito/collections.hpp:98 (`probeBucket` — bucket membership via `Object::equals`, no identity short-circuit); reached by SetVal::add/contains and DictVal::set/find
- **What:** `probeBucket` decides "is this key already in the bucket" purely with `Object::equals`, which for FloatVal is exact IEEE-754 (`NaN != NaN`, no identity check). Consequence: adding the *same* NaN handle to a Set twice inserts it twice (len becomes 2), and `s.contains(n)` returns False for the very object `n` that was added; a Dict likewise gets two entries for `d[n]=...; d[n]=...` and `d.get(n)` misses. This breaks the core Set guarantee (no duplicates) and the "a member is contained" invariant — for the SAME object, which Python collapses via an `is`-first check. (This is the Dict/Set face of A06-5; escalated because it violates the Set uniqueness contract, not just a `in`-list nicety.)
- **Repro / failure scenario:** (reproduced)
  ```
  var n = import("math").nan
  var s = Set([]); s.add(n); s.add(n)
  len(s)          # 2   (same object stored twice!)
  s.contains(n)   # False
  var d = {}; d[n] = "a"; d[n] = "b"
  len(d)          # 2
  d.get(n, "MISSING")   # "MISSING"
  ```
- **Confidence:** confirmed (reproduced on release)
- **Fix sketch:** add an identity short-circuit in `probeBucket`: treat `keyOf(bucket[i]) == key` (same Handle / `&deref == &key`) as a match before the value `equals`. Distinct NaN objects stay distinct (identity differs, equals false → still separate), matching Python's `is or ==`, but the same object collapses and is findable. NB: coordinate with A05/A08 — `probeBucket` and `kiEquals` (A06-5) share this "no scalar identity short-circuit" root cause.

### A06-6  `abs(INT64_MIN)` returns a NEGATIVE value (documented two's-complement wraparound, but a footgun)
- **Severity:** Low
- **Kind:** doc / edge semantics
- **Location:** src/kirito/runtime.hpp:2883 (`abs` builtin: `v < 0 ? wsub(0, v) : v`)
- **What:** `abs(-9223372036854775808)` -> `-9223372036854775808` (INT64_MIN has no positive int64 counterpart). This is the intended, UB-free wraparound (matches Kirito's fixed-width int64 policy) and ASan reports no UB, but `abs(x) < 0` is a surprising invariant break. Same class as arithmetic overflow wrap.
- **Repro / failure scenario:** (reproduced, ASan-clean) `abs(-9223372036854775808)` -> `-9223372036854775808`.
- **Confidence:** confirmed (reproduced on release + asan)
- **Fix sketch:** Acceptable as-is per the fixed-width-int design; if desired, throw "abs result out of Integer range" for the single INT64_MIN input (mirrors `round`'s out-of-range throw), or just document it. Arb-precision ints (a listed future enrichment) would remove it.
