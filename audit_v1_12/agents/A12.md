## Findings

### TENSOR-1  High-ndim tensor bypasses the 64-dim cap → recursive `str()`/`tolist()` SIGSEGV (uncatchable)
- **Severity:** High
- **Kind:** crash/UB · missing-guard
- **Location:** src/kirito/stdlib_tensor.hpp:150 (`TensorVal::format` recursive `rec`), :1656 (`tolist` recursive `build`), :201 (`tns::checkSize` — only bounds numel, not ndim), :2130 (the 64-dim cap that IS enforced, but only in the `Tensor(nested,...)` constructor).
- **What:** The `Tensor(nested[,dtype])` constructor caps rank at 64 *"to bound rec() depth (no stack overflow)"*. But every other tensor-producing path — `reshape`, `expanddims`, `broadcastto`, `squeeze` — routes through `tns::make` → `checkSize`, which bounds only the element **count** (`checkedNumel`), never the number of **dimensions**. So a rank-200000 tensor is trivially constructed. Its `str()` (nested-bracket `format`/`rec`) and `tolist()` (`build`) recurse once per axis, overflowing the native C++ stack → **SIGSEGV**, uncatchable by a Kirito `try`.
- **Repro:**
  ```
  var t = import("tensor")
  var ones = []
  for i in range(200000): ones.append(1)
  var y = t.Tensor([5.0]).reshape(ones)   # rank 200000, 1 element — accepted
  io.print(y)                              # -> exit 139 (SIGSEGV in str())
  # y.tolist() likewise dies (exit 137)
  ```
  Release: `io.print(y)` exits 139; `y.tolist()` exits 137. Confirmed.
- **Confidence:** confirmed (reproduced).
- **Fix sketch:** Enforce the rank cap at the single allocation choke point: add `if (s.size() > 64) throw KiritoError("Tensor: too many dimensions (max 64)")` inside `tns::checkSize` (or `tns::make`), so the 64-dim invariant is single-sourced and holds for reshape/expanddims/broadcastto/etc., not just the constructor. (Matches the "bound rec() depth" intent that the constructor comment already states.)

### TENSOR-2  `einsum` contraction-count is unbounded and overflows size_t → silent wrong result / DoS hang
- **Severity:** High
- **Kind:** correctness-bug · resource-exhaustion · missing-guard
- **Location:** src/kirito/stdlib_tensor.hpp:1416 (`std::size_t total = 1; for (std::size_t d : lsz) total *= d;`) and the loop at :1418 (`for (std::size_t t = 0; t < total; ++t)`).
- **What:** `einsum` iterates over the full Cartesian product of *all* labels (free **and** contracted). `total` is the unchecked product of every label's size. Two problems: (a) it can **overflow size_t**, wrapping to a small number (often 0), so the loop runs too few times and returns a **silently wrong** numeric result; (b) even without overflow there is **no cap** on the loop count, so a tiny input can request an astronomically long loop (DoS). The output tensor `FT out(oshape)` is bounded by `checkedNumel`, but the *contraction* loop is not — a spec whose contracted (summed-away) dims are large but whose output is small (e.g. `"ij,kl,mn->"`, output scalar) escapes every guard.
- **Repro (silent wrong answer via overflow):**
  ```
  var a = t.ones([2048,2048]); var b = t.ones([2048,2048]); var c = t.ones([2048,2048])
  var r = t.einsum("ij,kl,mn->", a, b, c)   # true value ~ 4194304^3 = 7.4e19
  io.print(r.item())                        # -> 0.0   (2048^6 = 2^66 wraps size_t to 0)
  ```
  Inputs are only ~4M elements each (allocatable); result printed `0.0` instead of ~7.4e19. Confirmed.
  A non-overflowing large `total` (e.g. `"ab,cd,ef->"` with dims ~1000) instead hangs indefinitely.
- **Confidence:** confirmed (reproduced, overflow branch).
- **Fix sketch:** Compute `total` with an overflow-checked running product (same pattern as `tensor::checkedNumel`) against a work-bound cap (e.g. reuse `kMaxElems`, or a larger explicit operation-count cap), and throw `KiritoError("einsum: contraction is too large")` on overflow/exceed instead of iterating a wrapped or astronomical count.

### TENSOR-3  Re-running `backward()` on the same graph double-counts; `zerograd()` clears only the leaf so it is unfixable
- **Severity:** Medium
- **Kind:** correctness-bug
- **Location:** src/kirito/stdlib_tensor.hpp:700 (`accumulateGrad`), :705-745 (`runBackward` — grads are stored persistently on *every* node and never reset), :1999 (`zerograd` resets only `asT(vm,self).grad`, i.e. the receiver, with no access to anonymous intermediate nodes).
- **What:** Each graph node keeps a persistent `.grad`. `runBackward` seeds the root's grad and accumulates into every reachable node, but nothing clears the **intermediate** node grads between passes. A second `backward()` on the same graph re-seeds and re-propagates on top of the stale intermediate grads, so the leaf gradient is **not** the correct per-call value — it compounds. `zerograd()` only clears the tensor it is called on (a leaf); the anonymous intermediates are unreachable from Kirito, so there is **no way** to correctly re-run backward on one graph.
- **Repro:**
  ```
  var x = t.Tensor([3.0], requiresgrad=True)
  var y = (x*x).sum()
  y.backward(); io.print(x.grad.tolist())   # [6.0]  correct
  x.zerograd()
  y.backward(); io.print(x.grad.tolist())   # [18.0] — should be [6.0] (PyTorch retain_graph)
  ```
  Confirmed. (Accumulating grads across *separate* fresh subgraphs sharing a leaf — the intended functional-rebind idiom — works correctly: `a.backward(); b.backward()` gives the right sum. Only re-running backward on the *same* graph misbehaves.)
- **Confidence:** confirmed (reproduced).
- **Fix sketch:** Either (a) make `runBackward` reset every reachable node's `.grad` for non-leaf nodes at the start of the pass (or accumulate into a pass-local `unordered_map<Handle,FT>` and only persist grads onto leaves), or (b) document that `backward()` is single-shot per graph and error on a second call (track a per-node "backward already run" flag). Option (a) matches the documented "accumulates t.grad" model without the compounding footgun.

### TENSOR-4  `norm(ord=2)` on a 2-D tensor returns Frobenius, not NumPy's spectral norm
- **Severity:** Low
- **Kind:** doc · correctness-bug (divergence)
- **Location:** src/kirito/stdlib_tensor.hpp:1336 (`normT`, `ord == 2.0` branch always does flattened `sqrt(sum x*x)`).
- **What:** For a matrix, `numpy.linalg.norm(M, 2)` is the spectral norm (largest singular value); Kirito returns the flattened Euclidean/Frobenius norm. `t.norm([[1,0],[0,3]])` gives `3.1623` (Frobenius) where NumPy's ord-2 gives `3.0`. The stdlib is documented as "Euclidean 2-norm", so this is arguably intended for vectors but is a silent divergence for matrices.
- **Repro:** `io.print(t.norm(t.Tensor([[1.0,0.0],[0.0,3.0]])))` → `3.16227766016838`.
- **Confidence:** confirmed.
- **Fix sketch:** Either document explicitly that `norm` is always the flattened Euclidean/Frobenius norm (never spectral), or special-case matrix ord=2/ord=1/ord=inf to match NumPy's matrix-norm semantics.

Suggest landing these as a `.ki` gradcheck regression harness — there is no numeric gradcheck in the suite that I could find.
- `norm` matrix semantics (TENSOR-4) untested for the matrix case.

## DRY / single-source-of-truth notes
- The **64-dimension rank cap** is written inline only in the `Tensor(nested,...)` constructor (:2130) with the numeric cap single-sourced in `checkedNumel`. The rank cap should be single-sourced alongside it (in `checkSize`/`checkedNumel`) so every allocation path shares it — see TENSOR-1. Currently the constructor is the only enforcer and every other path bypasses it.
- The `requiresgrad`-forbidden-for-Complex check ("gradients are Float-only ...") is duplicated verbatim ~5× across `Tensor`/`zeros`/`ones`/`full`/`eye`/`requiresgrad` (filled/likeFn lambdas + each factory). A single `markRequiresGrad(vm, h, wantComplex)` helper would DRY it.
- The NaN-last comparator `[](double x,double y){ return x < y || (y!=y && x==x); }` is repeated in `medianT`, `sortT`, `argsortT`, `uniqueT` (4 sites). Hoist to one shared functor.
- The per-op boilerplate `if (!wantsGrad(...)) return make(...); ... makeAutogradFloat(...)` is repeated across all ~20 `g_*` primitives — acceptable given each has a distinct backward, but a small `finishGrad(vm, out, parents, bw)` helper could remove the duplicated early-return.
