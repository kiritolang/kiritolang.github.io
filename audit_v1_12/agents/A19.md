# A19 — parallel / dispatcher

**Files in scope:** stdlib_parallel.hpp, dispatcher.hpp
**Status:** running
**Probe binaries:** `release/ki-linux-x64` (fast), `dist/debug-asan` (memory/UB), `dist/debug-tsan` (races).

## Findings

### A19-1  Spawning an entry-script function re-runs the entry file's unguarded top level → fork bomb
- **Severity:** Medium (High if a user routinely spawns from `<main>` without the `argmain` guard — catastrophic host DoS)
- **Kind:** resource-exhaustion / missing-guard
- **Location:** src/kirito/dispatcher.hpp:665-711 (`runWorker` → `worker.evalIn(src, modScope, sourceFile)`); guard at stdlib_parallel.hpp:529 only rejects `sf.empty() || sf == "<main>"`; main.cpp:227 runs the entry file with `chunkName = file` (the real path, NOT `<main>`).
- **What:** A worker rebuilds a spawned function's closure scope by RE-READING and RE-EVALUATING its whole source file. For an imported module this only re-runs harmless top-level definitions. But the ENTRY script is run (main.cpp) with chunkName = its real path, so its top-level functions carry a non-`<main>` sourceFile and pass the spawn guard. When such a function is spawned, each worker re-executes the entry file's **entire unguarded top level** — including the `spawn(...)` calls themselves — so every worker spawns more workers: an unbounded, exponential fork bomb of OS threads + worker VMs. The `<main>` check is dead for the CLI (the entry chunk is never named `<main>`).
- **Repro:** `p5b_inline.ki`:
  ```
  var io = import("io"); var p = import("parallel")
  io.print("top-level runs")
  var inline = Function(x): return x + 100
  var t = p.spawn(inline, 5)      # NO `if argmain:` guard
  io.print(t.join())
  ```
  → prints "top-level runs" forever, spawns threads until the box is killed (observed exit 137/143, machine saturated). Guarding the driver with `if argmain:` (p5c_guarded.ki) fixes it and is the documented pattern (sqldb/webserver examples), but there is **no diagnostic** when it is omitted.
- **Confidence:** confirmed (reproduced under release; had to `timeout`/kill).
- **Fix sketch:** Python's spawn detects this ("An attempt has been made to start a new process before the current process has finished its bootstrapping phase … you probably forgot `if __name__=='__main__'`") and raises. Kirito should do the same: set a per-VM "loading module body" flag around `worker.evalIn(src, …)` in `runWorker` (and around the entry-file `runSource`), and have `parallel.spawn` throw a clear, catchable error if called while that flag is set (i.e. from module/entry top-level during bootstrap rather than from inside the spawned function). Cheaper partial fix: name the CLI entry chunk `<main>` so entry-script functions are simply non-spawnable — but that breaks the legitimate `if argmain:` pattern the examples rely on, so the bootstrap-flag guard is preferred.

### A19-2  Dispatcher retains every Task forever — `tasks_` is never erased (confirms & extends Wave-1 A08-2)
- **Severity:** Medium
- **Kind:** resource-exhaustion
- **Location:** src/kirito/dispatcher.hpp:724 (`tasks_` map), 549-582 (`spawnTask`/`taskDone`/`joinTask`), 586-600 (`shutdown`).
- **What:** `spawnTask` inserts a `std::unique_ptr<Task>` into `tasks_` keyed by id and NOTHING ever removes it. `taskDone` and `joinTask` look up but never erase; only `~KiritoDispatcher`/`shutdown` frees the whole map. Every `spawn` therefore leaks a `Task` (its `std::thread` object + the full result **dump blob** `std::string`) for the entire lifetime of the dispatcher. A long-lived server that spawns one task per request (exactly the sqldb/webserver connection-worker model) grows without bound — the retained result blobs can be arbitrarily large. Sanitizers don't flag it (reachable until shutdown).
- **Repro:** spawn N tasks that each return a large value and join them; RSS grows ~linearly in N and never recovers after join (p1_manyspawn.ki scaled up; retention visible by code inspection — `tasks_.erase` appears nowhere).
- **Confidence:** confirmed (static; `grep tasks_.erase` = 0 hits).
- **Fix sketch:** After a successful `joinTask` (thread joined, result copied out), erase the entry from `tasks_` under `registryMutex_` — the Task is finished and its result already returned to the caller, so nothing else references it (`reap`'s non-claimer spins on `done` before the claimer erases; sequence the erase after both `reap` returns). A joined-but-not-yet-erased Task double-join is already prevented by the `joining` atomic. Alternatively cap/recycle ids and drop finished Tasks lazily. Note: the retention currently *masks* a would-be UAF (see Probe log) — the fix must keep `reap`'s pointer valid, so erase only the claimer, after it has finished `reap` and copied `result`.
