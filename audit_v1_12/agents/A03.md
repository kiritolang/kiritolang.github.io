### A03-1  Analyzer false-positive: "assigned but never used" for a var used only via a forward/later reference in a nested (or sibling) function
- **Severity:** Low
- **Kind:** correctness-bug (static-analysis false positive)
- **Location:** src/kirito/analyzer.hpp:77-82 (markUsed) + :65-72 (popScope unused check); root cause is order-sensitive use-tracking (markUsed only marks names already `declare`d)
- **What:** The unused-variable check is order-sensitive: `markUsed(name)` walks the live scope stack and marks the FIRST already-declared match. A name referenced inside a nested function that is defined **before** the variable's own `var` declaration is scanned while the variable is not yet `declare`d, so the use is silently dropped; the later `var` then looks "assigned but never used". The analyzer documents itself as "SOUND-leaning: it only warns when confident, to avoid false positives" — this violates that contract. The resolver correctly accepts the program (membership scoping / forward refs), and it runs fine.
- **Repro:**
  ```
  var io = import("io")
  var main = Function():
      var helper1 = Function():
          return shared      # forward reference to a not-yet-declared local
      var shared = 5
      return helper1()
  io.print(main())           # prints 5, but emits: warning: variable 'shared' is assigned but never used
  ```
  Observed: `p2.ki:5:5: warning: variable 'shared' is assigned but never used` then `5`.
- **Confidence:** confirmed (reproduced on release/ki-linux-x64)
- **Fix sketch:** Make the unused check order-insensitive. Two-pass per scope: first `declare` every name a block binds (a decl pre-pass, mirroring collectBlockDecls) THEN walk uses; or defer the unused-name emission to a set of "referenced names" collected across the whole scope subtree (including nested functions, which are already analyzed while the scope is live) and only warn for decls whose name never appears in that referenced set. Note the nested function bodies are analyzed while the enclosing scope is on the stack, so a single deferred pass that records every NameExpr seen anywhere under the scope, then diffs against decls at popScope, fixes it without changing traversal order.

### A03-2  Stale/incorrect doc comment for the Unpack opcode ("last target on top")
- **Severity:** Nit
- **Kind:** doc
- **Location:** src/kirito/bytecode.hpp:66
- **What:** The `Unpack` opcode is documented as "pop an iterable -> push its n spread slots, **last target on top**". The actual VM (bytecode_vm.hpp:309, `for (i = count; i-- > 0;) push(slots[i]); // reversed: slot 0 ends on top`) pushes them so **slot 0 / the FIRST target ends on top**, which is what the compiler relies on (compiler.hpp:190,206,276 store the first target first with the comment "first target on top"). The doc contradicts the code; unpacking works correctly, only the comment is wrong. A maintainer trusting the comment could reverse the store loop and silently break all destructuring.
- **Repro / failure scenario:** documentation only — no runtime effect. `var a, b = [1, 2]` correctly binds a=1, b=2.
- **Confidence:** confirmed
- **Fix sketch:** change the bytecode.hpp comment to "…push its n spread slots, first target (slot 0) on top".
