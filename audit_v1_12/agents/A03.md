### A03-1  Analyzer false-positive: "assigned but never used" for a var used only via a forward/later reference in a nested (or sibling) function
- **Severity:** Low
- **Kind:** correctness-bug (static-analysis false positive)
- **Location:** src/kirito/analyzer.hpp:77-82 (markUsed) + :65-72 (popScope unused check); root cause is order-sensitive use-tracking (markUsed only marks names already `declare`d)
- **What:** The unused-variable check is order-sensitive: `markUsed(name)` walks the live scope stack and marks the FIRST already-declared match. A name referenced inside a nested function that is defined **before** the variable's own `var` declaration is scanned while the variable is not yet `declare`d, so the use is silently dropped; the later `var` then looks "assigned but never used". The analyzer documents itself as "SOUND-leaning: it only warns when confident, to avoid false positives" — this violates that contract. The resolver correctly accepts the program (membership scoping / forward refs), and it runs fine.
- **Repro:**
  ```
  var io = import("io")
  var main = Function():
      var helper1 = Function():
          return shared      # forward reference to a not-yet-declared local
      var shared = 5
      return helper1()
  io.print(main())           # prints 5, but emits: warning: variable 'shared' is assigned but never used
  ```
  Observed: `p2.ki:5:5: warning: variable 'shared' is assigned but never used` then `5`.
- **Confidence:** confirmed (reproduced on release/ki-linux-x64)
- **Fix sketch:** Make the unused check order-insensitive. Two-pass per scope: first `declare` every name a block binds (a decl pre-pass, mirroring collectBlockDecls) THEN walk uses; or defer the unused-name emission to a set of "referenced names" collected across the whole scope subtree (including nested functions, which are already analyzed while the scope is live) and only warn for decls whose name never appears in that referenced set. Note the nested function bodies are analyzed while the enclosing scope is on the stack, so a single deferred pass that records every NameExpr seen anywhere under the scope, then diffs against decls at popScope, fixes it without changing traversal order.

### A03-2  Stale/incorrect doc comment for the Unpack opcode ("last target on top")
- **Severity:** Nit
- **Kind:** doc
- **Location:** src/kirito/bytecode.hpp:66
- **What:** The `Unpack` opcode is documented as "pop an iterable -> push its n spread slots, **last target on top**". The actual VM (bytecode_vm.hpp:309, `for (i = count; i-- > 0;) push(slots[i]); // reversed: slot 0 ends on top`) pushes them so **slot 0 / the FIRST target ends on top**, which is what the compiler relies on (compiler.hpp:190,206,276 store the first target first with the comment "first target on top"). The doc contradicts the code; unpacking works correctly, only the comment is wrong. A maintainer trusting the comment could reverse the store loop and silently break all destructuring.
- **Repro / failure scenario:** documentation only — no runtime effect. `var a, b = [1, 2]` correctly binds a=1, b=2.
- **Confidence:** confirmed
- **Fix sketch:** change the bytecode.hpp comment to "…push its n spread slots, first target (slot 0) on top".

### A03-3  Resolver false-positive: a parameter default that references an EARLIER parameter is rejected, though the runtime supports it
- **Severity:** Medium
- **Kind:** correctness-bug (resolver false positive; resolver/runtime divergence)
- **Location:** src/kirito/resolver.hpp:146-149 (checkExpr, FunctionExpr arm) — defaults are checked BEFORE `resolveScope` declares the params, and in the enclosing scope only. Runtime binder: src/kirito/runtime.hpp:1999-2012 (defines params incrementally, so param i's default sees params 0..i-1).
- **What:** The runtime call binder fills defaults inside the same loop that defines params (`env.define` at runtime.hpp:2011 happens per-iteration, after the default eval at :2003). So when a later parameter's default runs, all EARLIER parameters are already bound in the call scope and are visible. The resolver, however, checks every default in the enclosing scope with NO params declared (it calls `resolveScope(fn->body, &fn->params)` only AFTER checking the defaults, and only for the body). Result: `Function(n, size = n)` — where `size`'s default references the earlier param `n` — is thrown out at compile time as `name 'n' is not defined`, even though the runtime would bind `size = n` correctly. This contradicts the resolver's own documented contract ("THROWS … for any name reference that does not resolve to a binding — **a parameter**, …"): `n` IS a parameter and IS visible at runtime.
- **Repro:**
  ```
  var f = Function(n, size = n):     # rejected: resolver.hpp says name 'n' is not defined
      return [n, size]
  ```
  Runtime proof (add an enclosing `a` so the resolver accepts, then observe the binding):
  ```
  var a = 999
  var f = Function(a, b = a):        # resolver accepts (module a exists)
      return [a, b]
  var io = import("io")
  io.print(f(7))                     # prints [7, 7]  -> default 'a' resolved to the PARAM (7), not module a (999)
  ```
  So the runtime makes earlier params visible to later defaults; the resolver forbids the pattern outright.
- **Confidence:** confirmed (reproduced: d4 rejected, d3 prints [7, 7] on release/ki-linux-x64)
- **Fix sketch:** Make the resolver mirror the runtime's incremental binding. Open the function scope first, then for each param left-to-right: `checkExpr(default)` (with the enclosing scope AND the already-declared earlier params visible), then `declare(p.name)`. The analyzer (analyzer.hpp:305-310) ALREADY does exactly this (declare inside the param loop, after each default) — so the resolver is the sole outlier. Note: an earlier param that ALSO exists in the enclosing scope currently resolves (by existence) but validates against the wrong binding; the incremental fix also aligns which binding the check targets.

### A03-4  DRY / consistency: three passes encode "where a parameter default resolves", and they DISAGREE with each other and with the runtime
- **Severity:** Low
- **Kind:** DRY
- **Location:** resolver.hpp:147, locals.hpp:71-72 (CaptureScan::enterFunction), analyzer.hpp:305-310, vs runtime binder runtime.hpp:1999-2012
- **What:** The rule "a parameter default sees the enclosing scope PLUS all earlier params of the same signature" is implemented four times and inconsistently:
  * runtime.hpp — earlier params visible (defines params incrementally). GROUND TRUTH.
  * analyzer.hpp — earlier params visible (declare inside the param loop, after each default). Matches runtime.
  * resolver.hpp — earlier params NOT visible (all defaults checked before any param declared). DIVERGES → causes A03-3 (a hard error).
  * locals.hpp CaptureScan — earlier params NOT visible (defaults scanned before params added to the shadow set). DIVERGES, but benignly: it can only mark one of F's OUTER locals as captured when a same-named inner param actually shadows it → an over-approximation (keeps the local name-based), which locals.hpp itself declares "safe". No wrong read, only a missed slot.
- **Repro / failure scenario:** A03-3 for the resolver; the locals.hpp case is a silent lost optimization (e.g. an outer local `n` stays name-based when an inner `Function(n, size = n)` shadows it).
- **Confidence:** confirmed (resolver) / plausible (locals.hpp precision loss, reasoned from code)
- **Fix sketch:** Factor the "default resolution environment" into one shared helper (extend locals.hpp, already the single source of truth for scope membership) so resolver, capture scan, and analyzer all consume the same incremental-params rule, and it matches the runtime binder by construction.
