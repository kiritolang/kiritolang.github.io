# A16 — compression / hash

**Files in scope:** stdlib_zlib.hpp, stdlib_gzip.hpp, deflate.hpp, stdlib_hash.hpp
**Status:** DONE
**Probe binaries:** `release/ki-linux-x64` (fast), `dist/debug-asan` (memory/UB), `dist/debug-tsan` (races).

## Findings

### A16-1  gzip multi-member decompression bomb bypasses the 256 MiB inflate output cap
- **Severity:** High
- **Kind:** resource-exhaustion / missing-guard
- **Location:** src/kirito/stdlib_gzip.hpp:41-87 (`gzipfmt::decompress`), specifically `result += out;` at :82 with `deflate::inflateImpl(..., kMaxInflateOut, ...)` at :71
- **What:** The `kMaxInflateOut` (256 MiB) zip-bomb guard bounds **each member's** `out` buffer, but a gzip stream may hold arbitrarily many concatenated members (RFC 1952 §2.2) and their bodies are accumulated into `result` with **no aggregate cap**. A malicious multi-member `.gz` produces N × 256 MiB of output from tiny input — the exact OOM the guard is documented to prevent ("Throw instead, like the other guards", deflate.hpp:221-224). DEFLATE amplifies ~1000:1, so ~256 KiB of compressed data per member; a few MiB of input → many GiB of `result`. Reachable on untrusted data via the documented recipe `gzip.decompress(net.get(url).content)`.
- **Repro:** `python3 -c "import gzip;g=gzip.compress(b'\x00'*(200*1024*1024),1);open('m.gz','wb').write(g*2)"` (2 members, 4.5 MiB input) → `gzip.decompress` returns **419430400 bytes (400 MiB)** — exceeds the 256 MiB cap, exit 0. Scaled to 5+ members under a 2 GB vmem cap it throws `std::bad_alloc` instead of the clean "size limit" DeflateError; with no ulimit it OOMs the host. (Single-member gz/zlib bombs correctly throw "decompressed data exceeds the size limit".)
- **Confidence:** confirmed (reproduced, both the 400 MiB success and the bad_alloc)
- **Fix sketch:** Bound the *aggregate* output. Pass a shrinking budget to each member — `inflateImpl(view, kMaxInflateOut - result.size(), &consumed)` — and/or check `if (result.size() + out.size() > kMaxInflateOut) throw DeflateError("gzip: decompressed data exceeds the size limit")` before `result += out`. Reserve/append so the throw fires before the allocation blows up.

### A16-2  Stored (BTYPE=00) block NLEN is never validated
- **Severity:** Low
- **Kind:** correctness-bug / missing-guard
- **Location:** src/kirito/deflate.hpp:275-285 (stored-block case in `inflateImpl`)
- **What:** RFC 1951 §3.2.4 requires `NLEN == ~LEN`; the code reads LEN, does `p += 4` to skip **both** LEN and NLEN, and never checks the one's-complement. A corrupt stored-block length is silently accepted. Real zlib rejects it ("invalid stored block lengths", `Error -3`). For zlib/gzip streams the outer Adler-32/CRC-32 still catches the resulting corruption, but **raw `zlib.inflate` / `zlib.deflate` has no checksum**, so a corrupted stored-block stream silently returns wrong bytes.
- **Repro:** raw deflate `\x01` + LEN(=5) + NLEN(=0x0000, wrong) + "ABCDE" → Kirito `zlib.inflate` returns `"ABCDE"` (OK len=5); Python `zlib.decompressobj(-15)` raises "invalid stored block lengths".
- **Confidence:** confirmed
- **Fix sketch:** After reading LEN and NLEN, `if ((uint16_t)~len != nlen) throw DeflateError("invalid stored block lengths");`

### A16-3  zlib header FCHECK / CINFO not validated
- **Severity:** Nit
- **Kind:** correctness-bug (lenient parse)
- **Location:** src/kirito/deflate.hpp:351-360 (`zlibDecompress`)
- **What:** RFC 1950 §2.2 requires `(CMF*256+FLG) % 31 == 0` (FCHECK) and CINFO ≤ 7 (window ≤ 32K). Neither is checked — only the CM nibble (==8) and FDICT bit. A header real zlib rejects is accepted; the Adler-32 trailer still catches body corruption so impact is minimal.
- **Repro:** flip FLG so `(CMF*256+FLG)%31 == 1` → Kirito `zlib.decompress` returns the body fine (OK len=280); real zlib errors on the header check.
- **Confidence:** confirmed
- **Fix sketch:** `if (((cmf<<8)|flg) % 31 != 0) throw DeflateError("invalid zlib header check bits");` and optionally reject CINFO>7.

### A16-4  inflate output cap is soft by up to one max-match (258 B)
- **Severity:** Nit
- **Kind:** missing-guard (cosmetic)
- **Location:** src/kirito/deflate.hpp:228-245 (`inflateBlock`)
- **What:** The `if (out.size() > maxOut)` check runs once per symbol at the top of the loop; a single back-reference then appends up to 258 bytes, so the buffer can exceed `maxOut` by ≤258 B before the next iteration throws. Harmless (256 MiB + 258 B), noted for completeness — the stored-block path already checks `out.size()+len` up front and does not have this.
- **Confidence:** confirmed (by reading; overshoot is bounded and tiny)
- **Fix sketch:** none needed, or check `out.size()+len > maxOut` before the copy loop for symmetry with the stored path.

## Verified SAFE (attacked, held up)
- **INFLATE memory safety:** 6400 adversarial inputs (400 random + 3000 bit-flip mutations of dynamic-Huffman streams + 3000 more) through `zlib.inflate`/`decompress`/`gzip.decompress` under **ASan/UBSan — zero errors**. Plus hand-built malformed streams: bad block type, truncated stored header/body, truncated mid-stream, empty input — all throw cleanly.
- **`Huffman::build` / `decode` bounds:** reasoned + fuzzed. Code lengths are always ≤ `maxBits` by construction (CL lengths 0-7 built maxBits 7; lit/dist lengths 0-15 built maxBits 15), so `counts[l]++` never writes OOB. `decode` indexes `symbols[index + (code-first)]` which is bounded by total nonzero symbols ≤ `lengths.size()` even for oversubscribed/incomplete tables — no OOB, worst case throws "invalid Huffman code".
- **Back-reference / distance:** `if ((size_t)d > out.size()) throw "distance too far back"` blocks OOB reads before window start; overlapping copies are byte-by-byte via index (realloc-safe); `len`≤258, `dsym`/`sym` range-checked (`>=30`/`>=29` throw). No over-long matches, no OOB.
- **Dynamic-table length overshoot** (sym 16/17/18 repeats) is caught by the exact `lengths.size() != hlit+hdist` check after the loop; `sym==16` on empty `lengths` throws "invalid repeat".
- **gzip header abuse:** bad magic, bad CM, huge/lying FEXTRA xlen, unterminated FNAME, truncated header — all throw ("truncated header/stream" etc.); `pos` past buffer is caught by the `pos+8 > data.size()` guard before `substr`/reads. CRC-32 mismatch, ISIZE mismatch, trailing junk, second-member corruption all throw. Multi-member valid streams concatenate correctly.
- **Single-member bombs** (300 MiB zeros, gz + zlib): both throw "decompressed data exceeds the size limit".
- **String↔Bytes byte-transparency:** all-256-byte-values round-trips exactly through zlib/gzip; type preserved (Bytes→Bytes, String→String); `md5` of the 256-byte block matches `hashlib` (`e2c865db...`).
- **Hash vectors:** md5("")=d41d8cd9…, sha256("")=e3b0c442…, sha1("")=da39a3ee…, crc32("")=0, adler32("")=1, crc64("")=0 — all correct; "abc" vectors and crc64("abc")=3231342946509354535 match Python. FDICT and zlib preset-dictionary correctly rejected.

## Test gaps (C++ and .ki)
- **The `kMaxInflateOut` zip-bomb guard is entirely untested** — no C++ or `.ki` test asserts that oversize inflate output throws, single OR multi-member. This is why A16-1 (aggregate multi-member bypass) went unnoticed. Add: (a) a single-member bomb asserting "exceeds the size limit" throws; (b) a multi-member bomb asserting the *aggregate* is bounded (currently FAILS — the regression test for A16-1).
- **No malformed-DEFLATE-internals regression tests.** `errors/zlib_bad_data.ki` only covers a generic bad stream. Missing crafted cases: invalid block type (BTYPE=11), truncated stored header/body, distance-too-far-back, invalid length/distance symbol, oversubscribed/incomplete dynamic Huffman table, `sym==16` repeat with empty lengths. (I fuzzed these clean but there is no durable regression test.)
- **No stored-block NLEN** (A16-2) or **zlib FCHECK** (A16-3) negative test.
- Multi-member gzip is tested for *correctness* (concatenation, `test_compress_hash_deep.cpp:131`) but not for the size cap.

## DRY / single-source-of-truth notes
- **GOOD (single-source confirmed):** `crc32`/`adler32` live only in `deflate.hpp` and are reused by both `gzipfmt::compress/decompress` (internal) and the `hash` module (`stdlib_hash.hpp` includes `deflate.hpp` for exactly this). `crc64` lives only in `stdlib_hash.hpp`. The whole DEFLATE/INFLATE core (`compress`/`inflateImpl`) is shared by the `zlib` and `gzip` modules through `deflate.hpp`. No compression/checksum logic is duplicated.
- **Minor:** the String-or-Bytes `codec` wrapper lambda (`m.fn(...) { makeStringOrBytes(argStringOrBytes(...)) }`) is copied nearly verbatim in `stdlib_zlib.hpp:27-36`, `stdlib_gzip.hpp:99-108`, and the `digest`/`checksum32` helpers in `stdlib_hash.hpp:38-52`. They differ only in error-prefix handling (zlib prefixes `"zlib: "`, gzip conditionally, hash none). Could be a shared `native.hpp`/`bytes.hpp` helper taking a prefix; Nit — not worth the coupling unless a 4th module joins.
