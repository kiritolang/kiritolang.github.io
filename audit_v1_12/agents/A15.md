# A15 — serde (json / serialize / dump) adversarial audit

Scope: `src/kirito/stdlib_json.hpp`, `stdlib_serde.hpp` (shared flatten/rebuild core),
`stdlib_serialize.hpp` (text KSER1), `stdlib_dump.hpp` (binary KDMP).

**Headline: the deserialization surface is well-hardened.** Extensive fuzzing (below) found
**zero crashes / zero ASan-or-UBSan reports** across ~44k malformed inputs run through the
ASan/UBSan build. Every hard case the theme list calls out is already handled: bounded/negative
length prefixes are rejected before allocating, every back-reference index is bounds-checked in
`rebuild` (checkId), the recursive graph walk (`flatten`) and the JSON parser both carry native-stack
depth guards, resource-like natives throw a clean catchable error on serialize, and cycles/shared
refs round-trip. Findings below are consistency / defense-in-depth / test-gap, not live crashes.

## Fuzzing performed (all against `dist/debug-asan`)
- **Binary `dump.loads`** — 5 seed blobs (dict+list+set+None+float+string, nested lists, self-cycle,
  user-class instance, unicode string) × 4000 mutations each (bit-flips, truncation at every point,
  extreme-byte sets, random insertions) = **20000 runs → 0 crashes**.
- **Text `serialize.loads`** — 4 seed blobs (dict, self-cycle list, class instance, set) × 3000
  token/char mutations (big/negative/overflowing counts, char corruption, truncation, token
  insertion/deletion) = **12000 runs → 0 crashes**.
- **`json.parse`** — 17 seeds (surrogate pairs, lone/reversed surrogates, embedded NUL, huge
  numbers, leading-zero/`.5`/`-` edge cases, deep nesting) × 2500 byte-level mutations over the full
  0..255 alphabet = **~42.5k runs → 0 crashes** (see probe log).
- **Targeted crafted blobs** (hand-built, not just mutated): OOB link id, OOB rootId, deeply-nested
  reconstructed chain (2,000,000 deep), hostile native `_setstate_` state (Integer fed to
  `Matrix._setstate_`) — all throw a catchable error, no crash under ASan.

## Findings

### A15-1  serde::rebuild has no depth cap on the reconstructed structure
- **Severity:** Low
- **Kind:** missing-guard (defense-in-depth / consistency)
- **Location:** src/kirito/stdlib_serde.hpp:159 (`rebuild`); contrast stdlib_json.hpp:55-61 (JSON
  DepthGuard=1000) and stdlib_serde.hpp:86-92 (`flatten` cap 10000 / 1500 asan).
- **What:** `flatten` (serialize/dump write side) caps graph depth at 10000 and `json.parse` at 1000,
  but the *reverse* path `rebuild` (reached by `dump.loads` / `serialize.loads`) applies **no** limit
  to the depth of the structure it reconstructs. `rebuild` itself is iterative (two/three flat
  `for i<n` passes), so building a deep structure does not overflow the stack — but nothing stops an
  attacker's blob from encoding a structure far deeper than `flatten` would ever emit (a 2M-deep list
  chain loads in <1s). The danger is downstream: any recursive operation later performed on the
  loaded value could stack-overflow.
- **Repro:** crafted KDMP of N=2,000,000 nested single-element lists (`deepchain.bin`);
  `dump.load(path)` returns "loaded" successfully. Round-tripping the same shape through `flatten`
  (`dump.dumps` of a 12000-deep list) instead throws "structure too deeply nested to dump".
- **Mitigating factor (why Low, not High):** every downstream recursive traversal I could reach is
  itself guarded and throws catchably or is iterative — verified under ASan/release on the 2M-deep
  loaded chain: `print`/`str` → "structure too deeply nested to stringify"; `a == b` → caught;
  `copy.deepcopy` → succeeds (guarded/iterative); GC under allocation pressure → survives. So no
  *uncatchable* crash is currently reachable. It remains an inconsistency and a latent trap: the
  moment any unguarded recursive traversal of a value is introduced, untrusted `loads` input turns it
  into an uncatchable SIGSEGV.
- **Confidence:** confirmed (no cap reproduced; downstream currently guarded).
- **Fix sketch:** reject over-deep blobs at load time using the same ceiling `flatten` enforces so
  `dumps`/`loads` are symmetric — a pure structural check over the Node table (longest reference
  chain) before pass 2 rejects them without changing the iterative reconstruction.

### A15-2  dump.loads relies on the VM native-boundary catch for std::exception; serialize.loads wraps it explicitly (DRY/consistency)
- **Severity:** Nit
- **Kind:** DRY / consistency
- **Location:** src/kirito/stdlib_dump.hpp:172-175 (`dumpfmt::read`) vs stdlib_serialize.hpp:172-186
  (`serial::loads`, which translates `stoi/stol/stod` `what()` into "corrupt serialized data: …").
- **What:** the text reader can throw raw `std::out_of_range`/`std::invalid_argument` from
  `stoll`/`parseDouble`, so `serial::loads` catches `std::exception` and rewrites it to a clean Kirito
  diagnostic. `dumpfmt::read` performs no such translation; it depends on the VM's native-boundary
  catch (which surfaces any escaping `std::exception` as a catchable String). In practice the binary
  reader parses bytes directly and does not call `stoX`, so it rarely throws a bare std::exception,
  and fuzzing produced **no escapes** — but the two sibling codecs handle the same failure class
  differently. Cosmetic only.
- **Confidence:** confirmed (code inspection; fuzz shows no user-visible escape).
- **Fix sketch:** optionally give `dumpfmt::read` the same `try{…}catch(const std::exception&)`
  translation for uniform diagnostics, or document that both rely on the VM boundary.

### A15-3  Test gap: no regression for structurally-valid-but-hostile blobs (OOB indices, over-deep reconstruction)
- **Severity:** Low
- **Kind:** test-gap
- **Location:** tools/tests/scripts/probe_serde_truncation.ki, probe_serde_fuzz.ki.
- **What:** existing adversarial tests exercise truncation and random bytes/text, which almost always
  fail the header/length checks early, plus a `_setstate_`-throws case and cycles. They do **not**
  build a header-valid blob carrying a **hostile back-reference index** (exercising `rebuild`'s
  `checkId` / rootId bounds check) nor a **hand-crafted over-deep chain** (A15-1). Both were verified
  manually here to be safe (OOB → clean throw; deep chain → loads then guarded downstream), but
  neither has a pinned regression.
- **Confidence:** confirmed (grepped the suite).
- **Fix sketch:** add a `.ki` regression that builds Bytes for a KDMP blob with (a) a link id ≥ object
  count and (b) rootId ≥ object count, asserting `dump.loads` throws catchably; and one that asserts
  a 2M-deep loaded chain either throws at load (if A15-1 is fixed) or that `print`/`==` on it throw
  catchably (locking in the current guarantee).

## Test gaps (summary)
- OOB back-reference / rootId in a header-valid blob — no regression (A15-3).
- Over-deep reconstruction via crafted blob — no regression (A15-3).
- Native `_setstate_` fed a wrong-typed hostile state via crafted blob (e.g. Integer to
  `Matrix._setstate_`) — verified catchable here but not pinned by a `.ki`/C++ test.
- NaN / ±Infinity / -0.0 round-trip through dump + serialize + json — verified working here
  (`nan.ki`); good candidate for an explicit pinned assertion if not already covered by
  deep_serialization.ki.
- Positive: truncation-at-every-offset, random-bytes, cycles, shared refs, resource-like rejection,
  and `_setstate_`-throws are already well covered by probe_serde_truncation.ki / probe_serde_fuzz.ki.

## DRY / single-source-of-truth notes
- `serialize` (text) and `dump` (binary) share **one** graph walk + reconstruction
  (`serde::flatten`/`serde::rebuild`); each supplies only its byte codec (`serial::encode/decode`,
  `dumpfmt::encode/decode`). Confirmed single-source — no divergence in id assignment, cycle
  termination, bounds checking, or supported-type knowledge. Good.
- `json` is deliberately separate (flat interchange, no ref/cycle preservation) with its own
  recursive-descent parser + writer — documented design, not a DRY violation.
- Both binary/text decoders bound the untrusted object count by input size (dump: `n > data.size()`;
  serialize: `n<0 || n>s_.size()`) and `reserve(min(n, 1<<16))` to defeat the ~80× zeroed-alloc
  amplification of `vector(n)`. Both defer *all* id validation to `rebuild`'s single `checkId`/rootId
  guard — a good single choke point.
- Minor asymmetry (A15-2): std::exception→KiritoError translation exists only on the text path.

## Probe log
- `roundtrip.ki` — dict/list/None/bool/float/string round-trips through `dump` cleanly.
- `res.ki` (ASan) — regex, open file, serialize-of-regex all throw catchably ("… caught OK"); no
  crash. Resource-like natives (Socket/File/Regex) reject serialize as spec'd.
- `jdeep.ki` (ASan) — 5000-deep `[[[…]]]` JSON → caught (DepthGuard=1000 fires cleanly; no stack
  overflow even under ASan's enlarged frames).
- `jstr.ki` (ASan) — `json.stringify` of a 5000-deep in-memory list → caught (active.size()>1000
  guard).
- `sdeep.ki` (ASan + release) — `dump.dumps`/`serialize.dumps` of a 12000-deep list → caught by
  `flatten` depth guard on both.
- `deepchain.bin` (crafted 2M-deep KDMP) — `dump.load` → "loaded" (no rebuild cap, A15-1); then
  `print` → "too deeply nested to stringify"; `==` → caught; `copy.deepcopy` → ok; GC under 3M
  allocations → survives. No uncatchable crash.
- `oob.bin` / `oobroot.bin` (crafted OOB link / rootId) → both caught under ASan (checkId + rootId
  bounds).
- `mat_hostile.bin` (crafted Stateful "Matrix" with Integer state) → caught under ASan (native
  `_setstate_` validates; serde routes hostile state to it correctly).
- `nan.ki` — inf / -inf / nan / 0.0 / -0.0 / 1e308 round-trip through dump + serialize + json.
- **Binary dump fuzz:** 20000 ASan runs, 0 crashes. **Text serialize fuzz:** 12000 ASan runs, 0
  crashes. **JSON fuzz:** ~42.5k ASan runs, 0 crashes.
