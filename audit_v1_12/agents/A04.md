# A04 — Bytecode VM

**Files in scope:** bytecode_vm.hpp, control.hpp, function.hpp
**Status:** running
**Probe binaries:** `release/ki-linux-x64` (fast), `dist/debug-asan` (memory/UB), `dist/debug-tsan` (races).

## Findings

### A04-1  Call-depth guard bypass: recursion through a native higher-order fn (sorted/sort/apply/min/max with a recursive callback) SIGSEGVs the process
- **Severity:** High
- **Kind:** crash/UB · missing-guard (guard-limit vs frame-size mismatch)
- **Location:** src/kirito/vm.hpp:326-330 (`maxCallDepth_ = 3000`) + the native higher-order call paths in runtime.hpp (`sorted`/`min`/`max`/`.sort`/`.apply` key-callback invocation). The guard itself lives in `KiritoVM::enterCall` (vm.hpp:164) / `CallGuard` (vm.hpp:334) taken in `KiFunction::callFull` (runtime.hpp:1932).
- **What:** The call-depth guard caps *recursion count* at `maxCallDepth_ = 3000` (release) and throws a catchable `maximum recursion depth exceeded`. The limit is calibrated for the **shallowest** per-level native frame (plain Kirito→Kirito recursion). Recursion whose per-level native frame is much deeper — because it passes through a native higher-order function whose C++ frame + the comparator/callback machinery is large — overflows the 8 MB native stack at depth **~2940**, i.e. BEFORE the guard fires at 3000. Result: a hard **SIGSEGV** of the host process instead of a catchable error. This directly violates the CLAUDE.md invariant: "the call-depth guard … must throw a catchable error, not overflow native stack", and it defeats `try/catch` (the crash is uncatchable) and any embedder relying on encapsulation (one bad script kills the host process).
- **Repro / failure scenario:** (release binary, default 8 MB stack)
  ```
  var g = Function(n):
      return sorted([n], key=g)[0]
  var r = g(5)
  ```
  → `Segmentation fault` (rc=139). Same crash with `min([n], key=g)`, `max([n], key=g)`, `xs.sort(key=g)`, `[n].apply(g)[0]`. A manual counter shows the process survives a bounded depth of ~2930 through `sorted` but SIGSEGVs at ~2950; the guard's own limit is 3000, so it never fires. Plain recursion, deep-expression recursion, and method/operator/`_str_`/`_hash_`/`_bool_`/`_call_` recursion all fire the guard cleanly at 3000 (shallower frames) — only the sort/apply/min/max-key family (deep frames) crashes.
- **Confidence:** confirmed (reproduced; rc=139 on release, caught cleanly on asan where `maxCallDepth_=500`). **Proven a stack overflow**: with `ulimit -s 65536` (64 MB) the SAME script survives to the guard and throws the catchable `maximum recursion depth exceeded`; at 8 MB (default) and 4 MB it SIGSEGVs — i.e. the fixed count of 3000 outlasts the 8 MB stack for this frame size.
- **Fix sketch:** The count-based guard cannot bound a variable per-level frame size. Preferred fix: make the guard **stack-usage-aware** — capture a stack-base address when the VM/worker thread starts and, in `enterCall`, compare the address of a local against that base; throw when within a safety margin (e.g. 512 KB) of the configured stack limit, regardless of per-level frame size. Cheaper stopgap: lower `maxCallDepth_` default so even the deepest observed native frame has margin at 8 MB (the sort path crashes at ~2940, so ≤2000 would be safe *for that path* — but any deeper native frame, e.g. nested HOFs, could still exceed it, so a count-only cap is inherently fragile). Note the guard is only taken in `KiFunction::callFull`; native HOFs re-enter through it per callback, so the *count* is correct — it's the *frame size* the count can't see.

## Test gaps (C++ and .ki)
- **No test exercises recursion THROUGH a native higher-order function** (`sorted`/`min`/`max`/`.sort`/`.apply` with a `key=`/callback that recurses). This is exactly the A04-1 crash surface; a regression test should assert a catchable error (not a crash) for at least one such path, on the release-configured stack.
- No `.ki`/C++ test for **break/continue/return inside a `finally` on the EXCEPTION path** that swallows the in-flight exception (verified working by probe c5b/c5c/c5d, but uncovered by the suite — this is the subtle "$exc park" machinery in compiler.hpp:418-433).
- No test for **break/continue/return inside a catch HANDLER body that also has a `finally`, inside a loop** (probe d1/d2/d3 — the re-armed FINALLY frame at compiler.hpp:453/472). Works, but untested.
- No test for a **`with` manager whose `_exit_` throws while an exception is already in flight** (probe c1 — exception masking through the with-exit path). Works, untested.
- No test for `with` where **`_enter_` throws** (block must never be set up / `_exit_` must not run) — probe e2. Works, untested.
- Minor semantic gap (not a crash): `with CM() as m:` where the class has `_enter_` but **no `_exit_`** runs the body first and only fails at exit (probe e3: `'CM' object has no attribute '_exit_'`). CPython checks both dunders *before* entering. Low severity; no test either way.
