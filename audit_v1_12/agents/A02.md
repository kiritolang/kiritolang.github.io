### A02-2  f-string embedded expressions get spans relative to the `{...}` text (line 1), so runtime/resolver errors inside f-strings report the wrong file location
- **Severity:** Medium
- **Kind:** correctness-bug (diagnostic; feeds parallel span-matching)
- **Location:** src/kirito/parser.hpp:884-908 (`parseEmbedded` — `Lexer lex(code); Parser sub(...)` produce nodes with spans relative to `code`, starting at line 1 col 1; only the parse-time `catch` re-anchors to `t.span`).
- **What:** An f-string's embedded expression is re-lexed by a FRESH `Lexer` that starts at line 1, col 1, so every AST node it produces carries a span relative to the trimmed `{...}` text — NOT the real position in the source file. Parse-time errors are re-anchored to the f-string token span (the `catch (const KiritoError&)` block), but nothing offsets the spans that survive into the AST. Consequently RUNTIME errors and the RESOLVER's "name is not defined" error raised from inside an f-string expression report `line 1` / a bogus column, and the enclosing traceback frame line is wrong. CLAUDE.md is explicit that "errors should carry line and column"; this misreports both. The same relative spans also feed `dispatcher.hpp findFunctionBySpan` (parallel `spawn` resolves a function by its `(line,col)`) and `dispatcher.hpp` descends into f-string parts (dispatcher.hpp:454-457) — so an f-string-embedded `Function` literal has a line-1-relative span that can collide with a real top-level function's absolute span (contrived, but a latent correctness hazard for parallel).
- **Repro / failure scenario:**
  ```kirito
  var io = import("io")


  var x = 0
  io.print(f"{1 / x}")      # line 5
  ```
  Reports `File "/tmp/p.ki", line 1, in <module>` and `1:3: error: division by zero` — should be line 5. Control (same expression not in an f-string) correctly reports line 5, col 12. Undefined name: `io.print(f"{undefined_here}")` on line 4 reports `1:1: name 'undefined_here' is not defined`.
- **Confidence:** confirmed (reproduced on release/ki-linux-x64).
- **Fix sketch:** Offset the spans of all nodes produced by `parseEmbedded` by the f-string token's position and the byte offset of the `{` within the literal (accounting for the leading-whitespace trim). Simplest: after `sub.parseProgram()`, walk the returned expr and rewrite each node's `span` to `t.span` (coarse — every embedded error points at the f-string, matching the current parse-error behaviour and keeping diagnostics on the right line), or better, compute `line/col` deltas from the token's span + the offset of the embedded slice so columns stay meaningful. Add a `.ki`/C++ test asserting an error inside an f-string reports the f-string's real line.

### A02-4  f-strings silently accept invalid escapes (drop the backslash); plain strings reject them — divergent duplicated escape logic
- **Severity:** Low
- **Kind:** correctness-bug / DRY
- **Location:** src/kirito/parser.hpp:990-999 (`parseFString`, escape `switch` with `default: lit += e`) vs src/kirito/lexer.hpp:342-355 (`decodeEscape`, `default: throw "invalid escape"`).
- **What:** The escape set for string literals is implemented twice. The lexer's `decodeEscape` (cooked plain strings) THROWS on an unknown escape; the parser's f-string literal-part decoder silently drops the backslash and keeps the char. So `"\q"` is a lex error ("invalid escape '\q'") but `f"\q"` yields `q` with no error. CLAUDE.md documents "a bad escape ... is a clear lex error" and lists the exact cooked escape set — f-strings violate both. A typo like `f"\d{x}"` is silently accepted (`d` + value) instead of flagged. Also a maintenance hazard: adding an escape (e.g. `\u`) requires editing both code paths.
- **Repro / failure scenario:**
  ```kirito
  io.print("a\qb")    # error: invalid escape '\q'
  io.print(f"a\qb")   # prints: aqb   (silently drops backslash)
  ```
- **Confidence:** confirmed (reproduced on release/ki-linux-x64).
- **Fix sketch:** Make the f-string literal-escape `default` arm throw the same "invalid escape" `KiritoError` as `decodeEscape`. Better: factor the cooked-escape decode into one shared helper used by both the lexer (plain strings) and the parser (f-string literal parts), so the escape set is single-source. (Note the parser must keep its span re-anchoring to the f-string token.)

### A02-3  Trailing comma in a `var`/`for` single target silently does NOT unpack — inconsistent with rebind `a, = X`
- **Severity:** Low
- **Kind:** correctness-bug
- **Location:** src/kirito/parser.hpp:292-304 (`parseTargetNameList`, trailing-comma break); ast.hpp:253-258 (`VarDeclStmt` has no "was-a-tuple" flag, so `names.size()==1 && starIndex==-1` is indistinguishable from a plain single decl).
- **What:** `parseTargetNameList` breaks on a trailing comma and, for a single name with no star, produces a plain single binding — so `var a, = X` and `for a, in X` bind `a` to the WHOLE value with no unpacking and no count check. But the statement-level rebind path (parseStatement building a `TupleExpr`) DOES treat `a, = X` as a one-element unpack (with a count check). The two spellings diverge.
- **Repro / failure scenario:**
  ```kirito
  var a, = [5]      # a == [5]      (no unpack)
  var a, = [5, 6]   # a == [5, 6]   (no count error)
  # vs the rebind form:
  var a = 0
  a, = [5]          # a == 5        (unpacks 1 element)
  a, = [5, 6]       # error: expected 1 values to unpack, got 2
  for a, in [[9]]:  # a == [9]      (no unpack), unlike a 1-tuple target
  ```
  Same root cause (parseTargetNameList for var/for vs compiler-validated TupleExpr for rebind) also makes a LONE starred target diverge: `*a = [1,2,3]` (rebind) is correctly rejected ("invalid assignment target", like Python), but `var *a = [1,2,3]` (declare) is accepted and binds `a` to the whole list.
- **Confidence:** confirmed (reproduced on release/ki-linux-x64).
- **Fix sketch:** Have `parseTargetNameList` record whether a comma was ever consumed (a `bool sawComma`), and thread it into `VarDeclStmt`/`ForStmt` so the compiler unpacks (and count-checks) a single trailing-comma target exactly like the rebind path — or reject a dangling trailing comma. Consider rejecting a lone `var *a =` for parity with rebind. Align with the statement-level tuple semantics either way, and add tests.

### A02-1  Inline function body `return a, b` does not pack — trailing comma is stolen by the enclosing value sequence (silent wrong result)
- **Severity:** Medium
- **Kind:** correctness-bug
- **Location:** src/kirito/parser.hpp:734-778 (`parseInlineStatement`, KwReturn arm uses `parseExpr()` not `parseValueSeq()`); contrast parser.hpp:261-269 (`parseReturn` block form uses `parseValueSeq`).
- **What:** A block-bodied function packs a comma return (`return 1, 2` -> `[1,2]`, documented in CLAUDE.md: "`return a, b` returns `[a, b]`"). The INLINE function body form (`Function(): return 1, 2`) parses the return value with `parseExpr()` (a single expression), so the comma is NOT consumed by the return. Instead it escapes to the enclosing `parseValueSeq()` (var/assign RHS, `return` value, etc.), which packs the *whole function literal* into a tuple/List. Result: the function only returns its first value, and the binding silently becomes a List containing the function. No crash; wrong values.
- **Repro / failure scenario:**
  ```kirito
  var g = Function(): return 10, 20
  # type(g) is List (== [<function>]), NOT a function.  The `, 20` packed g into a list.

  var a, b = Function(): return 99, 88
  # a = <function> (which returns only 99, dropping 88); b = 88.  Silent — no error until g/a is called.
  ```
  Block form is correct: `Function():\n    return 1, 2` -> `type(f()) == List` ([1,2]). Inline diverges.
- **Confidence:** confirmed (reproduced on release/ki-linux-x64 and dist/debug-asan).
- **Fix sketch:** In `parseInlineStatement`, make the KwReturn arm call `parseValueSeq()` (matching `parseReturn`) instead of `parseExpr()` so an inline `return a, b` packs identically. The same comma-steal affects the KwThrow arm (`Function(): throw a, b` silently packs the fn into a List, whereas a block `throw a, b` correctly errors "expected end of statement") and the default (assign) arm — `node->value = parseExpr()`, and the `x = 1, 2` target. For consistency, mirror the corresponding block-form statement parsers (parseThrow errors on the trailing comma; parseReturn/assign pack via parseValueSeq), or explicitly reject a bare comma escaping an inline body with a clear diagnostic. Add `.ki` tests asserting inline and block `return a, b` / `throw a, b` behave identically.

### A02-5  Minor diagnostic-quality and spec-conformance issues (batch)
- **Severity:** Low / Nit
- **Kind:** missing-guard / doc / correctness-bug (minor)
- **Location / items:**
  1. **Inline body break/continue -> "expected an expression"** (parser.hpp:734-778, `parseInlineStatement` has no KwBreak/KwContinue arm; they fall to the default expr arm). `while True:` `    var f = Function(): break` reports `expected an expression`, not "'break' outside loop". No valid program is lost (break in a fn is always invalid), but the diagnostic is misleading.
  2. **`finally` before `catch`, and a second `finally`, -> "expected an expression"** (parser.hpp:203-226, `parseTry` stops after the first `finally`; a stray `catch`/`finally` is then parsed as a fresh statement and hits parsePrimary). Rejected, but with a confusing message instead of e.g. "'finally' must come after all 'catch' clauses".
  3. **`case` after `default` is accepted** (parser.hpp:178-195). CLAUDE.md documents "an optional TRAILING `default:`", but the switch-body loop accepts `case` arms after a `default`. Harmless (semantics are position-independent: exact-match with default as the fallback), but diverges from the documented grammar. Consider rejecting a `case` that follows `default`.
  4. **Trailing comma allowed in list/dict/set literals but NOT in call/keyword argument lists** (parsePostfix call loop, parser.hpp:579-582, calls `parseArg()` unconditionally after each comma). `[1,2,]`/`{1:2,}`/`{1,2,}` are fine; `f(1,)` and `f(a=1,)` report "expected an expression". Inconsistent with the literal forms and with common Python style.
  5. **List-literal assignment target diagnostic depends on whether names are defined** (compiler rejects `[a,b] = X` as "invalid assignment target" only after the resolver has already read `a`/`b`; if `a` is undefined you get "name 'a' is not defined" instead). Two different errors for the same unsupported construct.
  6. **Chained comparison is left-associative** (parseComparison, parser.hpp:485-513). `1 < 2 < 3` parses as `(1 < 2) < 3` and then errors "type 'Bool' does not support this binary operator" — a Python programmer writing `0 <= x < 10` gets a confusing runtime error rather than chained-comparison semantics or a clear diagnostic. This is a design choice (not undocumented-as-chained), but is a footgun; at minimum document it, ideally special-case a clearer message.
- **Confidence:** confirmed (all reproduced on release/ki-linux-x64).
- **Fix sketch:** Add explicit KwBreak/KwContinue arms to `parseInlineStatement` (throwing the real "outside loop" diagnostic); emit a targeted error for a `catch`/`finally` after `finally` in `parseTry`; reject `case` after `default` in `parseSwitch`; tolerate a single trailing comma in call/keyword argument lists (break the loop when the next token is `)`); and either document or improve the chained-comparison behaviour.

## DRY / single-source-of-truth notes
_(logic duplicated across files that should be unified)_

- **Full AST descent is implemented 3–4 times.** `dispatcher.hpp` (findInStmt/findInExpr/findInBlock) and `resolver.hpp` (checkStmt/checkExpr) each hand-roll a manual `dynamic_cast`-chain descent over EVERY Stmt/Expr node, and `analyzer.hpp` + `compiler.hpp` walk the tree again. The AST already provides a `StmtVisitor`/`ExprVisitor` double-dispatch interface (ast.hpp) — the analyzer/compiler use it, but the resolver and dispatcher do NOT. Adding a new AST node therefore requires touching several descents, and a miss is a SILENT bug (dispatcher failing to descend into a node = parallel can't resolve a function defined there; resolver failing to descend = undefined names not caught). The dispatcher comment even admits "Mirrors analyzer.hpp's descent". Recommend routing the resolver and dispatcher through the visitor interface, or providing one shared generic "walk all children" helper (analogous to the existing `ast::walkAssignTarget`) that every pass reuses.
- **Cooked-escape decoding duplicated** between lexer.hpp `decodeEscape` (plain strings) and parser.hpp `parseFString` (f-string literal parts) — and they DIVERGE on the invalid-escape case (throw vs silent drop; see A02-4). Factor into one shared escape decoder.
- **f-string brace/quote scanning implemented twice** in `parseFString` (parser.hpp:932-969): once to find the matching `}` (depth + quote/escape state) and once to split the `:format-spec` (bracket-depth + quote/escape state). Two near-identical string-state machines; unify into a single quote-aware scanner.
- **Integer base-prefix logic** partly duplicated: the lexer validates the `0x`/`0b`/`0o` prefix + digits (lexer.hpp:179-196) and the parser re-derives the base in `intLiteral` (parser.hpp:1016-1034). Acceptable (lexer validates, parser decodes) but the base table lives in both.
- **Statement-terminator / trailing-comma handling** is spread across `endSimpleStatement`, `expectStatementEnd`, `parseValueSeq`, `parseTargetNameList`, and the statement-level tuple loop — and they disagree (A02-1, A02-3, A02-5.4). A single "parse a target list" and a single "parse a value sequence" routine (with a consistent trailing-comma policy) used by var/for/assign/return/inline would remove the divergences.
