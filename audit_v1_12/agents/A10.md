### A10-1  BytesIO.truncate() bypasses the kMaxBuf guard — seek-then-truncate OOMs the host
- **Severity:** Medium
- **Kind:** missing-guard / resource-exhaustion
- **Location:** src/kirito/stdlib_io.hpp:342-347 (`truncate`), vs the guard at 251-258 (`streamWrite`)
- **What:** `BytesIO::seek` sets `pos` to any value up to INT64_MAX with no clamp to the buffer size
  (`np < 0 ? 0 : np`, upper end unbounded). `truncate()` then does `b.buf.resize(b.pos)` with **no
  size check**. The 256 MiB `kMaxBuf` guard exists only in `streamWrite`; `truncate` bypasses it.
  So `seek(N)` + `truncate()` allocates N bytes. For N ≈ 2 GiB it silently succeeds (host OOM in a
  constrained embedding / worker VM); for N ≈ 9e18 it throws `std::bad_alloc`. Note the reported
  error is the generic `std::bad_alloc` rather than a clean actionable "BytesIO too large" — the
  contrast case (`write`) throws the good message.
- **Repro:**
  ```
  var io = import("io")
  var b = io.BytesIO()
  discard b.seek(9000000000000000000)
  discard b.truncate()          # -> error: std::bad_alloc (uncaught); catchable but wrong message
  # and the silent-OOM variant:
  discard b.seek(300000000); discard b.truncate()   # allocates 300 MB, size() == 300000000, no error
  ```
  Confirmed on release/ki-linux-x64: 9e18 → `t1.ki: error: std::bad_alloc`; 300 M → `size 300000000`.
  Contrast: `seek(9e18)` + `write("x")` correctly throws the catchable "BytesIO too large".
- **Confidence:** confirmed (reproduced)
- **Fix sketch:** In `truncate` apply the same `kMaxBuf` bound: `if (b.pos > kMaxBuf) throw
  KiritoError("BytesIO too large");` before `resize`. Better: hoist `kMaxBuf` to a class constant and
  clamp `pos` in `seek` itself (as the doc for `write`'s guard implies) so every op shares one bound.

### A10-2  sys.exit() calls std::exit() unconditionally — a `parallel` worker (or embedded host) tears down / DEADLOCKS the whole process
- **Severity:** High
- **Kind:** crash/UB / resource-exhaustion (deadlock)
- **Location:** src/kirito/stdlib_sys.hpp:150-161 (`exit`)
- **What:** `sys.exit` calls `std::exit(code)` directly. `std::exit` runs static/atexit destructors and
  terminates the OS process. Called from a **`parallel` worker thread** (workers get the full stdlib
  incl. `sys`) this runs those destructors — the thread-local object pool, the `KiritoDispatcher`,
  cross-VM primitives — **while the main thread and sibling workers are still executing**. That is
  the classic "`std::exit` from a non-main thread" hazard: destruction races live threads, so the
  result is nondeterministic — sometimes the process exits with the worker's code, sometimes it
  **hangs forever** (the dispatcher shutdown/join deadlocks against a half-destroyed runtime). The
  same footgun hits any **embedded host**: a `sys.exit` in a script kills the host process outright,
  bypassing GC/RAII and the VM-encapsulation guarantee (one `KiritoVM` = one process is violated —
  `sys.exit` reaches past the VM boundary to the OS).
- **Repro / failure scenario:**
  ```
  var parallel = import("parallel")
  var worker = Function():
      var s = import("sys")
      discard s.exit(7)
  var t = parallel.spawn(worker)
  discard t.join()          # racy
  ```
  Confirmed on release/ki-linux-x64: across 15 runs the process exits `7` about half the time and
  **hangs (had to be SIGKILLed by an outer `timeout`)** the other half — a batch of 12 runs blew the
  2-minute wall clock on accumulated 12 s hangs. Nondeterministic teardown/deadlock.
- **Confidence:** confirmed (reproduced — both the exit-7 and the deadlock outcomes)
- **Fix sketch:** `sys.exit` must not `std::exit` from a worker or an embedded VM. Options: throw a
  dedicated, *non-catchable* `SystemExit` sentinel that unwinds the VM to its top level (the `ki`
  main() then does the real `std::exit`), so a worker's exit only ends that worker (surfaced to
  `join`) and an embedded host regains control. At minimum, gate the `std::exit` behind "am I the main
  VM's main thread" and otherwise raise/return so the dispatcher's shutdown protocol runs cleanly.

### A10-3  BytesIO.seek silently CLAMPS a negative result to 0 while File.seek THROWS — divergent semantics in "interchangeable" streams (and a false code comment)
- **Severity:** Medium
- **Kind:** correctness-bug / DRY
- **Location:** src/kirito/stdlib_io.hpp:334-336 (`BytesIO::seek`) vs 222-223 (`FileVal::seek`)
- **What:** The two stream types are documented as interchangeable, but their `seek` disagree on a
  negative resulting position. `FileVal::seek` throws `seek: resulting position is negative`;
  `BytesIO::seek` does `if (np < 0) np = 0;` — silently clamps to 0 and returns 0. So the same code
  redirected from a File to a BytesIO changes behaviour (a bug that surfaces as a throw on disk but a
  silent cursor-reset in memory). Worse, the FileVal comment at :222-223 asserts *"BytesIO::seek
  guards the same way"* — which is **factually wrong**: BytesIO does NOT guard the same way.
- **Repro / failure scenario:**
  ```
  var io = import("io")
  var b = io.BytesIO("hello")
  io.print(b.seek(-3))          # -> 0  (silent clamp)
  var g = io.open("f.txt", "r")
  io.print(g.seek(-3))          # -> throws "seek: resulting position is negative"
  ```
  Confirmed on release/ki-linux-x64.
- **Confidence:** confirmed (reproduced)
- **Fix sketch:** Pick one policy and share it. Preferably make BytesIO throw the same
  `seek: resulting position is negative` (matches File and the "strict, actionable errors" house
  style); at minimum correct the FileVal comment. A shared `resolveSeek(base, off, whence)` helper
  used by both `seek` impls would DRY the overflow-check + negative-check + whence-validation (all
  three are duplicated between the two classes today) and guarantee they can't drift again.

### A10-4  createprocess/shell capture UNBOUNDED child stdout/stderr with no size cap — OOM, and the bad_alloc escapes a drain std::thread → std::terminate (uncatchable SIGABRT)
- **Severity:** High
- **Kind:** resource-exhaustion / crash/UB / missing-guard
- **Location:** src/kirito/proc_compat.hpp:188-197 (`drainFd`) & 251-252 (drain-thread lambdas);
  Windows equivalents at :82-86 / :148-149. Consumed by sys.createprocess/shell (stdlib_sys.hpp:33-49).
- **What:** The stdout/stderr drain threads `res.out.append(...)` / `res.err.append(...)` with **no
  upper bound** — a child that emits unbounded (or merely very large) output grows the capture buffer
  until the host runs out of memory. Two failures compound:
  (1) **No output guard** — unlike the house policy (bound output, THROW cleanly), there is no cap and
  no way to kill the child once it has produced too much.
  (2) **The OOM is uncatchable.** `res.out.append` runs **inside a `std::thread`** whose lambda has no
  `try/catch`; when it throws `std::bad_alloc`, the exception escapes the thread function →
  `std::terminate` → **SIGABRT**. So this is not even a catchable Kirito error — the whole interpreter
  aborts. A Kirito `try/catch` around `sys.shell(...)` cannot save it.
- **Repro / failure scenario:**
  ```
  var sys = import("sys")
  try:
      var r = sys.shell("yes ABCDEFGH")   # infinite stdout, no timeout
      io.print(len(r["stdout"]))
  catch String as e:
      io.print("threw:", e)               # never reached
  ```
  Confirmed on release/ki-linux-x64 under `ulimit -v 800000`:
  `terminate called after throwing an instance of 'std::bad_alloc'` → exit 134 (SIGABRT). The
  `try/catch` does NOT catch it. (Without the vmem limit it OOM-kills / grows until the machine dies.)
- **Confidence:** confirmed (reproduced)
- **Fix sketch:** Add a configurable max-capture cap (e.g. default 256 MiB per stream, matching the
  BytesIO/Bytes `kMaxBuf`). When a drain exceeds it, stop appending, signal the run to kill the child
  (SIGKILL the process group / TerminateJobObject) and surface a clean catchable
  `ProcError("process output exceeded N bytes")` → KiritoError. Independently, wrap each drain-thread
  body in `try/catch(...)` so a `bad_alloc` (or any exception) can never escape a `std::thread` and
  abort the process — record it and let the main thread rethrow it as a KiritoError after join.

### A10-5  `for line in io.stdin:` / `for line in file:` is EAGER — it buffers the ENTIRE stream to EOF before the loop body runs once (defeats streaming; OOM on a large file)
- **Severity:** Medium
- **Kind:** correctness-bug / resource-exhaustion / doc
- **Location:** src/kirito/stdlib_io.hpp:122-129 (`FileVal::iterate`), 409-415 (`StdStream::iterate`)
- **What:** The iterate protocol returns a fully-materialized `std::vector<Handle>`, so iterating a
  stream reads **everything until EOF** and builds the whole line List *before* the first loop
  iteration executes. For `io.stdin` this means `for line in io.stdin:` cannot process input
  incrementally — it blocks until the input closes, then replays every line at once. That contradicts
  the in-code doc ("`for line in io.stdin:` reads stdin line-by-line") and breaks interactive /
  `tail -f` / pipeline-streaming use. For a large file, `for line in f:` loads the entire file into
  memory as N String objects at once (OOM on a multi-GB log), where the whole point of line iteration
  is bounded memory.
- **Repro / failure scenario:**
  ```
  var io = import("io"); var time = import("time")
  var t0 = time.monotonic()
  for line in io.stdin:
      io.print("t=" + String(round(time.monotonic()-t0, 2)), line)
  ```
  Fed `one; sleep 1; two; sleep 1; three`, all three print at **t=2.0** (after EOF), not at
  t=0/1/2 — proving the whole stream is buffered before iteration. Confirmed on release/ki-linux-x64.
- **Confidence:** confirmed (reproduced)
- **Fix sketch:** Streams need a *lazy* iteration path. The general iterate() protocol returns a
  vector, but a stream should instead drive the for-loop through a pull-based `_iter_`/`next` step
  (read one line per iteration) so memory stays bounded and stdin is processed as it arrives. If a
  full lazy-iterator protocol is out of scope for v1.12, at minimum fix the doc comment (it promises
  streaming that doesn't happen) and document the eager behaviour + memory cost.

## Test gaps (C++ and .ki)
_(methods/attributes/arguments with no or weak coverage; adversarial/edge cases missing)_
- **BytesIO.truncate / seek beyond size** — no test exercises `seek(N)`+`truncate()` past the buffer
  (A10-1) nor the `kMaxBuf` bound on truncate. Add a `.ki` test asserting a clean "BytesIO too large".
- **Negative-seek divergence** — no test pins File-throws vs BytesIO-clamps (A10-3). Once unified, add
  a shared expectation for both.
- **sys.exit** — untested from a `parallel` worker and from an embedded VM (A10-2). Add a test that a
  worker's `sys.exit` does not tear down the process / does not deadlock (needs the fix first).
- **Unbounded child output** — no test for `sys.shell`/`createprocess` capturing huge output (A10-4);
  add one that a runaway producer is killed and surfaces a *catchable* error (not SIGABRT).
- **Stream iteration** — no test that `for line in io.stdin` / `for line in file` streams (A10-5);
  add a timing/large-file test once lazy iteration lands (or a test that documents the eager cost).
- **Unsupported open modes** — `"w+"`, `"rb+"`, `"x"` (exclusive create) are rejected (only
  `r/w/a/r+` + trailing `b`). No error-suite (`errors/*.ki` + `.experr`) test pins the "unsupported
  file mode" message or documents which modes are intentionally out.
- **walk/listdir permission tolerance** — cannot be reproduced as root in this sandbox, but
  `recursive_directory_iterator`/`directory_iterator` are constructed with an `ec` (tolerant *at
  construction*) yet the range-for uses the **throwing** `operator++`; a permission-denied *subdir*
  encountered mid-recursion in `walk` would throw a `filesystem_error` (a `std::exception`, only
  caught by a *bare* `catch`), contradicting the documented "inaccessible dir lists as []" tolerance
  and lacking `directory_options::skip_permission_denied`. Plausible Low; add a non-root test +
  consider `skip_permission_denied`.
- **Negative `timeout`** to createprocess/shell silently means "no timeout" (`timeoutSecs > 0` gate) —
  untested; arguably should reject a negative timeout.
- **getsize** on a directory throws (good) but no `.ki` test pins it; huge (>2^63) file → int64
  overflow to negative untested (unrealistic).

## DRY / single-source-of-truth notes
_(logic duplicated across files that should be unified)_
- **seek resolution duplicated** — `FileVal::seek` (stdlib_io.hpp:201-227) and `BytesIO::seek`
  (:323-337) each independently implement whence-validation + `__builtin_add_overflow` +
  negative-target handling, and they **diverge** (File throws, BytesIO clamps — A10-3). Extract one
  `resolveSeek(base, off, whence)` so the policy is single-sourced and cannot drift.
- **`kMaxBuf` (256 MiB) is a local constant** inside `BytesIO::streamWrite` (:252). It is NOT shared
  with `truncate` (A10-1 gap) and the same "bound an in-memory capture" magic number should back a
  proc-output cap (A10-4). Hoist to a single named class/namespace constant reused by all three.
- **print/eprint/write stringify+join** — the "for each arg: append (sep) + stringify" loop is copied
  three times in `IoModule::setup` (print :536-540, eprint :551-555, write :580-581). One
  `joinStringify(args, sep)` helper would DRY them (only the separator + trailing newline differ).
- `runExternalProcess` correctly single-sources createprocess/shell — good, no change.
