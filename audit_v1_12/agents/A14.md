# A14 — net

**Files in scope:** stdlib_net.hpp, net_compat.hpp (+ deflate.hpp for the bomb path)
**Status:** done
**Probe binaries:** `release/ki-linux-x64` (fast, TLS on), `dist/debug-asan` (memory/UB).
Hostile servers under scratchpad `probe/A14/` (loopback only). Findings appended continuously.

## Findings

### NET-1  Cross-origin redirect leaks the Authorization header AND cookies (no host/scheme scoping)
- **Severity:** High
- **Kind:** correctness-bug / security (credential leak, CWE-200)
- **Location:** src/kirito/stdlib_net.hpp:752-800 (`netRequest` redirect loop); cookie jar has no domain scope; `hdrs` (incl. `Authorization`) is rebuilt-once and re-sent on every hop.
- **What:** When a server replies `3xx` with `Location:` pointing at a **different host, port, or scheme**, the client re-sends the *same* `Authorization` header and the *entire* cookie jar to the redirect target. There is no origin check (requests strips `Authorization` on cross-host redirect and scopes cookies by domain/path). This also covers the **https→http downgrade**: an https response redirecting to `http://` re-sends Basic-auth credentials and session cookies over cleartext. The redirect target is arbitrary and server-controlled, so a single hostile/compromised endpoint (or an open-redirect on a trusted one) exfiltrates the caller's credentials. `allowredirects` defaults to `True`.
- **Repro / failure scenario (CONFIRMED):** base server on `:20081` returns `302 Location: http://127.0.0.1:20082/landing` + `Set-Cookie: sess=secret123`. Client: `net.get("http://127.0.0.1:20081/redir", {"auth": ["alice","s3cr3t"]})`. Target `:20082` (a *different* origin) logged: `AUTH='Basic YWxpY2U6czNjcjN0'  COOKIE='sess=secret123'` — both leaked across the origin boundary. Scripts: `probe/A14/hostile.py`, `redir.ki`.
- **Confidence:** confirmed (reproduced).
- **Fix sketch:** On each redirect compare the new `parseUrl` origin (scheme+host+port) to the previous request's origin; when it differs, (a) drop the `Authorization` header, and (b) do not send cookies whose (future) domain/path doesn't match — at minimum drop the jar cookies seeded from a different origin. Also strip `Authorization` on any scheme downgrade. Match requests' `rebuild_auth`/`rebuild_proxies` semantics. A cheaper interim fix: strip `Authorization` and skip the cookie header whenever `net::parseUrl(newUrl).host/port/tls` differs from the original request's.

### NET-2  CRLF injection in request line and headers (method + user-supplied header values unvalidated)
- **Severity:** Medium (High if header/method values ever flow from untrusted data)
- **Kind:** missing-guard / security (request smuggling / header injection, CWE-93)
- **Location:** src/kirito/stdlib_net.hpp:756-770 (request assembly: `method + " " + target ...`, `req += k + ": " + v + "\r\n"`). `parseUrl` (line 155) rejects control chars in the *URL* but nothing validates the **method**, header **names/values**, or the body.
- **What:** A `\r\n` embedded in the request `method` or in any `headers=` value is written verbatim into the wire request, letting the caller inject arbitrary extra headers or a second request line (request smuggling). The URL got a control-char guard; the parallel request fields did not.
- **Repro / failure scenario (CONFIRMED):**
  - Header: `net.get(url, {"headers": {"X-Test": "a\r\nInjected: pwned"}})` → server received a *separate* header `Injected: pwned` (RAWHDRS shows `'X-Test: a', 'Injected: pwned'`).
  - Method: `net.request("GET / HTTP/1.1\r\nX-Smuggle: yes", url)` → server's request line became `GET / HTTP/1.1` with a smuggled `X-Smuggle: yes ...` header. Scripts: `probe/A14/inj.ki`, `methinj.ki`.
- **Confidence:** confirmed (reproduced).
- **Fix sketch:** Reject `\r`/`\n` (and other control bytes) in the method, in every header name and value, and in cookie names/values before assembly (throw a clear error, like `parseUrl` does for the URL). Mirror the existing URL guard into a shared `assertNoControlChars(field)` helper reused for URL/method/header/cookie.

### NET-3  `timeout=` is a per-recv idle timeout, not a total deadline — slow-drip peer defeats it
- **Severity:** Medium
- **Kind:** resource-exhaustion (self-DoS) / doc mismatch
- **Location:** src/kirito/stdlib_net.hpp:127-141 (`setTimeout` sets `SO_RCVTIMEO`/`SO_SNDTIMEO`); recv loop `recvAll` (114-125) and the TLS `SSL_read` loop.
- **What:** `SO_RCVTIMEO` bounds a *single* `recv()` call, not the whole exchange. A server that drips one byte just under the timeout interval keeps every `recv()` returning in time, so the client never throws and the request runs for `~kMaxRecvAll * timeout` seconds (effectively unbounded in time, bounded only in memory by the 256 MiB `kMaxRecvAll`). The comment at line 127 claims it makes "a stalled peer throw instead of hanging forever" — true only for a *fully* stalled peer, not a slow-drip (slowloris) one.
- **Repro / failure scenario (CONFIRMED):** `net_abuse_server.py` `/slow-body` drips 20 bytes at 80 ms each (1.6 s total). `net.get(".../slow-body", {"timeout": 0.3})` **completed** after 1.6 s (read all 20 bytes) instead of throwing at 0.3 s. Script: `probe/A14/to.ki`. (Contrast: the *fully*-stalled server in `test_net.cpp:493` correctly throws — that case is covered, the drip case is not.)
- **Confidence:** confirmed (reproduced).
- **Fix sketch:** Track a wall-clock deadline (`timeout` from the first byte / request start) and throw once it passes, independent of per-syscall progress; or use a non-blocking socket + `poll()` with a shrinking deadline. At minimum, document that `timeout` is a per-operation idle timeout, not a total request budget.

### NET-4  connect() ignores `timeout=` / `settimeout()` — a blackholed host hangs on connect
- **Severity:** Medium
- **Kind:** missing-guard / resource-exhaustion
- **Location:** src/kirito/stdlib_net.hpp:464-466 & 538-539 (`fd = dialTcp(u); setTimeout(fd, timeout);` — the timeout is applied *after* the blocking connect in `dialTcp`, 417-440); `SocketVal::connect` (880-891) also uses a blocking `::connect` and `settimeout` only sets `SO_RCVTIMEO/SNDTIMEO` (which do not affect `connect()`).
- **What:** The `timeout` option (and `Socket.settimeout`) bound send/recv but never the TCP handshake. A connect to an unreachable/SYN-dropped address blocks for the OS default (~1–3 min) regardless of `timeout=1`.
- **Repro / failure scenario:** `net.get("http://10.255.255.1/", {"timeout": 1})` blocks far longer than 1 s on connect (not reliably reproducible on loopback in the sandbox — reasoned from the call order: connect happens inside `dialTcp` before any timeout is set).
- **Confidence:** plausible (static, ordering is unambiguous).
- **Fix sketch:** Do a non-blocking connect + `poll()`/`select()` bounded by `timeout` in `dialTcp` and in `SocketVal::connect` (set `O_NONBLOCK`, `connect`, `poll` for writability within the deadline, then restore blocking mode). This is the standard connect-timeout idiom.

### NET-5  IPv6 Host header emitted without brackets (`Host: ::1`, not `Host: [::1]`)
- **Severity:** Low
- **Kind:** correctness-bug (RFC 7230 §5.4 violation)
- **Location:** src/kirito/stdlib_net.hpp:175-182 (`parseUrl` strips the brackets into `u.host`) then 756 (`"Host: " + u.host`).
- **What:** For an IPv6 URL `http://[::1]:8080/`, `u.host` = `"::1"` (brackets removed for `getaddrinfo`), and the request emits `Host: ::1`, which is malformed — an IPv6 literal in Host must be bracketed. A strict origin server rejects it.
- **Repro / failure scenario:** IPv6 loopback unavailable in the sandbox (`AF_INET6` unsupported), so confirmed by code reading, not execution.
- **Confidence:** confirmed (by code); repro blocked by environment.
- **Fix sketch:** Track "is IPv6 literal" on `Url` (or re-detect a `:` in `u.host` with no other host chars) and emit `Host: [" + u.host + "]` (plus `:port` only when non-default). Keep `u.host` bracket-free for `getaddrinfo`.

### NET-6  HTTP `parseUrl` doesn't strip `user:pass@` userinfo (inconsistent with `splitUrl`)
- **Severity:** Low
- **Kind:** correctness-bug / DRY
- **Location:** src/kirito/stdlib_net.hpp:149-199 (`parseUrl`) vs 246-277 (`splitUrl`, which does `rfind('@')` to drop userinfo).
- **What:** `parseUrl` (used by the actual HTTP client) leaves `user@host` in `u.host`, so `http://user@host/` sets `u.host = "user@host"` and fails to resolve — whereas `urlsplit` correctly reports `host`. Two URL parsers with divergent authority handling. Not directly exploitable (it errors rather than confusing the target host, so no userinfo-based SSRF), but it's a latent inconsistency and a DRY violation.
- **Repro / failure scenario:** `net.get("http://user@127.0.0.1:PORT/")` → "could not resolve host 'user@127.0.0.1'"; `net.urlsplit("http://user@h/")` → host `h`.
- **Confidence:** confirmed (by reasoning + the splitUrl code path).
- **Fix sketch:** Add the same `rfind('@')` userinfo strip to `parseUrl`; better, unify both parsers onto one authority-splitting helper (scheme/userinfo/host/port/IPv6) used by `parseUrl`, `splitUrl`, and `resolveUrl`.

### NET-7  A malformed chunk size silently truncates the body to empty (no error)
- **Severity:** Low
- **Kind:** correctness-bug (silent data loss)
- **Location:** src/kirito/stdlib_net.hpp:312-329 (`dechunk`): `long sz = strtol(sizeHex,...,16)` returns 0 for a non-hex size, then `if (sz <= 0) break;`.
- **What:** A `Transfer-Encoding: chunked` body whose first chunk-size line is non-hex (e.g. `NOTAHEX`) decodes to an **empty** body with no diagnostic — the client can't distinguish "server sent nothing" from "server sent a corrupt frame". (`strtol` overflow of a huge hex size clamps to `LONG_MAX`; on 64-bit the subsequent `i + (size_t)sz > body.size()` guard still holds and `std::string::append` clamps, so there is **no** OOB/OOM — only the silent-empty case is a real defect.)
- **Repro / failure scenario (CONFIRMED):** `net_abuse_server.py` `/bad-chunked` → `status=200 bodylen=0`. Script: `probe/A14/mal.ki`.
- **Confidence:** confirmed (reproduced).
- **Fix sketch:** Distinguish "no hex digits consumed" (via `strtol`'s end pointer) from a legitimate `0` terminator and surface a decode error (or fall back to returning the raw body) instead of silently emptying it.

### NET-8  No default timeout — a fully-stalled peer hangs the caller indefinitely
- **Severity:** Nit / Low
- **Kind:** missing-guard (doc/ergonomics)
- **Location:** src/kirito/stdlib_net.hpp:736 (`double timeout = 0.0;` default → `setTimeout` no-ops for `<= 0`).
- **What:** With no `timeout` option, `SO_RCVTIMEO` is not set, so `net.get(url)` blocks forever on a server that accepts and stalls. Matches requests' default (no timeout) but is a foot-gun; the only backstop is `kMaxRecvAll` (which requires data to keep flowing). Interacts with NET-3/NET-4.
- **Confidence:** confirmed (by code).
- **Fix sketch:** Document prominently; optionally ship a sane non-zero default (e.g. 30 s) for the high-level HTTP client.

## What held up (verified, no defect)
- **gzip/deflate decompression bomb is capped, not OOM.** `deflate::inflate` bounds output at `kMaxInflateOut` = 256 MiB (deflate.hpp:224-283) and throws `DeflateError`; `parseRaw` swallows it (line 409) leaving the body as the raw compressed bytes. Probe: a 400 MiB-of-zeros gzip (`/bomb`, Content-Encoding: gzip) → `net.get().content` returned ~407 KB (the compressed bytes), no OOM/crash. Script: `probe/A14/bomb.ki`.
- **`recvall` / HTTPS read are capped** at `kMaxRecvAll` = 256 MiB (shared plain-TCP + TLS constant) and throw on exceed (lines 122, 524).
- **Huge / negative Content-Length is harmless** — the client ignores `Content-Length` entirely (always `Connection: close`, reads to EOF), so `Content-Length: 99999999999999999999` and `Content-Length: -100` neither over-allocate nor crash (`/hugecl`,`/negcl` → correct short bodies).
- **Redirect loop is bounded** by `maxredirects` (default 10); `/loop` with `maxredirects:5` returned the final 302, no infinite loop.
- **Socket state machine is memory-safe under ASan/UBSan:** double `close()`, send/recv **after close**, `detach()` then use of the stale handle, `fromfd(999999)`/`fromfd(-1)` — all throw cleanly, **zero ASan/UBSan reports**. Script: `probe/A14/sock.ki`.
- **URL / Location control-char guard works:** `parseUrl` rejects raw CR/LF/control bytes in the URL and (re-parsed) redirect `Location`, closing the redirect-based header-injection path (NET-2 is only reachable via caller-supplied method/headers, not via a server `Location`).
- **Garbage / non-HTTP response** (`/garbage`) parses to `status=0`, empty body, no crash.

## DRY / single-source-of-truth notes
- **Two URL parsers** (`parseUrl` HTTP-only vs `splitUrl` general) duplicate the IPv6-bracket + port-split logic and **disagree** on userinfo (NET-6). `resolveUrl` re-derives origin a third way. Unify onto one authority-splitting helper.
- **Control-char validation** exists for the URL only; the method/header/cookie fields need the same check — extract one `assertNoControlChars` helper (NET-2).
- `base64Encode` (line 292) is a local reimplementation; the `base64` ki-module and `hash` also encode base64 — acceptable (native glue), but note the duplication.
- `asciiLower` (306) is a private helper; several stdlib modules carry their own ASCII-lower — candidate for a shared util (minor).
- `kMaxRecvAll` (net) and `kMaxInflateOut` (deflate) are two separate 256 MiB constants at different layers — fine as-is, but worth a shared "256 MiB resource ceiling" named constant referenced by both (and by string/list guards) for one-place tuning.
