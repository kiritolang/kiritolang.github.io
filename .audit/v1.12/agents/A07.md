### A07-1  `InstanceValue::equals` bad `static_cast` when `other` is a native object (also `ValueKind::Instance`)
- **Severity:** High
- **Kind:** crash/UB
- **Location:** src/kirito/runtime.hpp:1746-1764 (`InstanceValue::equals`), reached via collections.hpp:101 (`probeBucket`)
- **What:** `InstanceValue::equals` guards `other` only by `if (other.kind() != ValueKind::Instance) return false;` and then does
  `const auto& rhs = static_cast<const InstanceValue&>(other);` reading `rhs.selfHandle`. But **every `NativeClass` also reports
  `ValueKind::Instance`** (native.hpp:152) — DateTime, Bytes, Matrix, Regex, … So when a user instance with `_eq_` is compared against a
  *native* object, the `static_cast` is a wrong-type downcast (UB), and `rhs.selfHandle` reads whatever bytes live at that offset in the
  native object, yielding a garbage `Handle` that is then passed to the Kirito `_eq_` method as `other`. `kiEquals` is safe (it routes
  through `viaEq` → `_eq_` first), but Dict/Set key comparison calls the raw `Object::equals` protocol directly via `probeBucket`
  (collections.hpp:101), so it hits the unguarded cast. The C++ wrappers `List::contains` / `Set::discard` (value.hpp:585,785) and
  `Value::equals` (value.hpp:227) also call raw `Object::equals`, same exposure from the embedding API.
- **Repro / failure scenario:** UBSan-confirmed. Buckets are keyed by the full `size_t` hash, so an attacker-controlled `_hash_` returns
  the native's known hash to force a same-bucket collision:
  ```
  class C:
      var _hash_ = Function(self): return 6142509188972423790   # == std::hash<string>("") (empty Bytes)
      var _eq_ = Function(self, other): return True
  var d = {}
  d[C()] = 99
  d has key in bucket 6142509188972423790; then:  Bytes("") in d
  ```
  `dist/debug-asan` reports: `runtime.hpp:1759:28: runtime error: member access within address ... which does not point to an object of
  type 'InstanceValue'  ...  object is of type 'kirito::BytesVal'`. (Release happened to return `True` — silent UB.)
- **Confidence:** confirmed (reproduced under UBSan)
- **Fix sketch:** replace the `kind()` gate with a real RTTI check:
  `const auto* rp = dynamic_cast<const InstanceValue*>(&other); if (!rp) return false;` then use `rp->selfHandle`. (The same
  `dynamic_cast` idiom is already used elsewhere, e.g. `applyBinaryOp`/`checkPrivateAccess`.) Consider auditing the C++ `List::contains`
  / `Set::discard` / `Value::equals` wrappers to route instance comparisons through `kiEquals` for symmetry with the language.

### A07-2  Comparison dunders may return non-Bool → C++ `Value::operator==/<` bad cast + `==`/`!=` inconsistency
- **Severity:** Medium
- **Kind:** crash/UB (embedding) + correctness-bug
- **Location:** src/kirito/runtime.hpp:2087-2101 (`applyBinaryOp` Eq/Ne + InstanceValue::binary), src/kirito/value.hpp:998-1021 (`Value::operator==/!=/</<=/>/>=`)
- **What:** `_eq_`/`_lt_`/`_le_`/`_gt_`/`_ge_` return values are **not** checked to be Bool (unlike `_bool_`→Bool, `_len_`→Integer,
  `_hash_`→Integer, which ARE enforced). So `a == b` returns the raw dunder result. Two problems:
  (1) **C++ embedding UB.** `Value::operator==` (value.hpp:999-1001) does `static_cast<const BoolVal&>(vm_->arena().deref(h)).value()`
  on the `applyBinaryOp(Eq,...)` result. If `_eq_` returned a non-Bool (e.g. an Integer), this is a wrong-type downcast → UB. Same for
  `operator</<=/>/>=` casting `_lt_`/… results.
  (2) **`==` vs `!=` inconsistency in pure Kirito.** `a == b` returns the raw `_eq_` result (any type), but `a != b` is *synthesized*
  as `makeBool(!truthy(_eq_))` (runtime.hpp:2093-2096) → always a Bool. So the two operators disagree on return type for the same class.
- **Repro / failure scenario:** confirmed behaviourally:
  ```
  class C:
      var _init_ = Function(self, v): self.v = v
      var _eq_ = Function(self, other): return self.v
  a = C(7); b = C(9)
  type(a == b)  ->  Integer   (value 7)
  type(a != b)  ->  Bool
  ```
  A host that compares these two instances via the C++ `Value::operator==` would hit the bad `static_cast<const BoolVal&>` (UB); the
  `_lt_` returning a String case likewise. (Pure-Kirito conditionals are safe because they route through `truthy()`.)
- **Confidence:** confirmed (behavioural for the language inconsistency; reasoned/UBSan-class for the C++ cast — same bad-cast pattern as A07-1)
- **Fix sketch:** either (a) coerce comparison-dunder results to Bool at the `InstanceValue::binary`/`applyBinaryOp` boundary
  (`makeBool(result.truthy())`) so `==` matches `!=` and the C++ cast is always valid, or (b) enforce `_eq_`/`_lt_`/… must return a Bool
  (mirroring `_bool_`/`_len_`/`_hash_`) with a clear error. Option (a) is the additive, Python-incompatible-but-consistent choice;
  the codebase already synthesizes `!=` via truthy, so (a) is the smaller, self-consistent change. Independently, make the C++
  comparison operators defensive (`deref(h).truthy()` instead of the BoolVal cast) so the embedding API can never UB.

### A07-3  Private-member access is asymmetric across the class chain — subclass method cannot read a base instance's private
- **Severity:** Medium
- **Kind:** correctness-bug (spec violation)
- **Location:** src/kirito/runtime.hpp:2053-2060 (`checkPrivateAccess`, the `isInstanceOf(vm, obj, currentClass)` test)
- **What:** CLAUDE.md: a private `_member` is "accessible only from within a method of the same class **or a subclass** (privacy is per
  class *chain*, not per defining class)". The check is `if (hasCurrentClass && isInstanceOf(vm, obj, currentClass)) return;` — i.e. it
  only allows access when **`obj` is-a `currentClass`** (obj is an instance of the running method's class or a subclass of it). That makes
  privacy directional: a **base** class method can read a **derived** instance's private (`obj` is-a base ✓), but a **derived** class
  method canNOT read a **base** instance's private (`obj` is bare-base, not is-a derived ✗) — even though the derived class is in the very
  chain that owns the member. The doc conditions on the *accessing context* (the running method's class being in the owner's chain), not
  on the object's dynamic type.
- **Repro / failure scenario:** confirmed:
  ```
  class A:
      var _init_ = Function(self): self._x = 10
      var reachDown = Function(self, other): return other._x
  class B(A):
      var reachUp = Function(self, other): return other._x
  A().reachDown(B())  ->  10                       (base method reads derived private: allowed)
  B().reachUp(A())    ->  THROWS "cannot access private member '_x' of 'A' outside its class"
  ```
  A subclass method that legitimately takes a base-typed argument and touches an inherited private is wrongly rejected.
- **Confidence:** confirmed (reproduced)
- **Fix sketch:** allow access when the running method's class and the object are in the same lineage in *either* direction — e.g.
  `if (hasCurrentClass && (isInstanceOf(vm, obj, currentClass) || isSubclassOf(currentClass, classOf(obj)))) return;`, or more precisely:
  allowed iff `currentClass` is an ancestor-or-equal of `obj`'s class **or** a descendant of it (the two share a chain that "owns" the
  name). Since there is no name-mangling, the intent is purely "is the running method entitled to see names private to this hierarchy".

### A07-4  `Value::items()` over a String returns dangling views (fresh char Strings unrooted after iterate returns)
- **Severity:** Medium
- **Kind:** crash/UB (dangling handle) — C++ embedding API
- **Location:** src/kirito/value.hpp:216-223 (`Value::items()`); root cause src/kirito/runtime.hpp:1007-1023 (`StrVal::iterate`)
- **What:** `StrVal::iterate` allocates a **fresh** 1-code-point `String` per character, protected only by an *internal* `RootScope rs`
  that is destroyed when `iterate` returns. `Value::items()` then wraps each returned handle in a plain, **non-pinning** `Value` view
  (`out.emplace_back(*vm_, e)`). So after `items()` returns, the per-char Strings are reachable **only** through those non-pinning views —
  nothing the GC can see roots them. The next allocation that triggers a collection sweeps them, leaving the `Value`s holding dangling
  handles. The pin machinery (pinHandle / PinnedHandle) exists precisely to prevent this class of bug, but `items()` bypasses it for
  fresh-alloc iterables. (Lists/Sets/Dicts are safe: their `iterate` returns *existing* element handles already rooted as container
  children — confirmed by a control probe.)
- **Repro / failure scenario:** confirmed with a minimal embedding program under ASan/UBSan:
  ```cpp
  KiritoVM vm; vm.setGcThreshold(1);
  String s(vm, std::string("hello"));
  std::vector<Value> chars = s.items();                 // fresh char Strings, now unrooted
  for (int i=0;i<1000;i++){ Handle h=vm.makeString(std::string(64,'x')); }   // force GC
  for (auto& c : chars) acc += c.str();                 // -> throws "dangling handle (stale generation)"
  ```
  (The same pattern with `List xs(vm,{1..5})` survives — isolating the bug to fresh-alloc iterate.)
- **Confidence:** confirmed (reproduced under ASan via a standalone embed program)
- **Fix sketch:** make `Value::items()` keep the iterated objects alive for the returned vector's lifetime — e.g. build the results into a
  temporary arena List (or pin each handle) and return `Value`s that share that root, or return pinning `Value`s for fresh-alloc sources.
  Alternatively `StrVal::iterate` could return views onto an interned/backing store. At minimum document that `items()` results over a
  String must be consumed before any further allocation (fragile). The List/Set/Dict paths need no change.

## Test gaps (C++ and .ki)
- **A07-1:** no test compares a user instance carrying `_eq_`/`_hash_` against a *native* `ValueKind::Instance` object (DateTime, Bytes,
  Matrix, …) inside a Dict/Set. Add a `.ki` regression that stores such an instance and queries with a colliding-hash native (must not UB;
  UBSan gate).
- **A07-2:** no test asserts the return type of `a == b` / `a < b` for a class whose comparison dunder returns a non-Bool, nor the
  `==` vs `!=` return-type consistency. No C++ test exercises `Value::operator==` on two such instances.
- **A07-3:** privacy tests cover self-access and same-class-pair access but NOT the cross-hierarchy direction (a subclass method reading a
  base-typed instance's inherited private). `tools/tests/errors/` should gain both the allowed and rejected directions.
- **A07-4:** `Value::items()` is exercised widely over Lists but there is **no** embedding test that iterates a **String** via `items()`
  and holds the result across an allocation/GC. Add a C++ test (setGcThreshold(1)) that must survive; also cover Set/Dict `items()` under GC.
- General: `PinnedHandle` has `test_pinned_handle.cpp` (185 lines) but I did not verify it stresses copy/move + GC interleaving with a
  low GC threshold; worth confirming a copied-then-moved PinnedHandle keeps its object across a forced sweep.
- `_hash_`/`_eq_` reentrancy: the ProbeScope guard ("Dict changed size during a key comparison") and the `_hash_`-recursion→CallGuard path
  both HELD under ASan in probes, but I did not find a dedicated regression `.ki` — add one so the guard can't silently regress.
- Module-member rebinding to a non-stream (`io.stdout = 5`) is handled (duck-typed getAttr throw) but appears untested as an adversarial case.

## DRY / single-source-of-truth notes
- **Keyword-argument binding is implemented THREE times** with near-identical "match named→param slot, reject unknown/duplicate, fill
  defaults/report missing" logic: `NativeFunction::bindArgs` (function.hpp:78-102), the lambda inside `makeMethod` (native.hpp:170-192),
  and the named-arg loop in `KiFunction::callFull` (runtime.hpp:1986-2012). They differ only in default handling (native params carry
  a Handle default; Kirito params evaluate a default expr in the call scope). Factoring the slot-matching/error core into one helper would
  remove the triplication and guarantee identical diagnostics.
- **C++ equality wrappers bypass the shared `kiEquals`.** `List::contains` (value.hpp:582-588), `Set::discard` (value.hpp:780-789) and
  `Value::equals` (value.hpp:226-228) call the raw `Object::equals` protocol directly, whereas the language's `in`/`==` route through
  `kiEquals` (which safely handles instance `_eq_` in both directions). This duplication is exactly what exposes A07-1 and means the C++
  `.contains()` does not honor `_eq_` symmetrically like Kirito's `in`. The operator surface already delegates to `applyBinaryOp` (good);
  the membership/equality wrappers should likewise delegate to `kiEquals`.
- Negative-index normalisation is re-implemented in each wrapper's `operator[]` (`String`, `Bytes`, `List`) and again in the low-layer
  protocol; minor, but a shared `normIndex(i, n)` would DRY it.
