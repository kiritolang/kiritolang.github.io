### A08-1  Thread-local small-object pool LEAKS every worker thread's free-list at thread exit (unbounded leak under `parallel.spawn`)
- **Severity:** High
- **Kind:** resource-exhaustion (memory leak) / test-gap
- **Location:** src/kirito/pool.hpp:40 (`inline thread_local FreeBlock* freeLists[kClasses] = {};`),
  interacting with dispatcher.hpp:559-560 / 668 (each `spawn` = a fresh `std::thread` running a fresh
  `KiritoVM worker;` to completion, then the thread exits).
- **What:** The pool is a `thread_local` array of raw free-list head pointers. Blocks handed to
  `pool::deallocate` are pushed onto `freeLists[class]` and **never returned to `::operator delete`**.
  When a worker thread finishes, the `thread_local` array is destroyed *trivially* (it is an array of
  raw pointers), so the entire chain of `FreeBlock`s it heads is **leaked** ‚Äî that memory is never
  reachable again by any thread and is never returned to the process allocator. Every worker VM parks
  its whole pooled footprint (baseline builtins/stdlib + all transient boxes GC'd during the task) on
  its thread's free-list before `~KiritoVM`, so **each `spawn` leaks ‚âà the worker's peak pooled-block
  memory**. A program that spawns workers in a loop (a server spawning a worker per request, a batch
  pipeline) grows memory without bound even though only one worker is ever live at a time.
  Because the sanitizer builds **bypass the pool** (pool.hpp:25-28), LeakSanitizer uses plain
  `::operator new/delete` and sees balanced alloc/free ‚Äî so this leak is **invisible to the entire
  sanitizer/CI suite**. It is a release-only, undetected, unbounded leak.
- **Repro / failure scenario:** `scratchpad/probe/A08/spawn_worker.ki` spawns+joins N workers
  sequentially (only one live at a time). Peak `VmHWM` sampled from `/proc`:
  - **release (pool active):** N=50 ‚Üí 49 MB, N=200 ‚Üí 180 MB, N=800 ‚Üí 694 MB, N=2000 ‚Üí **1733 MB**
    ‚Äî grows ~linearly at ~0.87 MB/spawn, unbounded.
  - **asan (pool bypassed):** N=30 ‚Üí 143 MB, N=120 ‚Üí 345 MB, N=300 ‚Üí **354 MB** ‚Äî plateaus (glibc
    reuses each retired thread's arena; the block chain is not lost).
  The linear-vs-plateau split isolates the cause to the pool: the pool's blocks are permanently lost
  at thread exit, whereas direct `::operator delete` memory is reused by later worker threads.
- **Confidence:** confirmed (reproduced; the asan/release contrast pins it to the pool).
- **Fix sketch:** Return a worker thread's pooled blocks to the OS at thread exit. Options:
  (a) make each per-class free-list head a member of a `thread_local` object with a non-trivial
  destructor that walks each list and `::operator delete`s every block; (b) track all fresh blocks in
  a `thread_local` slab/vector freed at thread exit; (c) cap the free-list length per class (bounded
  pool) and `::operator delete` the overflow so steady-state retention is bounded. Whichever is
  chosen, add a release-mode test that spawns many workers and asserts peak RSS stays bounded (the
  sanitizer suite cannot cover this because it bypasses the pool). The same trivial-destruction path
  means the **main thread's** free-list is never returned either ‚Äî fine at process exit, but it makes
  peak RSS a high-water mark that never shrinks (see A08-3).

### A08-2  Dispatcher retains every `Task` for the whole program lifetime (never erased after join)
- **Severity:** Medium
- **Kind:** resource-exhaustion (unbounded growth)
- **Location:** src/kirito/dispatcher.hpp:558 (`tasks_[id] = std::move(task);`) ‚Äî no erase anywhere;
  `joinTask`/`taskDone` only look tasks up, `shutdown()` walks them but the map is cleared only when
  the dispatcher itself is destroyed (program end).
- **What:** `tasks_` is a `map<uint64_t, unique_ptr<Task>>` that accumulates one entry per `spawn`
  forever. Each retained `Task` keeps a (joined but still-constructed) `std::thread`, the result blob
  `std::string`, and error text. This is small per task (metadata + result) next to A08-1, but it is
  still unbounded in the *main* VM and grows with the number of spawns ‚Äî a long-running server that
  spawns per request accretes Task records without limit. A joined Task is never needed again once its
  result has been read.
- **Repro / failure scenario:** the same `spawn_worker.ki` loop grows `tasks_` to N entries; combined
  with A08-1 it is part of the linear RSS growth, though the pool dominates. Reasoned from the code:
  there is no `tasks_.erase(...)` on any path except dispatcher destruction.
- **Confidence:** confirmed (by inspection ‚Äî no erase path exists).
- **Fix sketch:** Erase a Task from `tasks_` once it has been reaped **and** its result consumed by
  `joinTask` (reference-count join/consume so a `done()`-poller doesn't drop it early). The
  join()/shutdown() race is already handled by `reap()`. Add a test asserting the task count does not
  grow after joined tasks are consumed.

### A08-3  Pool free-lists never shrink ‚Äî retained memory is a permanent high-water mark
- **Severity:** Low
- **Kind:** resource-exhaustion (retention)
- **Location:** src/kirito/pool.hpp:44-61 ‚Äî `deallocate` only pushes onto the free-list; nothing ever
  calls `::operator delete` on a pooled block, and the free-lists have no length cap.
- **What:** A transient burst that boxes, say, 10M short-lived values grows the free-lists to hold 10M
  blocks; after the burst those blocks stay parked on the thread-local free-lists for the life of the
  thread and are never returned to the OS. Peak RSS therefore only ever ratchets up. On the main
  thread this is a documented tradeoff ("pure speed win") but it means a single spike permanently
  inflates a long-running process's footprint even after it idles. Distinct from A08-1 (an outright
  *loss* of the blocks at thread exit); this is retention on a *live* thread.
- **Repro / failure scenario:** allocate a very large list of temporaries in a loop, let GC reclaim
  them, then idle ‚Äî RSS does not fall back. (Behavioural; follows from the code: no path frees a
  pooled block.)
- **Confidence:** confirmed (by inspection).
- **Fix sketch:** Cap each size class's free-list length (e.g. a few thousand blocks) and
  `::operator delete` beyond the cap, so steady-state pool retention is bounded while still absorbing
  churn. This also bounds A08-1's magnitude even without the thread-exit walk.

## Analysis notes (examined, NOT bugs ‚Äî recorded for triage + the test-gap map)

### Generation-wraparound handling is correct but has NO test
- `ObjectArena::sweep()` (arena.hpp:62-67) retires a slot permanently (leaves it `occupied=false`,
  OFF the free-list) instead of wrapping its generation from `UINT32_MAX` to `0` ‚Äî which would let a
  long-lived stale `Handle{slot, UINT32_MAX}` re-validate against a freshly reused slot (an ABA/UAF).
  Correct: a retired slot with `occupied=false` fails both `markIfUnmarked` (occupied check) and
  `at()` (occupied check), and `Handle{}` (gen 0) can never collide because `kFirstGen == 1`. The
  reserved-gen-0 sentinel + gen-1 first generation is sound: `none_` is the first alloc ‚Üí `{0,1}`, so
  `Handle{}` = `{0,0}` never aliases it.
  **Reachability of wraparound:** to wrap one slot's generation you must free+reuse *that same slot*
  2^32 times. With the LIFO free-list and the default 100 000-alloc GC threshold a slot's generation
  climbs ~1 per GC cycle, i.e. ~2^32 √ó 10^5 ‚âà 4√ó10^14 allocations ‚Äî not reachable, and safe if it
  were. **Test gap:** the retirement branch (arena.hpp:62-67) is dead in every test; a white-box unit
  test (friend access or a testing hook presetting a slot's generation to `UINT32_MAX-1`) should
  exercise "free at max gen ‚Üí slot retired, never reissued, old handle stays dangling".

### Value-model hash contract (equal ‚áí equal hash) is upheld across the Integer/Float boundary
- The brief points at "hashing.hpp ‚Äî equal values hash equal", but `hashing.hpp` is actually the
  crypto MD5/SHA-1/SHA-256 digests, not the value model. The value hash contract lives in each type's
  `hash()` (`builtins.hpp`) + `std::hash<Handle>` (`handle.hpp`). Verified the one genuinely risky
  case ‚Äî cross-type numeric equality ‚Äî is correct: `FloatVal::hash()` (runtime.hpp:275-283) special-
  cases an integral float in `[-2^63, 2^63)` to hash **identically to the equal Integer**, matching
  `intFloatEqual`/`compareIntFloat` EXACTLY, so `1 == 1.0` ‚áí equal hash, and `2^53+1` vs the float
  `2^53` stay distinct in both `==` and hash. `-0.0`/`+0.0` both hash to `hash<int64>(0)` and are
  `==`. `NaN` is never `==` anything (documented multi-key behaviour). Bool/None/String never
  cross-type-equal a number, so no contract break. No finding.

### `children()` coverage is complete for every Handle-owning Object I checked
- Verified each Object subclass that stores Handles reports them to the marker: ListVal/DictVal/SetVal
  (collections.hpp), EnvValue (environment.hpp), ModuleValue (module.hpp), KiFunction/NativeFunction
  (function.hpp ‚Äî captures + signature defaults), ClassValue/InstanceValue/SuperValue
  (class_value.hpp), IterCursor (bytecode_vm.hpp), TensorVal (stdlib_tensor.hpp ‚Äî `node->parents`;
  `grad` is a raw `Tensor<double>`, not a Handle), ResponseVal/SessionVal (stdlib_net.hpp),
  MatchVal (stdlib_regex.hpp). Default-constructed `Handle{}` children (e.g. an unset `headersH`) are
  harmless: `markIfUnmarked({0,0})` fails the generation check against None's `{0,1}`.
  **Follow-up for A12 (out of my scope):** `TensorGradNode::backward` is a `std::function` that may
  capture Handles NOT listed in `parents`; if any autograd backward closure captures a tensor Handle
  outside `parents`, it would be a missing root. Flag for the tensor auditor.

### Pool size-class math is correct at the boundaries
- `classOf(n) = (n+15)/16 - 1`: n‚àà[1,16]‚Üí0 ‚Ä¶ n=224‚Üí13 (14 classes, indices 0..13); n=225‚Üí`n>kMaxPooled`
  ‚Üí bypass to `::operator new`. A fresh block for class c is exactly `(c+1)*16` bytes ‚â• any n mapping
  to c, so every block on `freeLists[c]` fits any class-c request. new-size == delete-size for a given
  concrete type (the sized `operator delete` under the virtual dtor gets the complete-object size), so
  alloc/free map to the same class ‚Äî no cross-class corruption. **Test gap:** no C++ unit test
  directly exercises `pool::classOf` / boundary sizes (223/224/225 B) or alloc‚Üífree‚Üírealloc block
  reuse. Alignment: `::operator new(size)` yields 16-aligned memory and `kAlign==16`; safe only while
  no pooled Object subclass has `alignof > 16` (true today, nothing enforces it ‚Äî a
  `static_assert(alignof(T) <= pool::kAlign)` would harden it).
