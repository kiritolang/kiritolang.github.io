# A18 — regex

**Files in scope:** regex_engine.hpp, stdlib_regex.hpp
**Status:** DONE (read-only audit complete)
**Probe binaries:** `release/ki-linux-x64` (fast), `dist/debug-asan` (memory/UB).

## Verdict up front
The regex area is **remarkably well-hardened** — one of the strongest in the tree. Every Wave-1
theme (linear-time guarantee, guards that THROW rather than crash/OOM/overflow) holds under every
adversarial probe I could construct:
- Classic catastrophic patterns `(a+)+b`, `(a|a)*b`, `(a*)*c` on long inputs: **sub-millisecond**
  (linear). A 50,000-way alternation compiles+searches in **0.33 s**.
- Every quantifier/program-size/group-count/nesting explosion **throws a clean, catchable
  RegexError** ("repetition count too large (max 1000)", "compiled pattern too large", "too many
  capture groups (max 1000)", "pattern nested too deeply") before allocating unbounded memory or
  recursing off the native stack.
- **3,000 randomized pattern×subject cases run under ASan/UBSan → zero crashes, zero UB, zero
  hangs.** Targeted malformed patterns, zero-width/empty matches, Unicode, pos/endpos/count edge
  cases, and deep-nesting compile+match are all ASan-clean.

Findings below are minor (one DRY nit, test-coverage gaps). No correctness/crash/resource bug found.

## Findings

### A18-1  `compileRepeat` has a dead identical if/else branch
- **Severity:** Nit
- **Kind:** DRY
- **Location:** src/kirito/regex_engine.hpp:490-499 (`compileRepeat`, lines 493-495)
- **What:** For an unbounded `{n,}` repeat both arms of the branch are byte-for-byte identical:
  ```cpp
  if (n.rmax == -1) {
      if (n.rmin == 0) compileStar(body, n.greedy);
      else compileStar(body, n.greedy);   // same call
  }
  ```
  The `if (n.rmin == 0)` split is dead — both call `compileStar(body, n.greedy)`. (The `n.rmin`
  mandatory copies are already emitted by the loop above, so the tail is a plain `*` in both cases,
  which is correct — the branch is just redundant.)
- **Repro / failure scenario:** No behavioural bug — `a{2,}`/`a{0,}` compile and match correctly
  (verified). Purely a code smell / reviewer trap.
- **Confidence:** confirmed (reproduced; behaviour correct, branch redundant)
- **Fix sketch:** Collapse to a single `compileStar(body, n.greedy);` and drop the inner `if`.

### A18-2  Program-size emit cap ("compiled pattern too large") is untested
- **Severity:** Low
- **Kind:** test-gap
- **Location:** src/kirito/regex_engine.hpp:428 (`emit`: `if (p_.insts.size() >= 200000) throw`)
- **What:** This 200k-instruction emit cap is a **distinct** guard from the repetition-count cap
  (`a{5000}` → "repetition count too large", tested) and the group-count cap (`"()"*2000` → "too
  many capture groups", tested). Nothing in the suite exercises the emit cap. It is the guard that
  catches program blow-up from *products* of bounded repeats on a group, e.g. `(a{1000}){1000}`
  which yields ~10^6 instructions.
- **Repro / failure scenario:** `re.compile("(a{1000}){1000}")` → throws
  `invalid regex: compiled pattern too large` (verified on release + asan). No regression test pins
  this, so a future refactor of `emit`/`compileRepeat` could silently drop it and let a pathological
  pattern OOM the compiler.
- **Confidence:** confirmed (guard fires; grep of tools/tests shows no coverage)
- **Fix sketch:** Add a one-liner (C++ `CHECK_THROWS` + `.ki`) asserting
  `re.compile("(a{1000}){1000}")` throws with "compiled pattern too large".

- **Kind:** test-gap

## Test gaps (C++ and .ki)
Coverage here is genuinely strong (test_regex.cpp, test_regex_corpus.cpp, test_regex_deep.cpp,
plus many `.ki` golden scripts: audit_regex, spec_regex*, r4/r6/r7/r8 families — including a
chrono-timed `(a+)+b` linearity assertion, a structured 3,000-iter engine fuzz, empty-match/
zero-width cases, `\g<...>` templates, group-count cap, repetition cap, nesting cap, pos/endpos,
and negative-count-sub which is **intentional** and pinned: "sub with a non-positive count
replaces all"). Remaining gaps, all low:
- **Emit cap** ("compiled pattern too large") — see A18-2. The one guard with no test at all.
- **No `.ki`/timed test for a huge alternation** (e.g. 5,000–50,000-way `a|a|...`): the C++ suite
  times `(a+)+b` but not the alternation-fan-out linear case. Cheap to add.
- **sub callable returning non-String / throwing**: verified working (throws "sub replacement
  function must return a String"; a thrown error propagates), but I did not find a C++/`.ki`
  assertion for the *non-String return* path specifically — worth a pin.
- **Unicode class range** (`[а-я]`, `[^a]` under IGNORECASE) and **astral-plane literal** (`😀`
  match/span) verified correct by probe; add a golden assertion so the code-point layer stays pinned.

## DRY / single-source-of-truth notes
- A18-1 above (dead branch in `compileRepeat`).
- `expandTemplate` (stdlib_regex.hpp) and the parser's escape decoding (`decodeEscapeChar`) each
  implement their own `\n \t \r \\`-style escape handling. They serve different grammars
  (replacement template vs. pattern) with different rules (template has `\1`/`\g<>`, pattern has
  `\d`/`\b`/octal/hex), so a merge would be forced and is **not** recommended — noting only that the
  small overlap is intentional divergence, not accidental duplication.
- The three size/complexity caps live in three places by design and cover three orthogonal blow-up
  axes (repetition count → parser; capture groups → `reng::compile`; total instructions → `emit`).
  This is correct layering, not a DRY violation.

## Probe log
All under scratchpad/probe/A18. `io.print` (not bare `print`).
- **Linearity (release):** `(a+)+b` on 40a+X, `(a|a)*b` on 50a, `(a*)*c` on 5000a → all `None`,
  <5 ms each. `a{1000}` on 1000a → 29 ms. 50,000-way alternation (built via `"|".join`) compile+
  search → 0.33 s. (An earlier 16.9 s reading was Kirito's *quadratic string concat* building the
  pattern, NOT the regex — confirmed by isolating the build loop.)
- **Caps throw cleanly:** `a{1001}`→"repetition count too large"; `a{1000000}{1000000}`→same;
  `(a{1000}){1000}`→"compiled pattern too large"; `"()"*n` group cap→"too many capture groups";
  nesting: `(`×N…`)`×N compiles at N≤999, throws "pattern nested too deeply" at N≥1000 (holds under
  ASan; matching a valid 990-deep pattern is ASan-clean — compiler recursion + Node teardown stay
  bounded by the parser's depth guard).
- **Malformed patterns** (18 cases): `(`,`)`,`[z-a]`,`[a`,`*a`,`a**`,`\`,`a\x1`,`(?=a)`,`(?<=a)`,
  `(a)\1`,`(?P<n>a)(?P<n>b)`,`a{2,1}`,`(?P=n)`,`[]`,`[^]`,`\x1G` → each a clean RegexError. `a{`
  is lenient (literal `{`), matching Python.
- **Zero-width/empty** in findall/finditer/sub/split: no infinite loop; results match Python 3.7+
  empty-adjacent semantics (`re.sub("a*","-","aabaa")` → `--b--`). `findall("a*", "a"*200000)` →
  0.035 s (linear).
- **sub replacement:** callable→non-String throws; callable throw propagates; template out-of-range
  group `\2` throws; unmatched optional group → `""` (template) / `None` (`group`/`groups`);
  `\g<name>`, `\g<0>`, `\g<99>`, `\g<99999999999999999999>` (stoi-overflow path), `\g<>` all throw
  cleanly, no UB.
- **pos/endpos/count:** pos past-end→None, negative pos→0, endpos truncation/past-end, negative
  endpos, negative/zero count — all ASan-clean, no OOB.
- **Unicode:** dot per code point, code-point spans, `\w` ASCII-only (documented), Latin-1 case-
  insensitive (`É`~`é`), Cyrillic class range, astral emoji, `\b` split — all correct, ASan-clean.
- **Fuzz:** 3,000 random pattern×subject pairs (metachar-rich alphabet incl. `()[]{}|*+?^$\.-,`
  and Unicode/newline subjects) exercising compile/findall/sub/split/search/group/groups under
  `dist/debug-asan` → **0 crashes, 0 UB, 0 timeouts**.
