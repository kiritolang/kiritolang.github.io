# A11 — math / random

**Files in scope:** stdlib_math.hpp, stdlib_random.hpp
**Status:** done
**Probe binaries:** `release/ki-linux-x64` (fast), `dist/debug-asan` (memory/UB).
Probes in `scratchpad/probe/A11/`. Values cross-checked against Python `math`/`random`.

**Overall:** both modules are unusually clean. The math DOMAIN-ERROR-THROWS policy is
comprehensive and correct on every function I probed (sqrt/log/log2/log10/log1p/asin/acos/
acosh/atanh/gamma/lgamma/pow/fmod), NaN passes through, overflow-to-inf is left alone, and
every value matches Python bit-for-bit. Random: reproducibility per engine, randint inclusive,
randrange excludes stop + correct for negative/multi step, empty/negative-k guards, choice/
choices/sample/shuffle guards, and EXACT-stream serialize/dump round-trip all verified. No
Critical/High/Medium bug found — findings are Low/Nit (consistency, DRY, guard hygiene) plus
test-gap fills.

## Findings

### A11-1  `math.trunc` returns Float while `floor`/`ceil` return Integer
- **Severity:** Low
- **Kind:** correctness-bug (consistency) / doc
- **Location:** src/kirito/stdlib_math.hpp:83 (`unary("trunc", std::trunc)`)
- **What:** `trunc` is registered through the `unary` helper, which declares return type
  `"Float"` and returns `Value(vm, f(x))` — so `math.trunc(3.7)` -> `3.0` (Float) and
  `inspect(math)` shows `trunc(x: Number) -> Float`. Its siblings `floor`/`ceil` return
  Integer (lines 165–170). This diverges both from those siblings and from Python
  (`math.trunc` -> `int`). A caller doing integer-index math with `trunc` silently gets a Float.
  (Side effect: `trunc(1e300)` returns `1e+300` where `floor(1e300)` throws "result out of
  Integer range" — so trunc is more permissive on huge inputs, which is the one point in its
  favour and may be the reason it was left as Float.)
- **Repro:** `math.trunc(3.7)` -> `3.0`; `type(math.trunc(3.7))` -> `Float`; `type(math.floor(3.7))` -> `Integer`.
- **Confidence:** confirmed (reproduced)
- **Fix sketch:** decide the contract. Either route `trunc` through the same `toInt64Checked`
  path as floor/ceil (returns Integer, throws on out-of-range) for Python/sibling parity, or
  keep it Float **deliberately** and document the divergence in CLAUDE.md + the stdlib docs.
  Whichever, add a test asserting the chosen return type (currently untested — see gaps).

### A11-2  `mag` / integer-extraction lambdas duplicated across gcd/lcm/factorial/comb/perm
- **Severity:** Low
- **Kind:** DRY
- **Location:** src/kirito/stdlib_math.hpp:113–115 & 188–190 (`mag`); 107–111, 183–187, 172–174, 229–233 (`get`/`geti`)
- **What:** The magnitude helper that avoids `std::abs(INT64_MIN)` UB (`v < 0 ? 0ull -
  (uint64_t)v : (uint64_t)v`) is copy-pasted verbatim in both `gcd` and `lcm`. The
  "deref, require ValueKind::Integer, return `.value()`" extraction lambda is written four
  times (gcd, lcm, factorial, combPerm) with only the error-message name varying. Three-or-more
  repetitions per the DRY directive — should be one file-scope helper each (e.g. a free
  `i64mag(int64_t)` and a `requireInt(vm, h, who)`).
- **Confidence:** confirmed
- **Fix sketch:** hoist `i64mag` and `requireInt` to file scope; call from all sites.

### A11-3  Dead arity checks + inconsistent domain-error message wording
- **Severity:** Nit
- **Kind:** DRY / dead-code / doc
- **Location:** src/kirito/stdlib_math.hpp:57 (`unary`), 138, 149, 153 (pow/atan2/hypot)
- **What:** Every signatured native (all of `math`) has its arity enforced by the signature
  binder BEFORE the impl runs — `math.sqrt()` -> "missing required argument 'x'",
  `math.pow(1,2,3)` -> "takes at most 2 positional arguments". So the internal
  `if (a.size() != 1) throw ...` (unary) and `if (a.size() != 2) throw ...` (pow/atan2/hypot)
  are unreachable dead code, and applied inconsistently (fmod/copysign/degrees/… omit them).
  Separately, the domain-error text differs by function: the `unary` helper emits
  `"NAME: math domain error (got X)"` while `log`/`pow`/`fmod` emit
  `"NAME: math domain error (<reason>)"` — cosmetic inconsistency in an otherwise uniform policy.
- **Confidence:** confirmed
- **Fix sketch:** drop the redundant `a.size()` guards (binder covers them); optionally unify the
  domain-error message shape.

### A11-4  `uniform`/`gauss` don't validate bounds/sigma (non-finite / negative)
- **Severity:** Nit
- **Kind:** missing-guard
- **Location:** src/kirito/stdlib_random.hpp:133–141 (uniform), 142–150 (gauss)
- **What:** `uniform(a, b)` passes `a`,`b` straight to `std::uniform_real_distribution<double>`
  with no finiteness / `a<=b` check: `uniform(0, inf)` -> `inf`, `uniform(nan, 1)` -> `nan`,
  `uniform(-inf, inf)` -> `-nan`. `gauss(mu, sigma)` passes a possibly-negative `sigma` to
  `std::normal_distribution`, whose standard precondition is `stddev > 0` — a negative sigma is
  technically a precondition violation (UB), though libstdc++ tolerates it (just mirrors the
  draw) and Python's `random.uniform`/`gauss` are equally unguarded. No crash under ASan/UBSan.
  Given the Wave-1 theme "guards THROW not crash", this is the mild outlier where a garbage
  numeric (`nan`) is returned instead of thrown; kept Nit because it matches Python and can't
  crash.
- **Repro:** `random.Random(1).uniform(0-math.inf, math.inf)` -> `-nan`.
- **Confidence:** confirmed
- **Fix sketch:** optionally reject non-finite `uniform` bounds and `sigma < 0` (choose to
  match Python's permissiveness or Kirito's throw-on-garbage stance — document either way).

## Test gaps (C++ and .ki)

Existing coverage is strong (spec_domain_guards.ki, audit_math.ki, spec_math.ki,
test_domain_guards.cpp, test_random.cpp, errors/math_domain_*.{ki,experr},
errors/random_choice_empty, errors/random_choices_negative_k, random_generators.ki,
random_prng_quality.ki). Gaps found:

- **`trunc` return type** — no test asserts `type(math.trunc(x))`; add one once A11-1 is decided.
- **`lgamma` domain** — `gamma(0)` is domain-tested but `lgamma(0)`/`lgamma(-1)` (same pole guard)
  is not; add an errors/ case.
- **`log2`/`log10` domain** — negative/zero domain throw is exercised via spec_domain_guards but
  not as a standalone errors/*.experr like sqrt/asin/log/pow.
- **`fmod(x, 0)`** domain throw — verified by probe, no dedicated errors/ case.
- **`gcd`/`lcm` overflow guards** — `gcd(INT64_MIN, 0)` and `lcm(INT64_MIN, 1)` both throw
  "result too large for Integer"; add a test (magnitude = 2^63 can't be represented — a real
  edge, currently untested).
- **`comb`/`perm`/`prod`/`factorial` overflow boundaries** — `comb(66,33)` (fits, 7.2e18) vs
  `comb(67,33)` (throws), `factorial(20)` (fits) vs `factorial(21)` (throws), `perm(21)` (throws),
  `prod` int-overflow-throws vs float-in-mix-no-throw — verify with a golden test at the exact
  boundary (values confirmed == Python).
- **`randrange` negative & multi step** — `randrange(10,0,-2)` yields exactly `{2,4,6,8,10}` and
  `randrange(1,20,3)` yields `{1,4,7,10,13,16,19}` (== Python `range()`); the unsigned
  step arithmetic (stdlib_random.hpp:183–196) has no golden test for negative step.
- **`randrange` single-arg / kwarg forms** — `randrange(5)` == `randrange(0,5)`, `randrange(stop=5)`,
  `randrange()`->throws; not covered.
- **`sample`/`choice` from non-List iterables** — `sample("abcdef", 3)`, `choice({1:2,3:4})`
  work (iterate-based); untested.
- **`Random`/`seed` require Integer** — float/string seed throws ("Random seed expected
  Integer"), `.seed(3.14)` throws; untested.
- **`choices` has no `weights` parameter** — `choices([...], weights=[...])` -> "unexpected
  keyword argument 'weights'". This is a deliberate divergence from Python `random.choices`
  (CLAUDE.md documents `choices(population, k=1)`, weight-free). Worth a doc note + a test that
  pins the current no-weights contract (so it's an intentional, tested decision, not an
  oversight). The audit brief's "negative/zero weights" hunt is therefore N/A by design.
- **serialize/dump exact-stream round-trip per engine** — verified for both xoshiro and
  mersenne_twister; ensure both engines (not just one) are covered in the serialization suite.

## DRY / single-source-of-truth notes

- **`mag` magnitude helper** duplicated in gcd & lcm (stdlib_math.hpp:113/188) — see A11-2.
- **Integer-extraction lambda** duplicated 4× (gcd/lcm/factorial/combPerm) — see A11-2.
- **`pi`** literal appears as `std::numbers::pi` in the `pi` constant (45), `tau` (47),
  `degrees` (160) and `radians` (163). Acceptable (compile-time constant), but a single
  `kPi`/`kDegPerRad` constant would make the conversion factors self-documenting.
- **Domain-check pattern** is well-centralized for unary functions via the `unary` helper's
  `ok` predicate (good DRY); the multi-arg checks (log/pow/fmod) are necessarily separate.
  Only inconsistency is message wording (A11-3).

## Probe log

- Domain policy: probed sqrt(-1), sqrt(nan)->nan, log(0)/log(-1), log2(0), log10(-5),
  log1p(-1)/log1p(-2), asin(2), acos(-2), acosh(0.5), atanh(±1), gamma(0)/gamma(-2)/gamma(nan)->nan,
  lgamma(0)/lgamma(-1), pow(-2,0.5), pow(0,-1), fmod(5,0), cbrt(-8)->-2, exp(1000)->inf. ALL correct:
  domain cases throw a clear "math domain error", NaN passes through, overflow-to-inf preserved.
- Return types: floor/ceil -> Integer, trunc -> Float (A11-1). floor(1e300)/floor(nan)/floor(inf)
  all throw via toInt64Checked (no UB cast). trunc(1e300) returns the Float (no throw).
- Integer funcs: gcd(-12,8)=4, lcm(0,0)=0, lcm(4,6)=12, gcd/lcm(INT64_MIN,·) throw "too large".
  factorial(20) ok, 21 throws. comb(64,32)/comb(66,33) fit (128-bit intermediate widening works),
  comb(67,33)/perm(21) throw. prod([]) =1, prod int-overflow throws, prod with a Float -> Float
  (no overflow). All values == Python.
- Random: xoshiro & mt reproducible from same seed; the two engines differ; `.generator` correct.
  randint inclusive [a,b], lo>hi throws, lo==hi ok. randrange excludes stop, step=0 throws, empty
  throws, negative/multi step produce exactly Python's range() member set. sample k>len / k<0 throw,
  k=0 -> []. choice empty throws. choices k<0 throws, no weights kwarg. expovariate<=0 throws.
  Seed must be Integer (float/string throw). serialize & dump round-trip continue the EXACT stream
  for both engines; `.seed()` reseed matches a fresh Random(seed).
- ASan/UBSan (dist/debug-asan): uniform with inf/nan bounds -> inf/nan/-nan (no crash), gauss
  negative sigma -> value (no crash), shuffle empty/single ok, _setstate_("bogus") throws cleanly,
  randint over full int64 range ok. No memory/UB report.
