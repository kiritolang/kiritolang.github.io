### A09-1  `pow(base, exp, mod)` — signed-overflow UB and wrong result for large modulus
- **Severity:** High
- **Kind:** crash/UB + correctness-bug
- **Location:** src/kirito/runtime.hpp:3322 (`pow` 3-arg modular branch)
- **What:** `__int128 ... b = ((base % mod) + mod) % mod;` — `base % mod` and `mod` are both
  `int64_t`, so the subexpression `(base % mod) + mod` is evaluated in `int64_t` **before** the
  outer `% mod` and before the widening to `__int128`. When `mod` is large (roughly `> 2^62`) and
  `base % mod` is a large positive value, that sum overflows `int64_t` → signed-overflow UB. The
  `__int128` type does not help because the overflow happens in the int64 sub-add.
- **Repro:** `pow(9223372036854775806, 2, 9223372036854775807)`
  - ASan/UBSan: `runtime.hpp:3322:72: runtime error: signed integer overflow: 9223372036854775806 + 9223372036854775807 cannot be represented in type 'long int'`
  - Release: prints `9`. Correct answer is `1` (base ≡ -1 mod M, so base² ≡ 1). So the UB also
    yields a **wrong result** in the shipped build.
- **Confidence:** confirmed (reproduced under ASan/UBSan + wrong release output)
- **Fix sketch:** compute the residue in `__int128`: `__int128 b = ((__int128)(base % mod) + mod) % mod;`
  (or `b = base % mod; if (b < 0) b += mod;` with `b` already `__int128`). Add a regression test with
  a modulus near INT64_MAX.

### A09-2  Dict/Set add-then-remove churn leaks outer-bucket entries (unbounded memory growth)
- **Severity:** Medium
- **Kind:** resource-exhaustion / missing-guard
- **Location:** src/kirito/collections.hpp:208 (`DictVal::remove`); src/kirito/runtime.hpp:841
  (`SetVal` remove), :864 (`discard`), :875 (`pop`)
- **What:** `DictVal`/`SetVal` are two-level: an outer `fum::unordered_map<size_t, vector<...>>`
  keyed by the key's `hash()`, with a per-hash inner vector. Every removal erases the element from
  the **inner** vector only — it never erases the (now empty) **outer** entry when the inner vector
  becomes empty. So a Dict/Set reused as a transient with distinct-hash keys (add, then remove) grows
  its outer map without bound even though the logical size stays 0. For a long-lived embedded VM
  (Kirito's stated design goal) this is a slow, silent leak.
- **Repro / failure scenario:**
  ```
  var d = {}
  for i in range(3000000):
      d[i] = 1
      d.remove(i)   # len(d) stays 0
  ```
  Measured RSS: churn Dict ≈ 575 MB, churn Set (add/discard) ≈ 571 MB, vs a no-container baseline
  loop ≈ 236 MB — ~340 MB of leaked empty outer-bucket entries with `len == 0`.
- **Confidence:** confirmed (RSS measured via `getrusage(RUSAGE_CHILDREN)` per run)
- **Fix sketch:** after `it->second.erase(...)` in each of the four sites, drop the empty outer
  entry: `if (it->second.empty()) buckets.erase(it);`. Factor into one shared helper (DRY — the same
  probe-erase-cleanup pattern is copy-pasted across `DictVal::remove`, `SetVal::remove`,
  `SetVal::discard`, `SetVal::pop`). (Iteration order is documented "unspecified", so this is a
  pure memory/hygiene fix, not an ordering fix.)

### A09-3  `Set(iterable)` / `Dict(iterable)` constructors don't GC-root iterated handles (GC sweep → dangling handle)
- **Severity:** High
- **Kind:** crash/UB (memory-safety) — caught by the arena generation net in shipped builds, but a valid op fails
- **Location:** src/kirito/runtime.hpp:2848 (`Set` constructor) and :2859 (`Dict` constructor)
- **What:** Both constructors do `auto items = ...iterate(vm); for (Handle h : items.value()) s.add(vm.arena(), h);`
  without GC-rooting the handles in `items.value()`. Those handles live only in a C++ local vector
  (not scanned by the collector). `s.add()`/`d.set()` run the key's `_hash_`/`_eq_` — arbitrary
  Kirito code that can allocate and trigger a collection. If the source iterable yields **fresh**
  hashables that are not reachable from any root (a String/Bytes/`range`, or — the dangerous case —
  a user class whose `_iter_` returns fresh instances that are dropped once `iterate()` copies the
  element vector), the not-yet-added elements are swept mid-loop. The very same hazard is explicitly
  guarded in `SetVal`'s `union`/`difference`/… (runtime.hpp:897-900 roots every iterated element with
  `rs.add(e)` and documents exactly this) — so the constructor paths are an inconsistent omission.
  `List(iterable)` (:2837) is correct — it *does* `rs.add(h)`.
- **Repro / failure scenario:** (reproduced under both ASan and release)
  ```
  class Item:
      var _init_ = Function(self, n): self.n = n
      var _hash_ = Function(self):
          var junk = []
          for i in range(300000): junk.append(i)   # forces a GC
          return self.n
      var _eq_ = Function(self, o): return self.n == o.n
  class Source:
      var _iter_ = Function(self):
          var xs = []
          for i in range(20): xs.append(Item(i))
          return xs
  var s = Set(Source())      # -> "dangling handle (stale generation)"
  ```
  Both `dist/debug-asan` and `release/ki-linux-x64` report `error: dangling handle (stale generation)`.
  The Dict form (`Source._iter_` yielding `[Item(i), i]` pairs, `Dict(Source())`) fails identically.
  The arena's generation check downgrades what would otherwise be a use-after-free into a thrown
  error, but a legitimate construction still fails.
- **Confidence:** confirmed (reproduced under ASan + release)
- **Fix sketch:** mirror `List()` / `union()`: add `for (Handle h : items.value()) rs.add(h);` before
  the add loop in the `Set()` constructor; in the `Dict()` constructor root `h` and both
  `pair.value()[0]`/`[1]` before `d.set(...)`.

### A09-4  `zip`/`map`/`filter`/`sorted`/`enumerate` don't root iterated fresh handles up front (GC sweep → dangling elements)
- **Severity:** High
- **Kind:** crash/UB (memory-safety) — caught by the arena generation net in shipped builds, but produces corrupt results
- **Location:** src/kirito/runtime.hpp — `zip` (:3099), `map` (:3117), `filter` (:3130),
  `sorted` (:3052), `enumerate` (:3077), `all` (:3168), `any` (:3176). (`all`/`any` don't even open a
  `RootScope`; they iterate `items.value()` and call `.truthy()` — a user `_bool_` that allocates
  sweeps the remaining unrooted elements. Confirmed: `all(Source())` with a fresh-instance `_iter_`
  and an allocating `_bool_` → dangling-handle error.)
- **What:** Same root cause as A09-3. Each builtin iterates `items.value()` (a plain C++ vector, not
  a GC root) and only roots elements **incrementally** as the loop reaches them — `zip` roots none of
  the column handles at all; `map`/`filter`/`sorted`/`enumerate` `rs.add(x)` only the *current*
  element. Any allocation before an element is rooted (a per-element `vm.alloc`, or a user
  `key`/`function`/`_hash_` call) can trigger a collection that sweeps the not-yet-rooted elements
  when the source yields **fresh** handles (a String/Bytes, or a user `_iter_` returning fresh
  values). The result is a List whose elements are dangling handles — silently corrupt until access.
  Contrast `min`/`max`/`reversed`, which root **all** iterated elements up front and are safe.
- **Repro / failure scenario:** (all reproduced; `release/ki-linux-x64` reports
  `error: dangling handle (stale generation)`)
  - `zip("a"*300000, "b"*300000)` then `z[0]` — the first column's chars are swept while the second
    column is being iterated.
  - `enumerate("a"*300000)` then `r[299999]` — later pairs hold swept chars.
  - `sorted("abcde", key=f)` where `f` allocates ~150k objects — the stable_sort comparison derefs a
    swept key → immediate dangling-handle error.
  - `map(f, "abcde")` / `filter(f, "abcde")` with an allocating `f`, then `"".join(r)` — result
    elements are swept.
- **Confidence:** confirmed (reproduced under ASan + release for all five)
- **Fix sketch:** root the whole iterated vector up front, exactly like `reversed`/`min`/`max`:
  `for (Handle h : items.value()) rs.add(h);` immediately after `iterate()` succeeds, before any
  processing. For `zip`, root every element of every column right after collecting `cols`. Consider
  a shared helper (e.g. `rootAll(rs, items)`) to make the pattern uniform (DRY) and hard to forget.

### A09-5  `Integer()` converter vs literal parser disagree on out-of-range values (DRY / consistency)
- **Severity:** Low
- **Kind:** DRY / correctness-consistency
- **Location:** src/kirito/runtime.hpp:2720-2766 (`Integer()` converter) vs src/kirito/parser.hpp:1016
  (`Parser::intLiteral`)
- **What:** Integer-literal-string → int64 is implemented **twice**, with divergent overflow
  behavior. The parser's `intLiteral` accumulates in `uint64_t` and reinterprets, so an over-wide
  literal **wraps two's-complement and never throws**. The `Integer()` converter uses `std::stoull`,
  which **throws `out_of_range`** for a magnitude ≥ 2⁶⁴ (surfaced as a Kirito error). So the same
  textual value is accepted-and-wrapped as a literal but rejected as a conversion.
- **Repro:**
  - `99999999999999999999` (literal) → `7766279631452241919`; `Integer("99999999999999999999")` → throws.
  - `0x1FFFFFFFFFFFFFFFF` (literal) → `-1`; `Integer("0x1FFFFFFFFFFFFFFFF")` → throws.
  (The in-range cases agree: `0xFFFFFFFFFFFFFFFF` → `-1` both ways.)
- **Confidence:** confirmed (reproduced)
- **Fix sketch:** decide one policy (wrap or reject) and share a single parse routine between the
  parser and the converter. The converter already goes to great lengths (sign/prefix/whitespace
  handling) to "mirror the lexer's intLiteral" — a shared helper would remove the drift risk entirely
  and is the single-source-of-truth the audit is asking for. (Note the converter's strictness on
  doubled sign/prefix/embedded-prefix is good and should be kept; only the overflow policy differs.)

## Test gaps (C++ and .ki)
- **`pow(base, exp, mod)` with a large modulus** — every existing test uses a tiny mod (5, 100, 1000).
  No test exercises mod near INT64_MAX, which is where A09-1 (signed-overflow UB + wrong result)
  lives. Add `pow(2**62, 2, INT64_MAX)`-style cases and check the exact residue.
- **Dict/Set add-then-remove churn (A09-2)** — no test reuses a Dict/Set across many
  distinct-key add/remove cycles and asserts bounded memory (or asserts `buckets` is compacted when a
  bucket empties). `test_collections*.cpp` has no add/remove churn test.
- **GC-rooting of iterated fresh handles (A09-3, A09-4)** — there is **no** collection/builtin test
  that combines a fresh-handle source (String/Bytes, or a user `_iter_` returning fresh instances)
  with an allocating callback / `_hash_` / `_bool_` under GC pressure. This whole class
  (`Set()`/`Dict()`/`zip`/`map`/`filter`/`sorted`/`enumerate`/`all`/`any`) is untested against
  mid-loop collection. A C++ test can force it deterministically with `vm.setGcThreshold(1)`.
- **`min`/`max` with `default=` and multiple positional args** — currently silently accepts
  `min(3,1,2, default=0)` (Python rejects it); no test pins this behavior either way.
- **`format` presentation types** — no test asserts that uppercase `E`/`F`/`G` and `c`/`n` are
  *rejected* (they are — "unknown format type"); worth a test so a future change doesn't silently
  add half-support. Also `format(x, ".")` / `".f"` (empty precision) → precision 0 is untested
  (Python errors here; Kirito silently treats as 0).
- **Integer/Float overflow-string convergence (A09-5)** — no test compares the literal parser and the
  `Integer()` converter on the same over-wide value; the divergence is untested.
- **`Bytes()` from a non-List iterable** (e.g. a Set/range of ints) is lightly covered; the
  element-range (0..255) and negative-count guards have tests, but the fresh-handle path isn't
  stressed under GC.

## DRY / single-source-of-truth notes
- **Integer-literal string → int64 is implemented twice** (`Parser::intLiteral` in parser.hpp:1016 and
  the `Integer()` converter in runtime.hpp:2720) with divergent overflow behavior — see A09-5. The
  converter's own comment says it is "mirroring the lexer's intLiteral", which is a manual-sync smell.
  Extract one routine.
- **The probe-then-erase-from-inner-bucket pattern** is copy-pasted across `DictVal::remove`
  (collections.hpp) and `SetVal` `remove`/`discard`/`pop` (runtime.hpp) — none of them compacts the
  emptied outer bucket (A09-2). One shared "erase element, drop empty bucket" helper would fix the
  leak in one place.
- **The "root every iterated handle before touching it" pattern** is present in some builtins
  (`reversed`, `min`/`max`, `List()`, Set `union`/`difference`) and *absent* in others
  (`Set()`/`Dict()` ctors, `zip`/`map`/`filter`/`sorted`/`enumerate`/`all`/`any`) — A09-3/A09-4. A
  single `rootAll(rs, items)` helper used uniformly right after every `iterate()` would make the
  correct pattern the path of least resistance and eliminate the whole bug class.
- `parseDouble` (common.hpp) is the shared Float parse and is correctly guarded by the `Float()`
  converter against hex-float syntax; but `parseDouble` itself still accepts `strtod`'s hex floats
  and `nan(...)`/`inf` forms — other callers (lexer float, json) rely on their own pre-guards. Not a
  bug in my scope, but worth a cross-cutting note for A01/A15: the "reject hex float" guard lives in
  the converter, not in `parseDouble`, so each caller must remember it.
