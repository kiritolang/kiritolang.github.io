# A01 — Lexer

**Files in scope:** lexer.hpp (context: common.hpp, exceptions.hpp, parser.hpp intLiteral)
**Status:** done (scope fully covered)
**Probe binaries:** `release/ki-linux-x64` (fast), `dist/debug-asan` (memory/UB).
Probe scripts live under scratchpad `.../probe/A01/`.

The lexer is **iterative** everywhere (no recursion in `number`/`identifier`/`stringLiteral`/
`handleIndentation`), so it cannot stack-overflow on deeply nested/huge tokens — deep nesting is
guarded downstream by the parser's DepthGuard. 3000 random + structured adversarial inputs (biased to
quotes/backslashes/tabs/high-bytes/NUL) produced **zero crashes** under the release binary, and every
targeted repro is ASan/UBSan-clean. Findings below are correctness/diagnostic bugs, not memory bugs.

## Findings

### A01-1  `\xHH` string escape emits a raw byte, breaking the code-point invariant (data loss)
- **Severity:** Medium
- **Kind:** correctness-bug
- **Location:** src/kirito/lexer.hpp:340 (`decodeEscape`, the `\x` branch:
  `return std::string(1, static_cast<char>(hi * 16 + lo));`)
- **What:** `\xHH` appends the single raw byte `HH` to the String's backing buffer. A Kirito `String`
  is documented as a **Unicode code-point** sequence, and its `len`/indexing/iteration decode the
  buffer as (lenient) UTF-8. So a `\xHH` with `HH >= 0x80` (a UTF-8 lead byte) **greedily merges with
  the following byte(s)** — whether they came from another `\xHH`, from a valid UTF-8 sequence, or
  from ordinary literal text — collapsing two logical characters into one and making the following
  character unaddressable. The single-byte intent ("`\xff` is one latin-1 byte", asserted by
  `r4_strings.ki:481` / `r8_language.ki:353`) only holds when the byte is *isolated*; it is silently
  violated the moment a lead-byte escape is adjacent to anything.
- **Repro / failure scenario:** (`.../probe/A01/loss.ki`, `utf.ki`)
  ```
  len("\xC3\x28")     -> 1   (Python: 2)   # the '(' (0x28) is swallowed into a bogus U+00E8
  ord("\xC3\x28"[0])  -> 232 (Python: 195) # wrong character
  len("\xC3\xA9")     -> 1   (Python: 2)   # two latin-1 escapes merge into one code point (é)
  len("\xC3(end")     -> 4   (Python: 5)   # a *literal* '(' after \xC3 becomes unaddressable
  ```
  A `\xC3` escape followed by ordinary text corrupts that text: `"\xC3(end"` keeps all 5 bytes but
  reports 4 code points, and index 0 is a 2-byte char, so `(` can never be reached by `[i]`/iteration.
- **Confidence:** confirmed (reproduced on release + ASan; cross-checked against CPython).
- **Fix sketch:** In `decodeEscape`, encode `\xHH` as the **UTF-8 encoding of code point U+00HH**
  (Latin-1 → Unicode is identity for 0–255) instead of a bare byte: `HH < 0x80` → one byte `HH`;
  `HH >= 0x80` → the two bytes `0xC0|(HH>>6), 0x80|(HH&0x3F)`. Then every `\xHH` is a self-contained,
  valid-UTF-8 code point, `len`/`ord` match CPython, adjacent escapes never merge, and the
  "one latin-1 code point per `\xHH`" intent is actually honoured. (This changes the internal byte
  layout of high-byte `\x` strings — update `r4_strings.ki:481`/`r8_language.ki:353` accordingly, and
  add the adjacent-escape regression from this finding. If raw-byte semantics are instead intended,
  the right literal is a byte-string, which the lexer does *not* provide — see Test gaps.)

### A01-2  Backslash at end-of-input in a cooked string: wrong message, embedded NUL, column past EOF
- **Severity:** Low
- **Kind:** correctness-bug (diagnostic)
- **Location:** src/kirito/lexer.hpp:329-357 (`decodeEscape`): it `advance()`s past the backslash then
  reads `e = peek()` **without an `atEnd()` check**; at EOF `peek()` returns `'\0'`, which falls to the
  `default` case at :351.
- **What:** A cooked (non-raw, non-f) string that is unterminated and ends exactly on a backslash
  (`"...\<EOF>`) reports `invalid escape '\<NUL>'` at a column **one past the last character**, instead
  of the correct `unterminated string`. The message also embeds a raw NUL byte (see A01-4). The real
  problem is that the string is unterminated, not that an escape is invalid.
- **Repro / failure scenario:** file containing exactly `var s = "abc\` (13 bytes, no newline):
  ```
  p14.ki:1:14: error: invalid escape '\        # col 14 is past EOF (backslash is col 13); message
                                               # truncates at an embedded NUL; should be "unterminated string"
  ```
  (`.../probe/A01/p14.ki`, revealed with `cat -v`.)
- **Confidence:** confirmed.
- **Fix sketch:** In `decodeEscape`, after the first `advance()` (and after the `\x` sub-advances),
  check `atEnd()` and throw the `unterminated` string diagnostic (thread it through, or have the caller
  detect EOF before dispatching to `decodeEscape`). Same guard fixes the backslash-before-newline case
  (A01-4).

### A01-3  Column tracking counts bytes, not code points — misaligns diagnostics on non-ASCII lines
- **Severity:** Low
- **Kind:** correctness-bug (diagnostic)
- **Location:** src/kirito/lexer.hpp:109-112 (`advance()` does `++col_` once per **byte**).
- **What:** `col_` advances one per source byte, so any line containing a multi-byte UTF-8 character
  reports column numbers that are byte offsets, not visual/code-point columns. This contradicts the
  rest of the language (String indexing/slicing/`len` are all code-point based) and makes an
  editor's caret land in the wrong place for error spans on lines with non-ASCII text. (Tabs are
  likewise counted as 1 column each — internally consistent, but not the visual column.)
- **Repro / failure scenario:** (`.../probe/A01/p6.ki` vs `p6b.ki`)
  `var x = "é" @` and `var x = "ee" @` both report `1:14: unexpected character '@'`, even though the
  `@` is at visual column 13 in the first (é is 2 bytes) and 14 in the second.
- **Confidence:** confirmed.
- **Fix sketch:** Advance `col_` per UTF-8 code point, not per byte (only bump `col_` on non-
  continuation bytes, i.e. bytes whose top two bits are not `10`). Low urgency but cheap; note it also
  affects every token span's `col`, not just error spans.

### A01-4  Diagnostics embed raw control bytes from the offending character (NUL truncates the message)
- **Severity:** Nit
- **Kind:** correctness-bug (diagnostic)
- **Location:** src/kirito/lexer.hpp:352-353 (`decodeEscape` default: `"invalid escape '\\" + e + "'"`)
  and :424 (`op()` default: `"unexpected character '" + c + "'"`).
- **What:** These messages interpolate the raw offending byte directly. When that byte is a control
  character the diagnostic is garbled; when it is **NUL** the whole message is silently truncated at
  the NUL (the error is surfaced via `KiritoError::what()` → `c_str()`), so the user sees a message
  that just stops. A backslash directly before a newline in a single-line string yields a message
  containing a **literal newline** (`invalid escape '\<LF>'`) rather than the clearer "unterminated
  string".
- **Repro / failure scenario:** (`.../probe/A01/p11.ki`, `p14.ki`, `p16.ki`)
  - `var s = "ab\<newline>cd"` → `invalid escape '\` + literal newline + `'`
  - embedded NUL as a stray char → `unexpected character '` then truncated.
- **Confidence:** confirmed.
- **Fix sketch:** Render non-printable bytes with an escape (`\xHH`, or a named form) when building
  these messages, e.g. a small `printableByte(char)` helper shared by both default arms. Combined with
  A01-2's `atEnd`/newline handling, the backslash-newline case should instead report "unterminated
  string".

## Test gaps (C++ and .ki)

- **`test_lexer.cpp` is skeletal (~75 lines).** It covers only: `42`, `3.5 + 1`, `1 // 2 ** 3 #`, one
  `CHECK_THROWS`, CRLF-vs-LF equivalence, and lone-CR. It has **no** direct coverage of:
  - String literals of any spelling (single/double/triple/raw/f, `rf`/`fr`), and no escape decoding
    (`\n \t \r \0 \\ \" \'`, `\xHH`) — these are exercised only end-to-end through the VM.
  - **Numeric base prefixes** `0x`/`0b`/`0o` at the token level, and the INT64_MIN / overflow /
    two's-complement-wrap literals (`9223372036854775808`, `0xFFFFFFFFFFFFFFFF`, over-wide decimal).
    Behaviour is correct (confirmed via probes p7/p8) but untested as a lexer unit.
  - **Error-message text and span (line/col/length)** for lexer diagnostics — current tests only
    `CHECK_THROWS`, never the message or the reported column.
- **Adjacent-high-byte `\xHH` is untested (A01-1).** `r4_strings.ki:481` / `r8_language.ki:353` /
  `spec_string_literals.ki:34` only assert the *isolated*/ASCII cases (`len("\xff")==1`,
  `"\x41\x42"=="AB"`). No test pins `len("\xC3\x28")` / `len("\xC3\xA9")` / `"\xC3(end"`, so the
  merge/data-loss bug slipped through. Add these once A01-1 is decided.
- **Backslash-at-EOF in a string (A01-2) is uncovered** — the error suite has `bad_escape`,
  `raw_trailing_backslash`, `unterminated_*`, `surf_lex_0x`, but nothing for a cooked string ending on
  a backslash at EOF. Add an `.experr` expecting `unterminated string`.
- **Column-in-diagnostic on a non-ASCII line (A01-3) is uncovered.** No test asserts the reported
  column of an error that follows a multi-byte character.
- **No byte-string / raw-byte literal exists in the lexer** (only `r`/`f` prefixes; no `b"..."`), so
  `\xHH` in a `String` is the *only* literal path to a specific byte value — which is exactly what
  makes A01-1 sharp. Worth a design note/test either way.

## DRY / single-source-of-truth notes

- **Prefix scanning duplicated.** The `r/R/f/F` prefix loop is written twice: `tryStringLiteral`
  (lexer.hpp:266-271, validating) and `stringLiteral` (lexer.hpp:284, re-skipping
  `while (peek()=='r'||'R'||'f'||'F') advance();`). They must stay in lock-step; `stringLiteral`'s
  unbounded `while` is only safe because `tryStringLiteral` already validated the prefix. Consider
  passing the prefix length from the caller so `stringLiteral` doesn't re-scan.
- **Hex-digit decoding split across layers.** `decodeEscape` correctly uses the shared
  `hexDigitValue` (common.hpp:93). But `number()`'s base-digit test uses `std::isxdigit`
  (lexer.hpp:186) and, cross-cuttingly, the **parser's** `intLiteral` (parser.hpp:1028-1030)
  re-hand-rolls the hex `a-f`/`A-F` → value mapping instead of calling `hexDigitValue`. Not a lexer
  bug (parser scope A02/A03), but the hex-nibble logic should be single-sourced through
  `hexDigitValue`.
- **Control-byte rendering (A01-4)** is duplicated between the two `default` arms and would benefit
  from one shared `printableByte`/`quoteChar` helper.
