## A09-1 — `ModuleBuilder::fn` (signature overload) leaks heap-allocated parameter defaults across the `NativeFunction` alloc (mid-expression GC sweeps them) — CONFIRMED

- **file:** `src/kirito/native.hpp:117-121` (`ModuleBuilder::fn(name, sig, returnType, impl)`); root cause also `src/kirito/function.hpp:46-49` (`NativeFunction` signature ctor).
- **severity:** medium (memory-safety; latent, GC-alignment-dependent, reachable via normal `import(...)` of a stdlib module that declares a heap default, and reliably via an embedder using `setGcThreshold`).
- **failure scenario:** A signatured native declares a **non-interned** default value, e.g. `hash.hmac`:
  ```cpp
  m.fn("hmac", {{"key"}, {"msg"}, {"algo", "String", m.vm().makeString("sha256")}}, "String", impl);
  ```
  `m.vm().makeString("sha256")` allocates the default StrVal and stores its bare `Handle` inside a `NativeParam` in the `sig` vector. `fn()` then does `vm_.alloc(std::make_unique<NativeFunction>(name, std::move(sig), ...))`. The `NativeFunction` (which owns the default via `children()`) is still inside a **not-yet-arena'd `unique_ptr`** when `vm_.alloc` runs, so if that alloc crosses the GC threshold, `collectGarbage()` cannot reach the pending `NativeFunction` — the `"sha256"` StrVal is unrooted and is **swept**. The signatured member is then stored with a **dangling default handle**. A later `hash.hmac(key, msg)` (algo omitted) routes through `applyCall` → `bindArgs`, which fills `out[i] = sig_[i].defaultValue` (the dangling handle) → `deref` → stale generation.
- **triggerability:** Real. Stdlib native modules install their factory in the ctor but `setup()` (the `fn(...)` calls) runs **lazily on first `import`**, so this executes during user script execution when `allocsSinceGc_` may sit one alloc below the adaptive threshold (≥65536). Small-int defaults are safe (interned), but `makeString`/`makeFloat`/large-`makeInt` defaults are not. Confirmed empirically:
  ```
  KiritoVM vm; vm.setGcThreshold(1); vm.importModule("hash");
  applyCall(vm, hmac, {key,msg}, {})  ->  EXCEPTION: dangling handle (stale generation)
  ```
  (In a longer-running VM the swept slot is often reused by an unrelated object → silent type confusion instead of a clean throw.)
- **proposed fix:** Root the sig defaults across the alloc in the `fn` signature overload:
  ```cpp
  ModuleBuilder& fn(std::string name, std::vector<NativeParam> sig, std::string returnType, NativeFn impl) {
      RootScope rs(vm_);
      for (const auto& p : sig) if (p.hasDefault) rs.add(p.defaultValue);
      mod_.members[name] = vm_.alloc(std::make_unique<NativeFunction>(
          name, std::move(sig), std::move(returnType), std::move(impl)));
      return *this;
  }
  ```
  Alternatively, root inside `KiritoVM::alloc` more generally, but the localized fix matches the existing RootScope idiom (as used in `List`/`Dict`/`Set` ctors). Note the same latent hazard exists for **any** direct `vm.alloc(make_unique<NativeFunction>(name, sig, ...))` with heap defaults — grep shows `ModuleBuilder::fn` is the sole such path, so fixing it there suffices, but a defensive belt-and-braces option is to have the `NativeFunction` signature ctor not be the rooting site (it can't reach the vm).
- **doc/test gap:** No regression test installs/imports a signatured module under aggressive GC. Add a C++ test that `setGcThreshold(1)` then imports `hash`/`random`/etc. and calls a fn relying on a String/Float default.

## A09-2 — `Value::isBytes()`/`asBytes()`/`tryBytes()` type-confuse a user class named "Bytes" → SEGV/UB — CONFIRMED

- **file:** `src/kirito/value.hpp:170` (`isBytes()`), consumed by `asBytes` (953), the `Bytes(vm,h,who)` ctor (505-508), `tryBytes` (976-978).
- **severity:** medium (embedding-API soundness hole; clean crash / memory disclosure via type confusion).
- **root cause:** `isBytes()` discriminates by string compare:
  ```cpp
  bool isBytes() const { return kind() == ValueKind::Instance && typeName() == "Bytes"; }
  ```
  A user-defined `class Bytes:` (the name is not reserved) produces `InstanceValue`s with `kind()==Instance` and `typeName()=="Bytes"` (`class_value.hpp:58`). So `isBytes()` returns **true** for them, and `asBytes()` performs `static_cast<const BytesVal&>(ref())` on an `InstanceValue` — a bogus downcast. `Bytes::data()`/`size()`/`operator[]` then read the `BytesVal::data` member off an `InstanceValue`'s memory → garbage length + OOB read.
- **failure scenario (CONFIRMED under ASan):** a third-party native written with the ergonomic API does `if (arg.isBytes()) { Bytes b = arg.asBytes(); ...b[i]... }`; Kirito code defines `class Bytes` and passes an instance. Result:
  ```
  AddressSanitizer: SEGV ... in kirito::Bytes::operator[](long) const  value.hpp:526
  ```
- **triggerability:** Reachable from pure Kirito against **any** third-party native that peeks with `isBytes`/`asBytes`. The **shipped** stdlib is NOT affected — it detects Bytes with `dynamic_cast<const BytesVal*>` (bytes.hpp:64/108/314/345), not the string compare — so this is specifically an embedding-API (value.hpp) defect. Every other kind-peek (`isInt`/`isString`/`isList`/…) uses a distinct `ValueKind` and cannot collide; only `isBytes` (which has no dedicated `ValueKind`) is unsafe.
- **proposed fix:** Give `BytesVal` a non-string discriminator. Add to `Object` (object.hpp) a `virtual bool isBytesVal() const { return false; }`, override in `BytesVal` to `return true;`, and make `Value::isBytes()` return `bound() && ref().isBytesVal()`. (Or, since `value.hpp` is included before `bytes.hpp`, use a lazy `dynamic_cast` via a helper defined out-of-line in bytes.hpp — but the virtual flag is cheaper and matches the object-protocol style.)
- **doc/test gap:** No test defines a user `class Bytes` and round-trips it through the wrapper peeks. Add a C++ test asserting `Value(vm, userBytesInstance).isBytes() == false` and that `asBytes()` throws a clean type error.

## A09-3 — `Value::call(std::initializer_list<Anything>)` does not root the argument handles it allocates → swept during the callee's scope prologue — CONFIRMED

- **file:** `src/kirito/value.hpp:292-297`.
- **severity:** medium (embedding-API GC-rooting bug; GC-alignment-dependent; the documented convenience call path).
- **root cause:** the convenience overload materialises fresh argument handles and passes them straight into the call without rooting them:
  ```cpp
  Value call(std::initializer_list<detail::Anything> args) const {
      std::vector<Handle> hs;
      hs.reserve(args.size());
      for (const auto& a : args) hs.push_back(a.toHandle(*vm_));   // makeInt/makeFloat/makeString: fresh, UNROOTED
      return call(std::span<const Handle>(hs));                    // hs is a plain local, not on tempRoots
  }
  ```
  `Value::call(span)` → `ref().call(...)` → `KiFunction::callFull`, whose **first** action is `vm.newScope(closure_)` (`runtime.hpp:2028`) — an allocation that can trigger GC **before** the positional args are bound into the new scope. The `hs` handles are reachable from no GC root, so they are swept; the subsequent bind/deref hits a stale slot.
- **failure scenario (CONFIRMED under ASan):**
  ```cpp
  Value f = ...;                       // a Kirito Function(a,b,c): return a+b+c
  vm.setGcThreshold(1);
  f.call({1000000, 2000000, 3000000}); // non-interned ints -> real allocs
  // -> EXCEPTION: dangling handle (stale generation)   (in production: reused-slot type confusion)
  ```
  `f.call({1,2,3})` happens to work because 1/2/3 are interned small ints — which masks the bug in casual testing.
- **triggerability:** Any embedder calling a Kirito callable via the initializer-list convenience (`fn.call({...})`, `PinnedHandle::value().call({...})`) with non-interned primitive arguments, once the VM has enough live objects that a GC can fire on the callee's `newScope`. The `Value::call(std::span<const Handle>)` variant is caller-rooted-by-contract, but the initializer-list overload *owns* the handle creation and must root them itself.
- **proposed fix:**
  ```cpp
  Value call(std::initializer_list<detail::Anything> args) const {
      RootScope roots(*vm_);
      std::vector<Handle> hs;
      hs.reserve(args.size());
      for (const auto& a : args) hs.push_back(roots.add(a.toHandle(*vm_)));
      return call(std::span<const Handle>(hs));
  }
  ```
- **doc/test gap:** No C++ test exercises `Value::call({...})` under GC pressure with non-interned args. Add one. Also consider documenting in value.hpp that `call(span)` requires the caller to keep the arg handles rooted.
