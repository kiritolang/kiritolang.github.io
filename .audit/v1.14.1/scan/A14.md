# A14 — Kirito-authored frozen-source stdlib modules

Subsystem: `src/kirito/stdlib_kimodules.hpp` — itertools, functools, collections,
statistics, string, textwrap, base64, csv, tabular, xml, heapq, bisect, copy, enum,

Read-only audit. Findings appended as found.

## A14-1 — statistics.quantiles clamps endpoints instead of extrapolating (diverges from Python exclusive method)
- **file:line**: stdlib_kimodules.hpp:455-476 (`quantiles`)
- **severity**: medium
- **scenario**: The "exclusive" method (the documented default) must EXTRAPOLATE when a cut point falls outside the interior interval `[1, ld-1]`; the code CLAMPS to the min/max element instead.
  - `quantiles([10, 20])` → Kirito `[10.0, 15.0, 20.0]`, Python `statistics.quantiles([10,20])` → `[7.5, 15.0, 22.5]`.
  - `quantiles([1,2,3,4,5], 10)` → Kirito `[1.0, 1.2, ... 5.0]`, Python → `[0.6, 1.2, ... 5.4]` (first/last cut points differ).
- **root cause**: the `if lo < 1: append s[0]` / `elif lo >= ld: append s[ld-1]` clamps rather than linearly extrapolating with a negative/over-`n` `delta` the way CPython's exclusive method does (`interpolated = (data[j-1]*(n-delta) + data[j]*delta)/n` with `j` clamped to `1..ld-1` but `delta = i*m - j*n` allowed to go out of `[0,n]`).
- **triggerability**: only when a requested cut point lies outside the interior — i.e. small datasets or `n` large relative to `len(data)`. Interior quantiles (the common case) already match.
- **proposed fix**: mirror CPython exactly: `m = ld + 1`; for each `i` compute `j = clamp(i*m // n, 1, ld-1)`, `delta = i*m - j*n`, `out.append((s[j-1]*(n-delta) + s[j]*delta) / n)`. This keeps interior results identical and fixes the extremes.
- **coverage gap**: existing tests presumably only exercise interior cut points; add a small-dataset case (`[10,20]`) to pin the extrapolation.

## A14-2 — base64.decode rejects whitespace/newlines that MIME/PEM base64 legitimately contains
- **file:line**: stdlib_kimodules.hpp:641-664 (`decode`); also `urlsafedecode`.
- **severity**: medium
- **scenario**: `base64.decode("TWFu\nTWFu")` throws `invalid base64 character: '\n'`. Python's `base64.b64decode` (and every MIME/PEM producer) tolerates embedded newlines/whitespace — base64 in the wild is line-wrapped at 64/76 columns (email, PEM keys, HTTP). Any real-world base64 blob with line breaks fails to decode.
- **root cause**: the decode loop throws on any char not in the alphabet and not `=`, with no skip for ASCII whitespace.
- **proposed fix**: in the loop, `continue` (skip) on `ch == "\n" or ch == "\r" or ch == "\t" or ch == " "` before the `ch not in _index` rejection — matching Python's lenient default. Keep rejecting genuinely-invalid chars.
- **coverage gap**: no test decodes wrapped base64; add one.

## A14-3 — textwrap does not treat embedded newlines/runs of whitespace as word separators
- **file:line**: stdlib_kimodules.hpp:549-566 (`wrap`/`fill`)
- **severity**: low
- **scenario**: `textwrap.fill("line1\nline2 word")` keeps `line1\nline2` as a single token (never rewrapped, and its embedded newline is counted toward width); Python normalizes all whitespace and yields `line1 line2 word`. `fill("a  b   c")` preserves the runs (`a  b   c`) where Python collapses to `a b c`.
- **root cause**: `text.split(" ")` only splits on a literal single space — tabs/newlines are not separators and consecutive spaces produce empty tokens that still get a joining space.
- **proposed fix**: split on any whitespace run and drop empties, e.g. build words by scanning or `text` normalized with `.split()`-style behavior (Kirito `split(" ")` has no no-arg whitespace mode, so pre-replace `\t`/`\n`/`\r` with spaces then filter empty tokens).
- **note**: this is an accepted simplification if documented, but the newline-swallowed-into-a-token case is surprising; low severity.

## A14-4 — collections.deque has no maxlen (Python deque's defining feature)
- **file:line**: stdlib_kimodules.hpp:303-324 (`deque`)
- **severity**: low
- **scenario**: `deque(items, maxlen=N)` is unsupported; there is no bounded-ring eviction on append/appendleft. A user porting Python code that relies on `maxlen` gets a plain unbounded list.
- **triggerability**: only if a caller expects maxlen; the audit focus explicitly lists "deque maxlen eviction", which does not exist here.
- **proposed fix**: add an optional `maxlen` ctor arg; on `append`/`appendleft` past `maxlen`, drop from the opposite end. Or document the omission.
- **coverage gap**: feature/doc gap only.

## A14-5 — copy.copy() performs a DEEP copy of class instances / native objects (not shallow)
- **file:line**: stdlib_kimodules.hpp:902-912 (`copy`) via `_copyViaSerde`
- **severity**: low
- **scenario**: `copy.copy(box)` where `box.inner` is a shared list produces a fully independent deep copy — mutating `shallow.inner` does NOT affect the original (verified). Python's `copy.copy` is shallow (nested references are shared). A user relying on shallow semantics (shared nested state) gets isolated state instead.
- **root cause**: pure Kirito cannot enumerate/set an instance's attributes generically, so instances are copied through the serialize graph codec, which is inherently deep. Acknowledged in the source comment.
- **verdict**: documented limitation, not a latent crash. Note it in user docs so callers know `copy.copy` == `copy.deepcopy` for instances/native values.

## A14-6 — tabular Series-Series arithmetic aligns by POSITION, not index label (unlike pandas)
- **file:line**: stdlib_kimodules.hpp:1400-1412 (`Series._binop`)
- **severity**: low
- **scenario**: `s1 + s2` combines the i-th element of each regardless of index labels; pandas aligns by label and fills non-overlapping labels with NaN. Two series with the same data in different label order produce different results than pandas.
- **root cause**: explicit design choice ("aligned by position") stated in the comment.
- **verdict**: documented divergence; flagged for completeness. Also enforces equal length (throws on mismatch) where pandas would union-align — acceptable given the position model.

## A14-7 — Counter.mostcommon() tie order is Dict-hash order, not insertion order
- **file:line**: stdlib_kimodules.hpp:341-345 (`Counter.mostcommon`)
- **severity**: low
- **scenario**: For elements with equal counts, CPython's `Counter.most_common` preserves first-insertion order; here the order comes from `self._counts.items()` which Kirito documents as unordered, so ties are emitted in hash order. `sorted(..., reverse=True)` is stable but only stabilizes the already-unordered starting sequence.
- **root cause**: Kirito Dict is unordered by design (documented). To match Python, mostcommon would need a first-seen order list maintained on `add` (as `enum`/`multimode` already do).
- **verdict**: rooted in the documented unordered-Dict contract; low. If Python parity is desired, track insertion order in `Counter` like `multimode` does.
