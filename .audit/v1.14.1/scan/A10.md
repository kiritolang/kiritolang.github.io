# A10 — io / path / sys / time + process & platform compat

Subsystem: `src/kirito/stdlib_io.hpp`, `stdlib_path.hpp`, `stdlib_sys.hpp`, `stdlib_time.hpp`, `proc_compat.hpp`.
Read-only audit. Findings appended as found.

## A10-1 — FileVal.read(huge n) on a non-seekable file bypasses the size-clamp guard → bad_alloc

- **file:line**: `src/kirito/stdlib_io.hpp:95-112` (`FileVal::streamRead`)
- **severity**: medium
- **scenario**: `io.open("/tmp/fifo", "r").read(9000000000000000000)` on a FIFO/device/`/proc` file
  (any non-seekable stream). The clamp logic seeks to end to measure remaining bytes; on a
  non-seekable stream `tellg()` returns `-1`, so `endp < 0`, the `if (endp >= 0)` clamp is skipped,
  and `std::string buf(want, '\0')` allocates the full requested `n` bytes → `std::bad_alloc`.
  Confirmed reproduced (see below). It IS catchable (bare `catch` surfaces it as `"std::bad_alloc"`),
  but the documented guard ("a huge read(n) must never try to allocate n bytes ... including when the
  cursor is unknown (cur<0)") is defeated: the comment claims it clamps when the cursor is unknown,
  but it only clamps when `endp >= 0`, which is false for a non-seekable stream.
- **repro**:
  ```
  mkfifo /tmp/f; (printf hello > /tmp/f)&
  ki -e 'var io=import("io"); io.open("/tmp/f","r").read(9e18)'   # -> std::bad_alloc
  ```
- **triggerability**: needs a non-regular file opened via `io.open` + a huge `n`. Low-probability in
  normal code but a clean DoS/crash vector for untrusted input paths/sizes. StdStream (64 KiB chunks)
  and BytesIO (clamped to buffer) do not have this; only FileVal pre-allocates `want`.
- **proposed fix**: when the size can't be measured (`endp < 0`), don't pre-allocate `want`; read in
  bounded chunks up to `want` (mirror `StdStream::streamRead`'s `kChunk = 1<<16` loop), or cap the
  initial allocation (e.g. `min(want, 64KiB)`) and grow. The seekable-file fast path can stay.
- **coverage gap**: no test opens a non-seekable file and reads with a large n.

## A10-2 — File.read(None) throws while BytesIO.read(None)/StdStream read-all: inconsistent None handling

- **file:line**: `src/kirito/stdlib_io.hpp:162-171` (FileVal `read`) vs `319-327` (BytesIO `read`)
- **severity**: low
- **scenario**: `io.open(p,"r").read(None)` throws `read expected Integer, got 'None'`, but
  `io.BytesIO().read(None)` treats None as "read all" (Python semantics). Two stream types with the
  same `read` surface behave differently for the same argument.
- **triggerability**: direct; anyone porting Python code that passes `read(None)` to read-all.
- **proposed fix**: in FileVal's `read`, treat a None `a[0]` as "no size given" (read all), same as
  BytesIO's `read` already does (`vm.arena().deref(a[0]).kind() != ValueKind::None`).
- **coverage gap**: no test exercises `File.read(None)`.

## A10-3 — path.dirname/basename/splitext reject embedded NUL, but path.join does not; comment claims none do

- **file:line**: `src/kirito/stdlib_path.hpp:86-90` (comment) vs `113/123/128` (dirname/basename/splitext use `pathArg`) vs `96` (join uses `asStringRef`)
- **severity**: low
- **scenario**: These are documented as *pure string ops that don't touch the filesystem*. The
  comment states join/dirname/basename/splitext "keep asStringRef" (no NUL check). In fact
  dirname/basename/splitext call `pathArg`, which runs `requireNoNulPath` and throws on an embedded
  NUL, while `join` does not. So `path.splitext("a\0b.txt")` throws `path: embedded NUL byte in path`
  but `path.join("a\0b.txt","x")` yields `a\0b.txt/x`. Confirmed. A Kirito String is byte-transparent
  (may legitimately hold a NUL), so a pure string transform arguably shouldn't reject it — and the
  divergence between join and the other three is surprising.
- **triggerability**: only with a NUL in a path string; niche.
- **proposed fix**: pick one policy for the pure string ops. Either make dirname/basename/splitext use
  `asStringRef` (matching join and the comment), or reject NUL in join too and fix the comment. Since
  these never hit the OS, using `asStringRef` (accept NUL) is the more consistent choice and matches
  the comment as written.
- **coverage gap**: no test covers NUL in pure path string ops.

## A10-4 — io.print/eprint/input/read/write with stream=None throws an obscure attribute error

- **file:line**: `src/kirito/stdlib_io.hpp:549-559` (`pickStream`)
- **severity**: low
- **scenario**: `io.print("x", stream = None)` -> `type 'None' has no attribute 'write'`. An explicit
  `stream=None` is taken as "use this stream = None" rather than "fall back to default", producing a
  confusing diagnostic (vs Python's `print(file=None)` which uses stdout). Fails loudly (not silent),
  so not dangerous, just unfriendly.
- **triggerability**: direct, whenever a caller passes a possibly-None stream var through `stream=`.
- **proposed fix**: in `pickStream`, treat an explicit None the same as absent (fall back to the
  default slot), or raise a clearer `"stream= must be a stream object, got None"`.
- **coverage gap**: none for `stream=None`.
