## A02-1 — Expression continues across a block-bodied function literal's terminating dedent (silent mis-parse) — MEDIUM

**Where:** parser.hpp — `parsePostfix` (584), `parseAdd` (531), `parseMul` (543), `parseComparison` (501), `parseAnd`/`parseOr` (466/477), `parsePow` (572), `parseValueSeq` (307). The guard that FIXES this exists in exactly ONE place: `parseConditional` (447) checks `blockJustClosed_` before treating a following `if` as a ternary. No other continuation point honors the flag.

**Root cause:** A block-bodied `Function(): <indented suite>` literal is the trailing leaf of its expression; `parseIndentedSuite` (344) consumes the block's Newline+Dedent and sets `blockJustClosed_ = true` — a real statement terminator (there is no Newline token left between the Dedent and the next line's first token). But on the ascent out of `parsePrimary`, `parsePostfix`'s while-loop and every binary-precedence loop test only for their operator token — they never check `blockJustClosed_` — so the NEXT statement's leading `(`, `[`, `.`, `-`, `+`, `*`, `/`, `**`, `and`, `or`, `<`, `in`, ... is glued onto the function value.

**Failure scenarios (all empirically reproduced against build-debug/ki):**

Silent (no error, wrong result):
```
var f = Function(n):
    return n * 10
(5)                 # intended: standalone (5); actual: parsed as f(5), value discarded -> prints nothing
io.print("done")    # "done"  (rc=0, no warning even without -w)
```
```
var log = Function(msg):
    io.print("LOG:", msg)
    return None
(io.print("standalone"))   # parsed as log(io.print("standalone")) -> prints "standalone" then "LOG: None"
```
Loud but still a mis-parse (the following statement is swallowed into the function value):
```
var f = Function():
    return 1
- 2                 # parsed as (f - 2): "type 'Function' does not support this binary operator"
```
Also confirmed: `* 2`, `.foo`, `[1,2][0]`, `(...)` on the line after a block-fn all glue (member/index/call/binary). Only a following `if` correctly begins a new statement (parseConditional guard).

**Triggerability:** Requires a block-bodied function literal (ubiquitous: `var f = Function(): <block>`) immediately followed by a statement whose first token is an operator/bracket/dot. A leading-`(`, leading-`[`, or unary-`-`/`+` statement is legal Kirito, so this is a genuine silent landmine, not merely an invalid-syntax case. Real-world hit rate is modest (people usually interpose other statements), but it is a core-grammar correctness hole.

**Proposed fix (idiomatic, mirrors the existing parseConditional guard):** treat `blockJustClosed_` as a hard expression terminator at every continuation point, not just the ternary. Concretely, short-circuit each ascent loop while the flag is set:
- `parsePostfix`: `auto expr = parsePrimary(); if (blockJustClosed_) return expr;` before the while-loop (a block-fn leaf cannot be same-line call/index/member chained anyway — a block spans lines).
- each binary loop (`parseAdd`/`parseMul`/`parseComparison`/`parseAnd`/`parseOr`) and `parsePow`/`parseValueSeq`: fold `!blockJustClosed_` into the loop/continuation condition.
To keep it DRY, a tiny `bool moreExpr(...)` predicate or a single `if (blockJustClosed_) return left;` guard hoisted where each loop begins. No legitimate program relies on gluing across a block boundary (a block-bodied literal is inherently multi-line), so this is behavior-preserving for valid code — same reasoning that already justifies the parseConditional guard.

**Test gap:** no `.ki`/C++ test exercises a statement immediately following a block-bodied function literal. Add golden tests: `(expr)`, `[expr]`, `-expr`, `.name`, and `expr` starting the next line after a block-fn must each parse as a SEPARATE statement.

### A02-1 — addendum: comma facet + fix-safety proof

**Extra facet (same root cause):** `parseValueSeq` (307) also glues a following-line comma. Empirically:
```
var f = Function():
    return 1
, 2                 # -> f silently becomes [<function>, 2]  (rc=0)
```
So the guard must also cover `parseValueSeq`'s `at(Comma)` continuation.

**Fix is provably safe (no legitimate code relies on the glue):** the lexer suppresses Newline/Indent
tokens whenever `parenDepth_ > 0` (lexer.hpp lines 72 & 83; `parenDepth_` bumped by `( [ {`). A block
body needs a real Newline+Indent after the `:`, so a `Function():` written INSIDE `(`/`[`/`{` is forced
to parse as an INLINE body (a multi-statement body inside brackets errors — verified). `blockJustClosed_`
is therefore set ONLY by a *statement-top-level* block-fn (the expression's trailing leaf), and does NOT
leak out of brackets — confirmed: `[Function():\n return 5] if False else "x"` correctly yields `"x"`
(the bracketed fn is inline, flag stays false), and `[Function(): ...][0]` still chains. Hence adding a
`blockJustClosed_` short-circuit at every continuation point cannot break `[fn][0]`, `(fn).m()`, or a
ternary after a bracketed fn. It is the exact analogue of the guard already present (and load-bearing)
in `parseConditional`.

## A02-2 — f-string embedded expression: only leading/trailing spaces+tabs are trimmed, not newlines — LOW

**Where:** parser.hpp `parseEmbedded` (901-906): `find_first_not_of(" \t")` / `find_last_not_of(" \t")`
trim only spaces and tabs, to keep the indentation-sensitive sub-lexer from emitting a spurious INDENT.
A newline INSIDE the braces of a triple-quoted f-string is not trimmed, so the sub-lexer emits an
INDENT/Newline and the parse fails.

**Scenario (reproduced):**
```
io.print(f"""val={
  x
}""")
```
→ `error: expected an expression` (rc=1). A multi-line embedded expression (legal in Python 3.12+) is
rejected. Not a crash; the diagnostic is a little opaque (points inside the interpolation).

**Triggerability:** low — requires a newline literally inside `{ }` of a triple-quoted f-string. Docs
describe an f-string hole as "a single expression", so single-line is the intended contract.

**Proposed fix / doc:** either extend the trim set to include `\n\r` (and let the single-expression
check still reject a genuine two-statement body), or document explicitly that an f-string interpolation
must be single-line and improve the diagnostic to say so. Prefer the doc note unless multi-line holes are
wanted.

## A02-3 — trailing comma accepted in list/set/dict literals but rejected in call args & param lists — LOW (consistency)

**Where:** `parseListLiteral` (851)/`parseBraceLiteral` (864) accept a trailing comma (`[1,2,]`, `{1,2,}`,
`{a:1,}` all parse). But a call arg list (`parsePostfix` call loop, 597) and a parameter list
(`parseFunction`, 714) do not: `f(1,)` → `expected an expression`; `Function(a,):` → `expected a
parameter name`. Both verified.

**Impact:** cosmetic inconsistency; every case yields a clean parse error, no miscompile. Python allows a
trailing comma uniformly. Optional harmonization: allow a single trailing comma before `)` in calls and
param lists (break the loop when the token after a comma is `)`), mirroring the collection-literal rule.
Test gap: no test pins the current reject-in-calls / accept-in-literals split either way.

## Notes (checked, NOT findings)

- **Recursion/stack guards are complete and effective.** Deeply nested parens/lists/dicts/sets, unary
  `-`/`not` chains, right-assoc `**`, ternary else-chains, long left-deep `+`/postfix/comparison chains,
  and a 3000-deep indentation pyramid ALL throw a clean `expression nested too deeply` (rc=1) — no
  segfault. `DepthGuard` (recursive descent) + `ChainDepth` (iterative left-deep chains, guards the
  AST-teardown cascade) + the shared `exprDepth_` budget threaded into the f-string sub-parser
  (`startDepth`) cover every path. Sanitizer builds cap at 250 (larger frames).
- **Two starred targets** are rejected on both the `var`/`for` path (`parseTargetNameList`, clean parse
  error) and the assignment-statement path (compiler `two starred targets in assignment`). No gap.
- **Invalid assignment targets** (`1 = 2`, `f() = 2`) are rejected with a clean `invalid assignment
  target` (compiler). Empty subscript `a[]` rejected in the parser (`expected an index expression`).
- **f-string brace/spec scanning is correctly quote-aware AND bracket-depth-aware**: `f"{d['}']}"`,
  `f"{a[1:3]}"`, `f"{42:>10}"`, `f"{'a:b'}"` all parse right; nested f-strings with alternating quote
  styles work (same-quote re-nesting is a documented LEXER limit, not a parser/stack issue).
- **Chained comparison** `1<2<3` = `(1<2)<3` (a type error at runtime) is explicitly documented
  ("Comparisons do not chain", docs/pages/02-language-guide.md:99) — by design, not a bug.
- **KiritoVM self-containment:** parser.hpp/ast.hpp hold NO mutable global/static state — the only
  statics are `constexpr kMaxParseDepth` and pure static helpers (`intLiteral`/`literal`/`binary`/
  `logical`/`labelFunction`) and `walkAssignTarget` (a stateless template). Clean.
