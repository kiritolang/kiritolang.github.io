(329 L), `src/kirito/locals.hpp` (181 L). Read-only audit + empirical checks with `./build-debug/ki`.

## Summary verdict

No functional (correctness) bug found. The compile-time name-resolution pass, the captured-vs-slotted
local analysis, and the analyzer lint were exercised against adversarial scope cases (mutual
recursion, forward refs, closure capture through param defaults, class-method capture of an enclosing
local, loop-variable capture, write-only captured vars, nested-default undefined names) and all behave
correctly and consistently with the runtime. Slot/capture classification is consistent BY

`collectBlockDecls(body) - captured - params`). Findings below are all LOW / robustness / latent —
no HIGH/MED.

## A03-1 — Three passes bypass the type-safe AST visitor; a new node type is silently skipped (capture-scan skip = latent closure corruption)

**File:** resolver.hpp:119-160 (`checkExpr`/`checkStmt`), analyzer.hpp:254-300 (`analyzeExpr`),
locals.hpp:94-161 (`CaptureScan::scanExpr`/`scanStmt`), plus `ast::walkAssignTarget` (ast.hpp:416).
**Severity:** LOW (latent; coverage is currently complete — verified).

**Observation.** `ast.hpp` defines `ExprVisitor`/`StmtVisitor` with pure-virtual `visit(...)` per node
and an `accept()` on every node — a dispatch mechanism the *compiler* uses and that makes "handle every
node" a COMPILE ERROR if a node is unhandled. The resolver, the analyzer, and the capture scanner
instead hand-roll parallel `if (dynamic_cast<...>) ...` chains: 30 casts in resolver.hpp, 50 in
analyzer.hpp, 38 in locals.hpp (none of the three use `accept`). Adding a new `Expr`/`Stmt` subtype
compiles cleanly but is SILENTLY SKIPPED by each hand-rolled chain (trailing `else`/comment falls
through) — no compile error, unlike the visitor.

**Why it matters (the dangerous one).** A silent skip in `CaptureScan` (locals.hpp) means a reference
to an enclosing-function local nested inside the new node type is not seen -> the local is not marked
captured -> the compiler slots it (`assignLocalSlots`, compiler.hpp:113-114) -> the closure reads/writes
a frame slot instead of the shared name-based cell -> SILENTLY WRONG closure values, no crash. A skip
in the resolver = a missed `name 'X' is not defined`; a skip in the analyzer = a lost lint. The visitor
exists precisely to prevent this class of drift and is bypassed in all three passes.

**Triggerability.** Not triggerable today — I verified every current `Expr`/`Stmt` subtype
(Literal/Name/Unary/Binary/Logical/Conditional/Call/Member/Index/Slice/List/Set/Dict/FString/Tuple/
Star/Function; all Stmt kinds) is handled in all three chains, and an f-string `:spec` is a plain
string (no hidden sub-expression — `f"{x:{w}d}"` is rejected as `invalid format spec`, so no capture
leak there). It is a MAINTENANCE HAZARD, realized only when someone adds an AST node (e.g. a future
comprehension — listed as "not yet done" in CLAUDE.md, and a comprehension *binds names in a nested
scope*, exactly the capture-sensitive case).

**Idiomatic fix.** Route the three passes through `ExprVisitor`/`StmtVisitor` (or add a `default:`
exhaustiveness guard keyed off `ExprKind`) so a new node is a compile error in every pass rather than a
silent no-op. Lower-effort: one shared "walk children" helper the three passes call, so only one
traversal must be kept complete. Also the DRY defect — the child-traversal skeleton is triplicated ~3x.

## A03-2 — Duplicate-parameter warning points at the function span, not the offending parameter

**File:** analyzer.hpp:306-307 (`warnings_.push_back({fn.span, "duplicate parameter name '" ...})`).
**Severity:** LOW (diagnostic precision).

`Function(a, a)` warns at the `Function` keyword's span, not at the second `a`. Confirmed:
`/tmp/duppar.ki:2:9: warning: duplicate parameter name 'a'` points at `Function`, not the duplicate.
`ast::Param` should carry a span; use `p.span` for a precise caret. Minor — the message names the
parameter, so it is still actionable.

## A03-3 — `declare()` overwrites a same-name Decl, dropping the earlier decl's used-flag (missed "unused" lint on cross-sub-block re-declares)

**File:** analyzer.hpp:74-76 (`declare` does `decls[name] = Decl{...}`), interacting with the
per-block `declaredHere` re-decl check at analyzer.hpp:120-135.
**Severity:** LOW (analyzer false-negative; non-fatal).

`declare()` replaces the whole `Decl` for a name in the function scope's `decls` map, resetting `used`
and `span`. The "re-declared in this block" warning is scoped to a single `analyzeBlock`
(`declaredHere`), so two `var x` in *different* sub-blocks of one function (e.g. one in an `if` body,
one at function top level) are intentionally not flagged as re-declares — but the second `declare`
still overwrites the first `Decl`, discarding whether the first `x` was ever read. Result: if the
first `x` is used but the second is not, only the second's unused-ness survives (correct-ish); if the
first is unused and the second used, the first's dead assignment is never reported. A soundness-leaning
lint gap, not corruption. Consistent with the analyzer's stated "warn only when confident" stance, so
arguably acceptable — recorded for completeness. Distinct from the accepted A03 analyzer
false-positive entries in `.audit/README.md`.

## A03-4 — Divergent recursion caps (resolver 2800 vs analyzer 2500) silently stop descending; resolver relies on the compiler's nesting guard to reject deep undefined names first

**File:** resolver.hpp:120 (`if (depth_ > 2800) return;`), analyzer.hpp:258 (`if (depth_ > 2500)
return;`).
**Severity:** INFO.

Both passes silently stop descending past a hard-coded depth to avoid native stack overflow on a
pathological AST. For the resolver this means an undefined name buried deeper than 2800 expression
levels is NOT reported — the comment states the compiler's own nesting guard throws first, so no
genuinely-compilable program reaches that depth. I could not construct a program that both (a) parses
and (b) puts an unresolved name past 2800 levels without the compiler/parser depth guard firing, so
this appears safe in practice. The two thresholds (2800, 2500) are independent magic numbers with no
shared derivation; a comment cross-referencing the compiler's guard value (and keeping them ordered so
the resolver never gives up before the compiler rejects) would make the invariant auditable.

## Things checked and found CORRECT (no finding)

- **Slot/capture consistency** — resolver `collectDecls`, `capturedLocals`, and compiler
  `assignLocalSlots` all key off the same `collectBlockDecls`; captured names never enter `slotOf_`.
  Verified empirically: counter closure (1/2/3), loop-var capture (2/2/2 — single shared cell),
  class-method capture of enclosing local (42), capture via a nested param default that is mutated
  after definition (200 — proves shared cell, not a slot copy). No misclassification found.
- **Param-default scoping** — a default sees enclosing scopes + earlier params but not later params;
  `Function(a=b, b=1)` correctly errors `name 'b' is not defined` (matches left-to-right runtime bind);
  `Function(n, size=n)` resolves. `CaptureScan::enterFunction` grows `inner` per-param so
  `Function(a, b=a)` binds `a` to the param, not a capture.
- **Mutual recursion / forward refs** — resolution by scope membership; `isEven`/`isOdd` mutual
  recursion resolves and runs. `class B(A)` with `A` defined later resolves at compile time but fails
  at *run* time (`class B(A)` executes before `A` is bound) — correct, documented ordering, not a bug.
- **Assignment targets** — slice-assign (`a[0:2] = ...`) is rejected by the compiler as an invalid
  target, so `walkAssignTarget`'s `default: break` (which skips `SliceExpr`) can never drop a live
  name from resolution/capture. Index/member/tuple/star targets are fully walked.
- **f-string format specs** are literal strings, not sub-expressions — no hidden name escapes
  resolution/capture through them.
- **Write-only captured var** — a closure-set-only local with no read anywhere IS genuinely dead;
  the resulting "assigned but never used" warning is correct, not a false positive.
- **VM self-containment** — no mutable statics; the only `static`s (`alwaysReturnsValue`,
  `isTerminator`) are pure functions. `isGlobal` does a read-only `envLookup`. All state is
  instance-scoped. Safe for multi-VM / `parallel` workers.
