# A07 — builtin scalar types + String/Bytes surface

Scope: src/kirito/builtins.hpp, src/kirito/bytes.hpp (None/Bool/Integer/Float; String surface & .format() mini-spec; Bytes; base64Encode helper).
Round: v1.14.1 patch loop. READ-ONLY on codebase.

Findings appended continuously below.

## Summary

Deep re-audit of the builtin scalar types and the String/Bytes surface. This subsystem has been
audited exhaustively across v1.12/v1.13/v1.14/1.12.1 and is very hard-nosed: every OOB/negative/step
path is count-driven and clamped (`sliceIndices`), every repetition/padding path is `kMaxRepeat`-bounded,
the format-spec parser rejects malformed input and buffer-sizes float output dynamically, encode/decode
validate UTF-8 and reject bad codecs, and `utf8DecodeAt` is guarded against truncated multibyte reads.
No memory-safety issues found (no OOB read/write, no UAF, no overflow UB). Findings are all LOW /
DOC / INFO correctness-or-parity nits.

Severity counts: HIGH 0 · MED 0 · LOW 2 · DOC 1 · INFO 1

## A07-1 (LOW) — str.format silently mishandles mixed automatic `{}` and manual `{0}` fields

- File: src/kirito/runtime.hpp:1376-1407 (the `"format"` method), specifically the `auto_i` counter at 1380/1389.
- Scenario:
  - `"{0} {}".format("a","b")` -> `"a a"`  (expected `"a b"` or an error)
  - `"{} {0} {}".format("a","b")` -> `"a a b"`  (expected error)
  - `"{} {0}".format("a","b")` -> `"a a"`
- Root cause: the automatic index `auto_i` is only bumped by empty `{}` fields; an explicit `{0}`
  neither advances nor is checked against `auto_i`. CPython raises
  `ValueError: cannot switch from automatic field numbering to manual field specification`; Kirito
  instead produces silently-wrong output.
- Triggerability: pure Kirito, any `.format()` call that mixes `{}` and `{n}`. Uncommon pattern, so
  impact is a latent surprise, not a crash — hence LOW.
- Idiomatic fix: track whether the template has used automatic vs manual numbering (two mutually
  exclusive modes); on a mode switch, throw the same clear error the mini-spec parser throws elsewhere.
- Test gap: no test exercises a mixed `{}`/`{0}` template (the format tests use one or the other).

## A07-2 (DOC) — `\xHH` string escape writes a RAW BYTE, so a String literal can hold invalid UTF-8

- File: lexer `stringLiteral` (out of A07's edit scope) feeding StrVal; observed via
  builtins.hpp utf8* helpers. Confirmed empirically.
- Scenario:
  - `"\xff"` builds a 1-byte String whose byte is 0xFF (invalid standalone UTF-8). `ord("\xff")` -> 255,
    `len("\xff")` -> 1, and `"\xff" == Bytes([255]).decode("latin-1")` is **False** (latin-1 decode
    gives U+00FF = bytes C3 BF, i.e. "ÿ", a different String).
  - `"\xc3\xa9" == "é"` is **True** (the two bytes ARE é's UTF-8), so `\x` is byte-level, diverging from
    Python where `\xHH` is a code-point escape (`"\xff"` == "ÿ" in Python).
  - A truncated lead like `"\xe0\x41"` reports `len` 1 but round-trips oddly through `upper()`.
- Impact: memory-SAFE — every String method is byte-bounded (`utf8Starts` only yields in-range offsets,
  `utf8DecodeAt` guards truncated reads, all `substr` calls stay within `size()`), so no OOB. The only
  effect is surprising semantics under code-point indexing/len/case-mapping on a String that is not
  well-formed UTF-8, and only `Bytes.decode("utf-8")` gates against constructing such Strings — the
  `\xHH` literal path does not.
- Verdict: consistent with the documented "byte-transparent String" design, but the byte-vs-code-point
  meaning of `\xHH` (and that it can yield non-UTF-8 Strings) is undocumented and diverges from Python.
  Recommend a one-line note in the string-literal docs (09-types / language guide) clarifying `\xHH`
  emits a raw byte. Not a code change.

## A07-3 (LOW) — `format(value, "s")` accepts numbers and stringifies them (Python rejects)

- File: src/kirito/runtime.hpp:2731 (`type == 's' || (type == 0 && !numeric)` branch of applyFormatSpec).
- Scenario: `format(5, "s")` -> `"5"`, `format(3.14, "s")` -> `"3.14"`. CPython raises
  `ValueError: Unknown format code 's' for object of type 'int'`.
- This is lenient, not unsafe (the String presentation just routes through `stringify`). It is a
  parity divergence only. Numeric-only flags on a String ARE correctly rejected (sign/`#`/`=`/`,`),
  so the asymmetry is one-directional. LOW / arguably by-design leniency — flagging for the triage
  to decide; if kept, worth a doc line.
- Test gap: no test asserts `format(<number>, "s")` behavior either way.

## A07-4 (INFO) — `String.levenshtein` has no input-size guard (CPU DoS on huge inputs)

- File: src/kirito/runtime.hpp:1108-1128 (levenshteinDistance) + 1545 (the method).
- Scenario: `big.levenshtein(big2)` is O(m·n) time. Strings are bounded only by `kMaxRepeat` (256 MB),
  so a pair of multi-megabyte Strings costs >1e12 operations with no abort — a self-inflicted hang.
  Memory is fine (O(min(m,n)) rolling rows).
- Consistent with typical edit-distance libraries (Python's are equally unbounded), so INFO not a bug;
  noted for completeness. No fix recommended unless a global compute budget is desired.

## Coverage / parity notes (not bugs)

- `casefold`, `title`, `swapcase` are absent (not claimed in CLAUDE.md); `"x".casefold()` throws. By design.
- `format` presentation type `'c'` (character) is unsupported — `format(65, "c")` throws
  `unknown format type 'c'`. Undocumented gap vs Python; the docs list the supported set
  (b/o/x/X/d/f/e/g/s/%), so this is a known limitation, not a bug.
- `String(Bytes)` returns the Bytes **repr** (`b'hi'`), matching the documented text/binary separation
  (see the README false-positive on `io.write`/`io.print` stringifying Bytes). Use `.decode()` for text.
  Consistent, not flagged.
- Verified robust (no defect): negative/OOB indexing and slicing on String and Bytes; slice with
  huge/negative/zero step; `*` repetition guards on String and Bytes; padding (`ljust`/`rjust`/
  `center`/`zfill`) code-point width + byte-length caps; multibyte fill rejection; `replace`/`join`
  `kMaxRepeat` result guards; empty-pattern `replace`/`count`/`split`; `find`/`rfind`/`index`/`rindex`/
  `startswith`/`endswith`/`count` start/end windows (negative, past-end, start>end); `strip` code-point
  matching; encode/decode round-trips (utf-8 validation, latin-1 lossless, ascii cap, unknown codec);
  `fromhex` odd-length/non-hex/embedded-whitespace; `Bytes()` ctor from list/int/String/Bytes with
  range checks; `Bytes` contains/ordering type checks; format-spec `,`/`=`/`#`/`0`/sign/precision/width
  including negative integers and overflow bounds; Integer/Float exact cross-type equality + hash
  agreement. All correct.
