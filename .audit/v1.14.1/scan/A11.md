## A11-1 (LOW) — engine `tensor::reshape`/`flatten` bypass the rank/size cap; the comment claims they don't

**File:** `src/kirito/tensor.hpp:107-109` (two-arg ctor), `133-141` (`reshape`/`flatten`), comment at `50-56`.

The rank cap (`kMaxRank = 64`) and element cap (`kMaxElems`) are enforced **only** in the one-arg fill
constructor `Tensor(Shape s, T fill)` (line 106 → `checkedNumel`). The **two-arg data constructor**
`Tensor(Shape s, std::vector<T> d)` (line 107) validates only `data.size() != numel(shape)` — it calls
plain `numel`, never `checkedNumel`, so it enforces **no rank cap and no element cap**. `reshape`
(line 133) and `flatten` (line 139) build their result through this two-arg constructor, so at the
**engine level** `tensor::reshape(t, shape)` accepts a shape of arbitrary rank as long as its numel
matches `t.size()` (e.g. reshaping a 4-element tensor to `[1,1,…,1,4]` with 100 leading ones →
rank-101 tensor).

The comment at `tensor.hpp:52-56` explicitly and **incorrectly** claims the opposite:
> "Enforced here — the single point every allocated shape passes through — so ops that build a
> high-rank shape from a valid one (reshape/expanddims/broadcastto) are covered too, not just the
> nested-list constructor."

That single point is the *fill* constructor; reshape/flatten do not pass through it.

**Failure scenario (C++ engine consumers only):** the engine is documented as "reusable,
unit-testable on its own" — a third-party C++ user who calls `tensor::reshape(t, veryHighRankShape)`
trusting the comment gets a high-rank tensor, then the **recursive** per-axis traversals
(`TensorVal::format`/`str`, `tolist`, indexing) can overflow the native stack — the exact class of
crash `kMaxRank` was introduced to prevent.

**Triggerability from Kirito:** none. Every Kirito-facing tensor allocation routes through
`tns::make` → `tns::checkSize` → `tensor::checkedNumel` (stdlib_tensor.hpp:201-213), and `_setstate_`
calls `checkSize` before building (stdlib_tensor.hpp:2066), so a Kirito program cannot reach the gap.
Verified: `t.reshape([1]*200 + [4])` throws `Tensor has too many dimensions (max 64)`.

**Severity:** LOW (defense-in-depth for the standalone engine + a misleading comment; not user-reachable).

**Proposed fix:** make the two-arg data constructor call `checkedNumel(shape)` instead of `numel(shape)`
(it already needs the element count and can compare against `data.size()`), which single-sources the
rank + element caps for *every* allocation and makes the comment true. Alternatively, have `reshape`
validate `shape.size() <= kMaxRank` explicitly.

**Doc/test gap:** no engine-level unit test asserts that `tensor::reshape` to a >64-rank shape throws;
the Kirito-level guard is tested but the engine path is not.

## Minor observations (not findings; recorded for completeness)

- **`hashing::pbkdf2Raw` with `iterations == 0`** (hashing.hpp:283-306): the inner loop `for (i = 1;
  i < iterations; …)` doesn't run, so `iterations == 0` silently produces the **same output as
  `iterations == 1`** (T = U1) rather than being rejected. RFC 8018 requires c ≥ 1. This is the
  caller's responsibility and the `hash` module wrapper (`stdlib_hash.hpp`, out of this subsystem's
  file scope) is expected to reject `iterations < 1`; the primitive itself does not. Harmless if the
  wrapper guards it; worth a one-line assert if it doesn't.

- **`math.prod` overflow tracking** (stdlib_math.hpp:204-231): after an Integer overflow is detected,
  the running `int64_t n` keeps wrapping, but the result is only returned as Integer when `!isFloat`,
  in which case `intOverflow` throws — so the wrapped value is never observable. Correct.

- **`rand_compat.hpp` Linux path** (rand_compat.hpp:46-56): if `getrandom` fills some bytes then
  fails, the code falls through to `/dev/urandom` and correctly **resumes** at `p + got`; no
  double-fill or gap. Correct.

## Verified correct (high-confidence positive confirmations)

Confirmed by running scratch programs against the debug build:

1. **Scalar `math` domain errors THROW** (not silent NaN): `sqrt(-1)`, `log(0)`, `log(-5)`,
   `asin(2)`, `acos(-2)`, `acosh(0)`, `atanh(1)`, `gamma(0)`, `gamma(-1)`, `fmod(5,0)`, `pow(-2,0.5)`,
   `pow(0,-1)`, `log2(0)`, `log10(-1)` — all throw. NaN **passes through** (`sqrt(nan)`, `sin(nan)` →
   NaN, no throw). Overflow-to-inf is not treated as a domain error (by design).

2. **Same policy on tensors:** `tensor.sqrt(-1)`, `.log(0)`, `.asin(2)`, `.acosh(0)`, `.atanh(1)`,
   `.reciprocal(0)`, tensor `/ 0`, `% 0`, `// 0` (tensor and scalar RHS), `** 0.5` on a negative base
   — all throw; `tensor.sqrt(nan)` passes NaN through.

3. **Same policy on complex analytic set:** `complex.log(0)`, `.log10(0)`, `.atanh(1)`, `.atan(i)`,
   `.pow(0,-1)`, `one/zero` throw; `complex.sqrt(-1) = i` and other everywhere-defined values do NOT
   throw. `complex.polar` rejects a non-finite modulus/angle (finite negative modulus allowed —
   matches the false-positives table).

4. **Singular / non-square linear algebra throws:** `tensor.inv`/`solve` on a singular matrix,
   `inv`/`det` on a non-square matrix, and inf/NaN elements (→ "non-finite value") all throw; same for
   `ComplexMatrix.inverse`/`determinant`.

5. **Exact `==` vs tolerant `.compare` boundary holds** for Tensor, Complex, Matrix, ComplexMatrix:
   `(0.1+0.2) == 0.3` is `False` (bit-exact) while `.compare(...)` is `True`. NaN never equal.

6. **Autograd correctness:** `d/dx sum(x²) = 2x`; broadcasting gradients (`a[2,1]+b[1,3]` → correct
   sum-to-shape reductions); matmul gradient; cumsum gradient (reverse-cumsum); slice gradient
   (scatter to sliced positions only); take-axis-0 gradient; chained scalar ops; `nograd()` context
   disables tracking; `detach()` drops grad; **ops are pure/non-mutating** (operand unchanged);
   gradient accumulation clears non-leaf grads correctly (no double-count across `backward()` when
   `zerograd()` is used); the rebind-not-mutate SGD idiom yields correct per-step gradients across
   multiple steps (verified equal to the explicit `nograd()+detach()` form). The shipped
   `examples/deep_learning/lib/nn.ki` optimizer wraps updates in `with T.nograd()` + `.detach()`, and
   `docs/pages/10-stdlib.md:1715` teaches the same, so the documented idiom is correct.

7. **NaN semantics consistent across tensor ops:** `unique` folds multiple NaNs to one (sorted last);
   `argmax`/`argmin` return the first NaN's index; `max`/`min` (whole and per-axis) propagate NaN;
   `sort` puts NaN last; `median` treats NaN as sorting last (pinned behaviour).

8. **Broadcasting + per-axis reductions** (`sum`/`mean`/`max` over axis 0 and 1) correct; math exact
   integer combinatorics (`comb(64,32)=1832624140942590534`, `comb(30,15)=155117520`, `perm(20,3)`),
   `gamma(0.5)=√π`, `gamma(-0.5)` (valid non-integer negative).

9. **`random` self-contained (no global state), reproducible + serializable:** same seed → identical
   stream on both xoshiro and Mersenne engines; `serialize.dumps`/`loads` restores the exact stream on
   the **same** engine kind (`.generator` preserved); `randombelow` is bias-free and in range;
   `hasentropy()` probes the OS CSPRNG; secure-random funcs throw (never weak bytes) on entropy
   failure.

10. **Hash primitives correct** against standard test vectors: SHA-256/MD5/SHA-1/SHA-512 of `"abc"`
    and HMAC-SHA256 all match published vectors.

11. **einsum** hardening intact: `ii->ii` (repeated output label) and `ij->z` (output label absent
    from inputs) are rejected, as NumPy does.
