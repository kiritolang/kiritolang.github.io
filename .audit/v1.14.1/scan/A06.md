# A06 — Object/Memory Core + VM Encapsulation Audit (v1.14.1)

Scope: object.hpp, arena.hpp, pool.hpp, vm.hpp, environment/function/module headers.
Special mandate: KiritoVM self-containment (no mutable global/static state).

Findings appended continuously below.

## A06-1 — `_activeVM` thread-local left DANGLING after non-LIFO VM destruction
- **file:** src/kirito/vm.hpp:303,31-51 (ctor) + src/kirito/runtime.hpp:3542-3546 (dtor)
- **severity:** MEDIUM (VM-encapsulation invariant break; latent UAF of a static pointer)
- **mechanism:** The ctor does `_prevActiveVM = _activeVM; _activeVM = this;`; the dtor restores
  `if (_activeVM == this) _activeVM = _prevActiveVM;`. This is a save/restore *stack* that is only
  correct if VM lifetimes on a thread are strictly LIFO. With two VMs on one OS thread destroyed
  out of order, the chain leaves `_activeVM` pointing at a freed VM.
- **empirical PoC (asan build, confirmed):**
  A=VM(active=A,prev=0); B=VM(active=B,prev=A); destroy A (active stays B); destroy B
  (active==B → active = B.prev = A) → `_activeVM` now points at the already-freed A.
  Program output: `after ~B: active=0x...80  <-- points at freed A=0x...80 ? YES DANGLING`.
- **scenario → UAF:** any subsequent read of `KiritoVM::activeVM()` on that thread that is then
  dereferenced (InstanceValue `_hash_`/`_eq_`/`_bool_` dispatch, e.g. a host holding a Dict/Set
  wrapper and hashing after its owning VM died, or a freshly constructed VM inheriting the dangling
  prev and later propagating it) reads freed memory. Also: a newly constructed VM captures the
  dangling pointer as its own `_prevActiveVM`, so the corruption persists across VM generations.
- **triggerability:** requires ≥2 KiritoVMs alive simultaneously on one thread (allowed by the
  embedding API — the whole "multiple independent VMs coexist" design goal) with non-LIFO teardown.
  Not hit in-tree (dispatcher puts each worker VM on its own thread), so untested. Low practical
  exploitability but a definite dangling static — it contradicts "independent VMs never interfere".
- **idiomatic fix:** stop tying `_activeVM` to *construction lifetime*. Set it via an RAII scope at
  each public VM entry point (runSource/runRepl/evalIn and any embed call boundary) so it always
  reflects the currently-*running* VM and is always restored to a still-live value — never left
  pointing at a destroyed VM. (Model already guarantees one running VM per thread, so a per-run
  scope is the natural fit.) Add a C++ regression test with two VMs + non-LIFO destruction.
- **doc/test gap:** no test exercises >1 VM per thread; CLAUDE.md's "one KiritoVM = one
  fully-encapsulated process, independent VMs never interfere" is silently violated here.

## A06-2 — Premature sweep in `IoModule::setup()`: stdout/stderr/stdin handles swept mid-setup → dangling
- **file:** src/kirito/stdlib_io.hpp:535-540
- **severity:** HIGH (dangling handle / use-after-free of the most-used module's std streams)
- **mechanism:** setup allocates three StdStreams into C++ locals *before* storing any of them:
  ```
  Handle out = vm.alloc(std::make_unique<StdStream>(StdStream::Dir::Out));  // A
  Handle err = vm.alloc(std::make_unique<StdStream>(StdStream::Dir::Err));  // B  <- GC here sweeps `out`
  Handle in  = vm.alloc(std::make_unique<StdStream>(StdStream::Dir::In));   // C
  m.value("stdout", out).value("__stdout__", out);   // stores a STALE handle
  ...
  ```
  `KiritoVM::alloc` collects *before* creating the object when `++allocsSinceGc_ >= gcThreshold_`.
  While `out` is held only in the C++ local (not yet in `members`, so not a GC root), the alloc of
  `err`/`in` can trigger a collection that sweeps `out` (and then `err`). The subsequent
  `m.value("stdout", out)` stores a dangling handle; the slot is later reused, so at print time
  `streamWriteTo` → `vm.arena().deref(target)` throws `dangling handle (stale generation)` (or, worse,
  dereferences a reused slot of another type). Same for stderr/stdin.
- **empirical proof (asan, KIRITO_SANITIZER_BUILD so GC + arena run):**
  1. `setGcThreshold(1)` + `io.print("hi")` → `dangling handle (stale generation)` (backtrace ends at
     stdlib_io.hpp:495 deref of `target`).
  2. Importing io *before* lowering the threshold → prints "hi" fine (proves the sweep is in setup,
     not in the print path).
  3. Brute force over thresholds {3,5,8,16,32} × pre-alloc pads 0..39 → dozens of natural failures
     (e.g. threshold=3 pad=0/2/3/5/6/8/9/11/12). No threshold=1 needed.
- **real-world trigger:** any GC firing during io's *first* (lazy) import — i.e. a program that
  allocates roughly `gcThreshold` objects before its first `import("io")` (extremely common: compute,
  then print), OR any embedder/test using a low `setGcThreshold` (the project explicitly supports
  `setGcThreshold(1)`: "tests use setGcThreshold(1) to collect on every allocation"). With the default
  adaptive threshold the window is ~1 alloc in 65536 per import, but non-zero and catastrophic, and it
  compounds under `parallel` (every worker VM rebuilds io) and long-running servers.
- **idiomatic fix:** root the three handles across the remaining setup allocations, e.g. add a
  `RootScope rs(vm); rs.add(out); rs.add(err); rs.add(in);` immediately after the three allocs (before
  any further `vm.alloc`), or store each into `members` immediately before allocating the next. The
  `install<T>` factory already wraps setup in a RootScope for the *module* handle but not for these
  sub-allocations.
- **test gap:** no test builds a module under GC pressure. Add a C++ test that imports io with
  `setGcThreshold(1)` (or pre-churns allocation then imports) and calls io.print — it fails today.

## A06-3 — (NOTE, not a defect) `sys` env is process-global, shared across VMs
- **file:** src/kirito/stdlib_sys.hpp:36 `inline std::mutex& envMutex() { static std::mutex m; ... }`
- **severity:** INFO / documented tradeoff
- The mutex guards `getenv`/`setenv`/`unsetenv`/`environ`, which act on the OS **process** environment
  — genuinely process-global. So `sys.setenv` in one VM is visible to another VM in the same process.
  This is inherent to the OS environ (can't be per-VM without emulating it) and the mutex is the correct
  guard for a shared process resource. Flagged only for completeness of the self-containment audit;
  not a Kirito-owned mutable-state leak. Worth one sentence in the embedding docs.

## Statics / self-containment review (special mandate)
Grepped the whole `src/kirito` tree for mutable `static`/global state. Findings:
- **No mutable global/static state holds Kirito values or VM state.** All `static` in the tree is one
  of: (a) `static constexpr`/`static const` read-only lookup tables (hashing K-tables, hex digit
  strings, month-day arrays, tensor math-op maps, type-name constants) — immutable, shareable;
  (b) `static` member *functions* (helpers) — no state; (c) function-local `static const std::string
  empty` (vm.hpp:228) — read-only; (d) process one-time-init guards that are correct-by-necessity:
  Winsock startup + SIGPIPE-ignore (`net_compat.hpp:46,63`), SIGPIPE-ignore (`proc_compat.hpp:209`),
  `SSL_library_init` (`stdlib_net.hpp:628`), and the env mutex (A06-3) — all guard genuinely
  process-global OS resources, not Kirito/VM state.
- **thread_local, VM-scoped by the one-thread-per-VM rule (allowed by design):**
  `KiritoVM::_activeVM` (vm.hpp:303) — but see A06-1, its lifetime management is buggy for non-LIFO
  multi-VM teardown; `detail::g_equalsDepth` (object.hpp:42) and `iterDepth` (runtime.hpp:1908) —
  balanced RAII recursion counters, always restored; the pool free-lists (`pool.hpp:55`) — explicitly
  documented, drained at thread exit.
- Two independent `KiritoVM`s in one process share **no** mutable Kirito state through any static: the
  arena, environments, module cache, class/deserializer registries, GC roots, interned singletons and
  small-ints are all instance members. Only the OS process environment (A06-3) is shared, unavoidably.

## VM self-containment verdict: **YES** (with one caveat)
Two KiritoVMs are mutually encapsulated for all Kirito state. The single caveat is the
`_activeVM` thread-local *lifetime* bug (A06-1): with ≥2 VMs on one thread torn down out of LIFO order
it is left dangling — a latent UAF of a static pointer, not a cross-VM *data* leak. The only genuinely
shared surface is the OS process environment via `sys` (A06-3), which is inherent, not a Kirito static.
