# A08 — collections.hpp (List / Set / Dict / Array)

Deep-audit round v1.14.1. Scope: src/kirito/collections.hpp — List/Set/Dict methods, slicing, sort, apply, set-algebra, len, repeat guard.

Findings appended continuously below.

## Method note on structure

The `collections.hpp` file (369 lines) only *declares* the collection value classes and their
hash-bucket internals (`DictVal::set/find/remove/clear`, `SetVal::add/contains/clear/popArbitrary`,
`ListVal::equals/str`). Every Kirito-visible method (`getAttr`/`getItem`/`setItem`/`slice`/`binary`/
`contains`) is *defined* in `runtime.hpp` (~L439–1022, 1588–1659) once `KiritoVM` is complete. This
audit covers both halves.

Prior rounds (v1.12 A06, v1.13 A09, v1.14 A08, 1.12.1) hardened this subsystem heavily:
reentrancy `ProbeScope` guards on every hash-bucket mutator, GC-rooted snapshots in
sort/apply/remove/index/count/contains, `kMaxRepeat` overflow guards, count-driven slice indices
(no `i += step` overflow), NaN-total-order in `kiLessThan`, empty-bucket reclamation (A08-4). I
re-verified all of these empirically and found them intact.

## A08-1 — `ValueKind::Array` is dead: no class ever produces it (LOW, code clarity / DRY)

**File:** src/kirito/object.hpp:23 (enum), and dead branches at
src/kirito/runtime.hpp:266,421-422,449,477 · src/kirito/value.hpp:171,537 ·
src/kirito/stdlib_json.hpp:281,321 · src/kirito/stdlib_tensor.hpp:1583

**Severity:** LOW (not a correctness bug — no runtime impact).

**Finding:** `ValueKind::Array` is enumerated and CLAUDE.md still describes "an internal `Array` —
same value model as `List`". But an exhaustive scan of `return ValueKind::` across all headers shows
**no class overrides `kind()` to return `Array`** — the concrete `ArrayVal` type no longer exists
(there is no `class ArrayVal`, and value.hpp:171 `isList()`, the List `+`/`*`/`<` guards in
runtime.hpp, and the json/tensor list checks all test `|| kind == ValueKind::Array` defensively).
Every one of those `|| Array` branches is therefore unreachable dead code, and the several
`static_cast<const ListVal&>(b)` after an `Array`-kind check can never actually see an Array.

**Scenario:** no user-visible effect — purely internal. There is no way to create an Array value from
Kirito or from the C++ API (no factory returns it).

**Triggerability:** none at runtime; this is a maintenance/clarity hazard: a future reader wires new
logic behind a `ValueKind::Array` branch believing such values exist.

**Idiomatic fix:** either (a) remove the `Array` enumerator and its dead `|| ValueKind::Array`
branches, and update CLAUDE.md's "internal Array" wording, or (b) if a future Array type is planned,
leave a one-line comment at the enum that no class currently produces it. Low priority; a
documentation/DRY tidy, not a bug. (Confirm no out-of-scope header revives it before deleting — I
checked all `src/kirito/*.hpp`.)

## Confirmed CORRECT (adversarially verified, no fix needed)

All run against ./build-debug/ki. Recording so a later round doesn't re-litigate.

- **Slicing** (A08 list): `xs[::-1]`, `xs[::2]`, `xs[-3:]`, out-of-range `xs[100:200]`→`[]`,
  reversed empty `xs[5:2]`→`[]`, `xs[5:2:-1]`, and **huge steps** `xs[::1e12]`→first elem,
  `xs[::-1e12]`→last elem — all correct; `sliceIndices` count-driven math avoids the `i+=step`
  overflow. Zero step throws "slice step cannot be zero".
- **Sort stability**: multi-key stable sort preserves input order of equal keys; `reverse=True`
  is *also* stable (equal elements keep original order). A `key` that throws, or a mixed-type
  comparison ("cannot order 'Integer' and 'String'"), aborts the sort with the **original list
  left untouched** (snapshot-then-write-back). `[None,None].sort()` correctly throws (None
  unorderable); single-element list needs no comparison.
- **Set algebra operators**: `-` difference, `<`/`<=`/`>`/`>=` proper/improper subset-superset all
  correct. Operator form requires a Set RHS (`{1,2} - [1]` → clean "does not support this binary
  operator", defers to base); the **method** form (`.difference`/`.union`/`.intersection`/
  `.symmetricdifference`) accepts any iterable (String/range/List). Python-consistent asymmetry,
  intentional. `.issubset`/`.issuperset`/`.isdisjoint` GC-root the iterated `other` (A08-5) — verified
  with range/String args.
- **apply**: List (order-preserving), Dict (over values, keys kept), Set (transformed dups collapse),
  String (chars), Bytes (bytes) all correct; empty containers return the right empty type; an
  `apply` fn returning an **unhashable** value on a Set throws "unhashable type 'List'". Original
  container not mutated (snapshot).
- **`*` repeat guard**: `[1,2,3] * 1e12` throws "repeated List too large"; negative/zero count →
  empty list; overflow guard is `n > kMaxRepeat/elems.size()` (no multiply overflow).
- **pop/popitem on empty**: `{}.popitem()` → "popitem: dictionary is empty"; `Set().pop()` →
  "pop from an empty Set"; `[].pop()` → "pop from empty List". `list.pop(i)` bounds-checks negative
  and out-of-range indices; `insert` clamps negative-past-0 to 0 and past-end to len.
- **Key/equality/hashing consistency**: `1 == True` is **False** in Kirito (Bool is not numeric), and
  membership agrees — `True in {1}`→False, `1 in {True}`→False, so `{1: 'a', True: 'b'}` keeps two
  distinct keys. `1 == 1.0` is True and they collapse to one key (`{1:.., 1.0:..}`→len 1); `0.0`/`-0.0`
  collapse. Equality and hashing/membership never disagree → no Set/Dict corruption. (This is
  *stronger* than Python's `1==True` conflation and is correct-by-design for the strongly-typed model.)
- **Reentrancy / mutation-during-iteration**: a contained value's `_str_` that mutates the same
  Dict/Set during stringify is rejected ("Dict changed size during a key comparison") instead of a
  heap-UAF. A `for x in container:` loop iterates a **snapshot** (`iterate()` returns a copy), so
  appending during the loop is safe and bounded. A key's `_hash_` that mutates the *same* container
  is *allowed* (it runs before any bucket reference is cached — verified safe); a key's `_eq_` that
  mutates mid-probe is rejected by the ProbeScope.
- **Cycle + depth guards**: self-referential List/Dict stringify as `[...]`/`{...}`; a 2000-deep
  nested list throws "structure too deeply nested to stringify" (depth bound 1000) rather than
  overflowing the native stack. `a == a` on a self-referential list is True (identity short-circuit).
- **len**: correct on List/Set/Dict and Unicode String (`len("héllo")`==5 code points).
- **Unhashable handling**: List/Dict/Set as a Dict key or Set element throws
  "unhashable type '<T>'" (canonical message, consistent across add/set/remove — the 1.12.1
  Set.remove message-drift fix is in place).
