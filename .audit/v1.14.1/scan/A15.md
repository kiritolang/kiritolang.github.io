## Findings

### A15-1 — f-string unknown-escape "leniency" is stale (doc-critical)
- Page: `docs/pages/02-language-guide.md`, section "String literals" (lines 152-153).
- Code: `src/kirito/parser.hpp:993-1001` (f-string decode routes through `decodeCookedEscape`), `src/kirito/common.hpp:137`.
- Doc says: "(One exception: inside an f-string, an unrecognized escape is lenient — the backslash is dropped, so `f"\q"` yields `"q"`.)"
- Code does: `f"\q"` throws `invalid escape '\q'` — identical to a plain string. The parser comment even says: "Previously these diverged (f\"\\xff\" != \"\\xff\"; f\"\\q\" silently became \"q\")" — i.e. this was deliberately fixed.
- Verified: `./build-debug/ki` on `f"\q"` errors `invalid escape '\q'`.
- Fix: DOC side. Remove the parenthetical claiming f-string leniency; f-strings decode escapes identically to plain strings (unknown escape is a lex error). Note `test_docs_examples.py` would NOT catch this because the language-guide "String literals" fence is presumably not a runnable ```kirito block / or the offending line is prose, not a fenced example.
