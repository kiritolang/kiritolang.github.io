## NET findings

### A13-1 — HTTP `Host` header omits a non-default port (medium)
- file: src/kirito/stdlib_net.hpp:933
- `req = method + " " + target + " HTTP/1.1\r\nHost: " + u.host + "\r\n";` uses only `u.host`, never `u.port`.
- Scenario: `net.get("http://localhost:8080/")` sends `Host: localhost` instead of the RFC-7230-required `Host: localhost:8080`. Virtual-host routers, proxies, and strict servers misroute or 400 the request. Same for HTTPS on a non-443 port.
- Triggerability: any request to a non-standard port (very common in tests/dev/self-hosted services).
- Fix: build the Host value as `u.host + ((u.tls && u.port!=443) || (!u.tls && u.port!=80) ? ":"+std::to_string(u.port) : "")`. Bracket IPv6 literals (`[::1]:8080`) — u.host is unbracketed after parseUrl, so the Host header for an IPv6 URL is also currently malformed (`Host: ::1` — needs `[::1]`). Two bugs in one line.
- Test gap: no test asserts the Host header contents for a non-default port or IPv6 literal.

### A13-2 — IPv6 literal Host header unbracketed (medium, same line as A13-1)
- file: src/kirito/stdlib_net.hpp:933 (+ parseUrl strips brackets at :330)
- parseUrl stores `u.host` WITHOUT brackets (`substr(1, rb-1)`), so for `http://[::1]/` the request emits `Host: ::1` — an invalid Host header (a bare IPv6 must be bracketed). Fix alongside A13-1: re-bracket when host contains ':'.

### A13-3 — protocol-relative redirect (`Location: //host/path`) mis-resolved (low)
- file: src/kirito/stdlib_net.hpp:432 resolveUrl
- A `//other.host/p` Location has `loc[0]=='/'`, so it returns `origin + loc` = `scheme://orig//other.host/p` instead of `scheme://other.host/p`. Browser/requests semantics: a scheme-relative URL adopts the base scheme and switches host. Kirito instead stays on the original host with a doubled-slash path.
- Impact: mostly cosmetic/robustness (it actually fails-safe by not switching origin), but it silently produces a wrong request rather than following the intended redirect. Low.
- Fix: in resolveUrl, before the `loc[0]=='/'` check, test `loc.rfind("//",0)==0` and return `b.scheme + ":" + loc`.

## PARALLEL / DISPATCHER findings

### A13-4 — worker blocked in a bare blocking syscall hangs shutdown forever (medium)
- file: src/kirito/dispatcher.hpp:597-611 (shutdown), :624-630 (reap) — and the design claim at :6-10 / CLAUDE.md ("a worker blocked anywhere always unwinds").
- shutdown() aborts every Waitable then UNCONDITIONALLY joins every worker thread (`reap` → `thread.join()`, no timeout). A worker blocked in a non-Waitable blocking call — `socket.recv()`/`accept()` with no timeout, a blocking pipe/FIFO read, `io.input`, `sys.shell`/`sys.createprocess` with no timeout — never unwinds, so the join blocks and the whole process hangs at exit.
- Reproduced (exit code 124 / hang): a spawned worker calls `net.fromfd(fd).recv(10)` on a socketpair whose peer main still holds open; main exits; `~KiritoDispatcher::shutdown()` hangs joining that worker. Scratch: /tmp/par3.ki. Parallel-primitive blocking (Queue.get) DOES correctly unwind (verified /tmp/par2.ki, exit 0).
- Triggerability: any user program that spawns a worker doing a blocking socket/pipe/subprocess op that never completes, then returns. The bundled servers dodge it via timeouts, but nothing enforces that.
- Fix: (a) at minimum, correct the invariant in CLAUDE.md + dispatcher.hpp to "blocked in a parallel primitive" (Waitables) — a bare blocking syscall is NOT covered. (b) Better: bound shutdown's join with a grace period and, on expiry, fall through to the process-teardown path (sys.exit already uses std::_Exit for exactly this deadlock-safety) or detach the stuck thread rather than joining it. The dispatcher can't abort a raw fd, so a timed join + hard-exit is the pragmatic guarantee.
- Doc/test gap: no test exercises shutdown with a worker in a non-Waitable blocking call (only Queue-blocked workers are tested).

### A13-5 — cross-VM primitives are never reclaimed (low, by-design tradeoff — noted, not a fix demand)
- file: src/kirito/dispatcher.hpp:635-649 makeWaitable; maps queues_/locks_/events_/semaphores_/barriers_ + waitables_.
- Every Queue/Lock/Event/Semaphore/Barrier ever created stays in its map (and a weak_ptr in waitables_) for the dispatcher's whole life; there is no destroy/erase (unlike tasks_, which are erased on join per A08-2). A long-running server that creates per-request primitives grows memory without bound.
- This is largely FORCED by cross-VM-by-identity: a serialized handle carries only the id, so a worker can resurrect a primitive by id at any time; erasing early would break `*ById` rebind. So it is a real leak but hard to fix safely. Low; flagging for awareness, not demanding a change. A future ref-counted/last-holder GC of primitives would need a protocol that no in-flight blob still references the id.

## More NET findings

### A13-6 — multipart `files` field-name/filename not sanitized for CR/LF or quotes (low-medium)
- file: src/kirito/stdlib_net.hpp:858-860
- The multipart body is built by string-concatenating the caller's Dict key (`field`/`filename`) straight into `Content-Disposition: form-data; name="<field>"; filename="<filename>"` with no escaping. A field name / filename containing a `"` or CRLF breaks out of the header (corrupts the encoding or injects multipart headers).
- Contrast: setHdr (:881) and the cookie loop (:906) DO reject CR/LF (the A18-1 hardening) — multipart was missed. It is caller-controlled data (self-injection, so lower severity than remote header injection), but the same rejection should apply for consistency and to avoid silently-mangled uploads.
- Scenario: `net.post(url, {"files": {"a\r\nX: y": "data"}})` emits an injected `X: y` line inside the part headers.
- Fix: reject (or percent/quote-escape) CR/LF and `"` in `field`/`filename` before building the part, mirroring setHdr.

### A13-7 — no default read timeout: `net.get` can hang on a keep-alive server that ignores `Connection: close` (low)
- file: src/kirito/stdlib_net.hpp:266-277 recvAll / :949 (timeout defaults to 0 = none)
- The client sends `Connection: close` and reads to EOF. A server that ignores it (keep-alive, Content-Length-delimited, never closes) leaves recvAll blocking until kMaxRecvAll (256 MiB) or forever when no `timeout` opt is given. Most servers honor close, so this is low, but a hostile/misbehaving server can pin the caller.
- Fix: consider a sane default socket timeout, or parse Content-Length and stop reading at the body boundary rather than relying on EOF.

## SELF-CONTAINMENT / ISOLATION (confirmed good)
- A bare embedded KiritoVM has NO `parallel` (requireDispatcher throws a clear error); only the dispatcher-configured VMs get it. Confirmed.
- Worker VMs share nothing: args/results cross only as `dump` blobs (packArgs/unpackArgs, Task.result). The only shared C++ state is the internally-mutex-guarded dispatcher primitives (shared_ptr<ConcurrentQueue/Lock/…>), addressed cross-VM BY ID — no Kirito Object/Handle/arena is shared. Verified spawn/join, Queue round-trip, and shutdown-aborts-Queue-blocked-worker all behave.
- configureVM snapshots libPaths_/maxCallDepth_ under registryMutex_ (dispatcher.hpp:578) — no data race with a concurrent addLibPath (the TSan concern the comment flags is handled).
- Socket lifecycle verified clean: fileno()==-1 after close/detach, use-after-close throws "op: socket is closed", double-detach throws, UDP starttls rejected, family/type/port validated.
