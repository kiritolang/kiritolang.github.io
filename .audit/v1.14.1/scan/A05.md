## Empirical verifications (all PASS)

Confirmed correct end-to-end (each would be a real bug if wrong):

- **INT64_MIN two's-complement edges** (no signed-overflow UB):
  - `-9223372036854775808 // -1` → `-9223372036854775808` (ifloordiv b==-1 special-case, runtime.hpp:115)
  - `-9223372036854775808 % -1` → `0` (imod b==-1 special-case, :121)
  - `abs(-9223372036854775808)` → wraps to itself (abs via `wsub(0,v)`, :3027)
  - `-(-9223372036854775808)` → wraps (IntVal::unary via wsub, :273)
  - `hex(-9223372036854775808)` → `-0x8000000000000000` (radix magnitude via `0ull - (uint64_t)v`, :3419)
  - `divmod(INT64_MIN, -1)` → `[INT64_MIN, 0]`
- **Modular pow __int128 widening** (:3480-3490): `pow(9223372036854775807, 2, 4611686018427387847)`
  and `pow(-9223372036854775808, 2, 1000000007)` both match CPython bit-for-bit. Modulus-near-INT64_MAX
  path is also covered by existing tests (`pow(9223372036854775805, 2, 9223372036854775807)`).
- **Float↔Integer exact `==`/hash consistency**: `{2:...}[2.0]` collapses to one key (FloatVal::hash
  int-fold, :283); `2^53+1` (Integer) and `2^53` (Float) stay distinct keys (intFloatEqual avoids the
  lossy double round-trip, compareIntFloat :89).
- **Reflected `==`/`!=` symmetry** (the invariant the task flags): verified both operand orders
  dispatch the same instance dunder, including left-instance-without-`_eq_`/right-instance-with-`_eq_`
  falling through to `kiEquals`'s `viaEq(b,a)` (:1605-1606). Arithmetic/ordering correctly does NOT
  reflect (documented invariant, :2231-2235). No accidental double-dispatch.
- **NaN total ordering** for sort/min/max: `kiLessThan` imposes NaN-sorts-largest strict-weak-order
  (:405-411) so `sorted([3,nan,1,2])` → `[1,2,3,nan]`, `min` → real minimum, `max` → nan. No
  std::sort UB.
- **Keyword binding**: out-of-order (`g(z=3,x=1,y=2)`), default referencing an earlier already-bound
  param (`Function(a, b=a+1, c=b*2)`), and mixed positional+keyword all bind correctly; defaults are
  evaluated in the call scope after earlier params are `env.define`d (:2087-2100).
- **Resource guards**: `range(0, 10**18)` and `range(INT64_MIN, INT64_MAX-...)` (≈2^64 span) both
  throw "range too large" (unsigned span/count arithmetic, :3124-3134) instead of OOMing; String/List
  `*` and padding are `kMaxRepeat`-bounded.
- **Native arity errors are clean** (no read-past-span UB): `map(f)`, `abs(1,2)`, `len()`, `len(1,2)`,
  `chr(1,2)`, `isinstance(1)`, `round(1.5,2,3)` all raise catchable errors via bindArgs/explicit checks.
- **Recursion guards**: deeply nested-list `<` is bounded by `EqualsGuard` (caught, no C++ stack
  overflow); `_iter_`/`_str_` self/cycle returns are depth-bounded (:1908, :1932).

## INFO / notes (no action required)

- **A05-INFO-1** — `String.format()` rejects a `:format-spec` (`"{0:05d}".format(3)` throws
  "format field must be an index", runtime.hpp:1391-1392). This is **intentional and documented**:
  `docs/pages/09-types.md:150` explicitly states named `{x}` fields and `:format-spec` are *not*
  supported by the method — use an f-string or the `format()` builtin. Not a bug; noted only so a
  future auditor doesn't re-flag it.
- **A05-INFO-2** — `sum()` maintains a parallel `double f` accumulator even while the running total is
  still all-Integer (:3161-3167). This is **necessary, not dead code**: when a Float first appears
  mid-iterable the code switches to `f`, which must already hold the sum-so-far, avoiding a second
  pass. Correct; the Integer path uses `wadd` (documented int64 wrap).
- **A05-INFO-3** — Coverage is thorough. INT64_MIN arithmetic (`test_adversarial.cpp`,
  `r6_language.ki`), 3-arg pow with near-INT64_MAX modulus (`r4_numbers.ki` et al.), range-too-large
  (`errors/range_too_large.experr`), and starred unpack (`r7/r8_language.ki`) all have pinned tests.
  No test gap identified for the paths audited here.
