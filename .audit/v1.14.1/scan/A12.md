## Findings

### A12-1 (low) — deflate::compress uses `int` for window positions; >2 GiB input can wrap to a negative index (latent OOB read)

- **file**: `src/kirito/deflate.hpp:91,111,121,142,150` (`compress`)
- **severity**: low (latent; hard to reach in practice)
- **scenario**: `compress` builds `std::vector<int> prev(n, -1)` and stores positions with
  `static_cast<int>(i)` into `head[]`/`prev[]` where `i` is a `std::size_t` source position. For an
  input string longer than `INT_MAX` (~2 GiB) the cast wraps to a negative value; a later
  `j = head[h]` / `j = prev[j]` then yields a negative `j`, and `prev[j]` / `in[j + len]` reads out of
  bounds. `zlib.compress`/`deflate`/`gzip.compress` place no size guard, so the only thing standing
  between a >2 GiB argument and the OOB is the `prev(n,-1)` allocation (≈8 GiB) likely throwing
  `std::bad_alloc` first (caught by a bare `catch` as a std::exception). On a large-RAM host the
  allocation can succeed and the wraparound becomes reachable.
- **triggerability**: low — needs a single >2 GiB String/Bytes fed to a compressor. No malformed input
  needed, but such inputs are unusual and the allocation is a partial backstop.
- **proposed fix**: widen the hash-chain positions to `int64_t` (or `std::size_t` with a sentinel),
  or add an explicit `if (n > <cap>) throw` size guard at the top of `compress` (a few hundred MiB,
  mirroring `kMaxInflateOut`). The inflate side is already capped; the compress side is not.
- **coverage gap**: no test compresses a >2 GiB input (understandably); a cheap guard + a
  "compress input too large" error test would document the boundary.

### A12-2 (low) — AES-GCM `aesdecrypt` accepts a truncated authentication tag (1..16 bytes)

- **file**: `src/kirito/stdlib_crypto.hpp:259` (`if (tag.empty() || tag.size() > 16) throw ...`)
- **severity**: low (footgun, not a default-path defect)
- **scenario**: `aesencrypt` always emits a full 16-byte (128-bit) tag, but `aesdecrypt` accepts any
  tag of 1..16 bytes and passes it straight to `EVP_CTRL_GCM_SET_TAG`. If application code ever
  verifies against a caller-supplied short tag (e.g. a 1-byte tag), forgery resistance collapses to
  ~1/256. NIST SP 800-38D recommends a 96-bit (12-byte) minimum for general use.
- **triggerability**: low — only when the application deliberately supplies a sub-16-byte tag; the
  library's own encrypt never produces one.
- **proposed fix**: require `tag.size() == 16` (matching what encrypt produces), or enforce a
  documented minimum such as 12 bytes, with a clear error. Keeps the API foolproof.
- **coverage gap**: no test asserts the tag-length policy; add one either way.

### A12-3 (informational) — BigInt/BigInt true division of huge equal operands yields NaN, not 1.0

- **file**: `src/kirito/stdlib_int.hpp:601-603` (`BinOp::Div` → `toDouble(val)/toDouble(b)`)
- **severity**: informational (documented-lossy edge)
- **scenario**: `big(10)**400 / big(10)**400` gives `nan` because each operand's `toDouble` overflows
  to `inf` and `inf/inf == nan`. `huge / big(2)` gives `inf`. This is consistent with the documented
  rule that BigInt `/` is Float true-division "lossy beyond double range exactly as Integer/Integer",
  but unlike native Integer (which always fits int64 so never overflows to inf), BigInt can produce
  `inf`/`nan` — a mild surprise where an exact ratio (1.0) was expected.
- **triggerability**: only with operands whose magnitude exceeds ~1.8e308.
- **proposed action**: none required (by design). Optionally document that BigInt `/` can yield
  `inf`/`nan` for out-of-double-range operands, or offer an exact `//` (which already works) as the
  recommended path. Not a bug — recording so it is not re-flagged as one.
