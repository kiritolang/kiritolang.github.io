## A04-1 [VERIFIED-CLEAN] Control-flow + slot machinery is correct and GC-safe

Empirically confirmed correct (debug `ki`) and GC-safe (embedded VM, `setGcThreshold(1)`):

- break / continue / return inside `try/finally`, `with`, nested loops, and combinations — finally
  and `with._exit_` run on EVERY exit path, innermost-first; loop cursors stay balanced.
- The A04-1 (v1.14) return-value-parking across a crossed `finally`/`with` (`$retN` hidden slot) and
  the exception-parking across a `finally` (`$excN` hidden slot) both hold at a clean operand height —
  a `break`/`continue` inside a crossed finally does not Pop the return value / exception.
- `return`/`break`/`continue` inside a `catch` handler runs the re-armed finally (SetupBlock/PopBlock
  vs. compile-time `CFrame` stay in LIFO sync on every path, incl. no-match Reraise and Lfin).
- Exception thrown inside a `finally` replaces the in-flight one; `return`/`break`/`continue` inside a
  `finally` swallows a pending exception (Python semantics) — verified.
- `switch` fast path (O(1) `SwitchDispatch`) and O(n) `SwitchMatch` fallback: subject is popped before
  any arm body, so a `break`/`continue` targeting the enclosing loop is balanced; no-match/no-default
  falls through cleanly; negated-literal case keys dedup correctly; multi-value cases evaluate L→R.
- Slot-addressed locals (`LoadLocal`/`StoreLocal`/`AssignLocal`), the numeric fast path, constant
  dedup (scalars keyed by exact bits/value, non-scalars never deduped): all correct.
- GC-safe under collect-on-every-alloc: heavy loops building Dict/Set/List, f-strings with specs,
  slicing, closures, user `_add_` in a loop, `return`/exception parking through `finally`/`with` with
  allocations in the cleanup — all pass with no dangling handle (`gg1/gg2/gg3`, `q1/q2/q4/q5`, `z*`).

Repro harness + scripts: `/tmp/.../scratchpad/a04/` (gctest_dbg.cpp sets `setGcThreshold(1)`).

## A04-X1 [HIGH — OUT OF SCOPE: stdlib native rooting, NOT the engine]

Title: stdlib native methods dangle under a low pinned `setGcThreshold` (io stream write, BytesIO
default arg, Random.*), while the execution engine does not.

Where: NOT in `compiler.hpp`/`bytecode.hpp`/`bytecode_vm.hpp`. Traces to stdlib native code / the GC
root set — candidates: `stdlib_io.hpp` (`streamWriteTo` @494, `print`/`write` kwfns @571/@615, the
StdStream members set up @535-540), `stdlib_random.hpp`, native default-arg handling in `native.hpp`
(`NativeFunction::bindArgs`), and `ModuleValue::children()` / the module-cache root scanning in
`vm.hpp`/`runtime.hpp`.

Scenario (embedded VM, `vm.setGcThreshold(1)` — a configuration the codebase documents as supported:
vm.hpp:161 "tests use setGcThreshold(1) to collect on every allocation"):
```
var io = import("io")
io.print(1)          # -> KiritoError "dangling handle (stale generation)"
io.BytesIO()         # -> dangling handle (uses the setup-time default arg initial="")
var g = import("random").Random(7)
g.randint(1, 10)     # -> dangling handle
```
`io.print()` dangles even with ZERO args (nothing in the call allocates), and `type(io.stdout)`
dangles — so the swept object is the module's StdStream member / a setup-time default-arg handle,
dereferenced when a GC fires mid-native-call. DateTime.iso / Regex.search / Matrix.transpose /
math.sqrt / json.dumps / user functions / builtins (min/sorted/len/range) / String methods are all
GC-safe under the same threshold — the fault is specific to those native impls, not systemic and not
the engine.

Triggerability: reproducible 100% under a pinned `setGcThreshold(1)`. NOT reachable at the default
adaptive threshold (floor 65536): a program that forces >300k allocations (many real GCs) then calls
io.print / BytesIO / Random works fine (real1.ki). So practical impact is limited to embedders who
pin a low GC threshold — but it falsifies the documented invariant that `setGcThreshold(1)` is safe.

Idiomatic fix (for the stdlib/GC auditor, not A04): root the transient stream/module-member/default
handles across allocating native bodies (RootScope), and/or ensure `ModuleValue::children()` traces
its `members` and native default-arg handles are pinned as permanent GC roots at registration.
