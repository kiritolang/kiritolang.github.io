# A16 — Test-Coverage Gap Map (round v1.14.1)

Scanner **A16**. Mandate: map the ENTIRE public surface (builtins, stdlib module functions/methods/
attributes, native object types, operators, language features) against EXISTING tests; produce a
prioritized list of untested / under-tested surface.

**Read-only.** Gaps only — the triager writes tests later. Severity: high = core behaviour, zero
tests / medium = happy-path only, no edge/adversarial / low = minor.

(298), `tools/tests/scripts/*.ki`+`.expected` (365), `docs/pages/*.md`, big-project/example self-tests.

## crypto (NEW 1.14.0) — OpenSSL-gated

Existing: `test_crypto.cpp` (TLS-only), `spec_crypto.ki` (branches on `.enabled`). Good coverage of
happy paths: AES-GCM KAT + round-trips + tamper + fuzz, RSA sign/verify/OAEP, EC sign/verify, X.509
parse, and several adversarial cases. Debug build has TLS on. Gaps (all confirmed by probe):

- **A16-1** (crypto.aesdecrypt / tag length — medium): the `tag must be 1..16 bytes` guard is
  UNTESTED. Neither an over-16-byte tag nor an empty tag (`Bytes(0)`) is exercised. Probe:
  `aesdecrypt(k, ct, iv, tag + Bytes([0]))` throws `aesdecrypt: tag must be 1..16 bytes`. Add both a
  too-long tag and an empty-tag case to `test_crypto.cpp` / `spec_crypto.ki`.
- **A16-2** (crypto RSA/EC unknown-algo — medium): `mdByName` throws `crypto: unknown hash '...'`
  for an algo outside sha1/256/384/512, but no test passes a bad `algo=` to rsasign/rsaverify/ecsign/
  ecverify. Probe `rsasign(priv,"m","md5")` throws as expected. Add `rsasign(..., "md5")` +
  `ecsign(..., "whirlpool")` throwing cases.
- **A16-3** (crypto EC non-sha256 algo + wrong-key reject — medium): RSA is tested with sha512 and a
  wrong key, but EC only ever uses default sha256 and never a *different* EC key. Add
  `ecsign(priv,"m","sha512")` round-trip and `ecverify(otherkey.public, "m", sig) == False`.
- **A16-4** (crypto sha1 digest path — low): `mdByName` supports `"sha1"` but no test uses it. Add a
  `rsasign(..., "sha1")` round-trip so the branch is covered.
- **A16-5** (crypto keyword-argument uniformity — medium): every crypto fn is signatured (accepts
  kwargs) but NO test calls any of them by keyword. Given the round's kwargs-uniformity mandate, add
  e.g. `aesencrypt(key=..., plaintext=..., iv=..., aad=...)`, `rsasign(private_pem=..., message=...)`.
  Probe confirms kwargs work.
- **A16-6** (crypto String-vs-Bytes inputs — low): `argStringOrBytes` accepts a String key/iv/aad but
  tests always pass Bytes (except plaintext). Add a case passing a String key/iv to aesencrypt and a
  Bytes message to rsasign to pin the dual-type acceptance.
- **A16-7** (crypto.rsagenerate upper bound — low): `bits > 16384` throws (`[512,16384]`) but only the
  too-small `rsagenerate(64)` is tested. Add `rsagenerate(20000)` throws. Also `rsagenerate(511)`
  (just under min) untested.
- **A16-8** (crypto.x509parse field coverage — low): `not_before` and `serial` value/format are only
  weakly checked (len>0). `subject`/`issuer` checked by substring. Add an assertion on `not_before`
  being a non-empty String and that a cert with NO SANs yields `sans == []` (empty-SAN branch of the
  extension loop is untested — the loop's `X509_get_ext_d2i` returning null path). Would need a
  second no-SAN cert fixture.
- **A16-9** (crypto.aesencrypt empty-plaintext explicit — low): the fuzz can hit len-0 plaintext by
  chance (`randombelow(200)`), but there is no *deterministic* empty-plaintext round-trip asserting
  ciphertext is empty and the tag still authenticates. Add `aesencrypt(k,"",n)` → empty ct, decrypt
  back to `""`.
- **A16-10** (crypto.enabled=False branch under TLS build — n/a low): the `spec_crypto.ki` non-TLS
  branch only runs under asan/tsan. Fine — noted for completeness; the "requires KIRITO_ENABLE_TLS"
  messages ARE asserted there.

## int / BigInt (NEW 1.14.0)

Existing: `test_bigint.cpp` (thorough — KAT, differential fuzz vs native Integer, reflected-op
invariant, cross-type eq/hash, serde, adversarial) + `spec_bigint.ki`. One of the best-covered new
modules. Gaps (probed):

- **A16-11** (BigInt unary negation `-b` — medium): `-int.BigInt(5)` → `-5` works but is UNTESTED
  (no `_neg_` coverage for BigInt). Add `String(-int.BigInt(5)) == "-5"` and `-int.BigInt(0) == 0`.
- **A16-12** (BigInt methods `.isprime()` / `.isprobableprime()` — medium): only the *module* forms
  `int.isprime(n)` are tested; the BigInt *method* forms `int.BigInt(97).isprime()` /
  `.isprobableprime()` are untested (both work per probe). Add method-form assertions.
- **A16-13** (reflected-op invariant incomplete — medium): only `+`, `*`, `<` are tested throwing on
  the RIGHT (`3 + BigInt`). The remaining binary ops `-`, `/`, `//`, `%`, `**`, `<=`, `>`, `>=` with a
  native int on the left and BigInt on the right are NOT tested. Probe: `3 - int.BigInt(2)` throws.
  Add the full set so the invariant is pinned for every operator, not a sample.
- **A16-14** (BigInt `**` negative exponent → Float — medium/latent-behavior): `int.BigInt(2) ** -1`
  returns `0.5` (Float), mirroring native `2 ** -1`. This is UNTESTED and non-obvious (a negative
  exponent silently leaves BigInt-land for Float). Pin it: `int.BigInt(2) ** -1 == 0.5` and
  `type(...) == "Float"`. Verify it's intended (consistent with true-division rule) — likely correct.
- **A16-15** (int.modpow negative exponent — low): `int.modpow(2,-1,5)` throws `modpow: negative
  exponent` (probed). Untested. Add the throwing case. (Note: Python's pow allows a negative exp with
  a coprime base via modinv; Kirito rejects it — worth pinning the rejection as intentional.)
- **A16-16** (int.abs / int.big — low): module fns `int.abs(b)` and the `int.big(x)` alias are never
  exercised (only `int.BigInt`). `int.abs(int.BigInt(-42))==42` and `int.big(123)` work. Add both.
- **A16-17** (int.randomprime rounds + bad bits — low): `randomprime(bits, rounds)` is only called
  with default rounds; the `rounds=` argument is untested, and no test checks tiny/invalid `bits`
  (e.g. `randomprime(1)` or `randomprime(0)` behaviour). Add a rounds= call and a bad-bits case.
- **A16-18** (BigInt large-value hashing — low): hash-equality is only checked for small values (7,
  42) that fit int64. A BigInt exceeding int64 that is NOT equal to any native Integer should still
  hash + key a Set/Dict consistently with itself. Add `{int.pow(10,50), int.pow(10,50)}` → len 1.
