# A01 — Lexer scan (src/kirito/lexer.hpp) — v1.14.1

Scope: tokenization only — string literals (plain/raw/f-string, single/double/triple, escapes),
numeric literals (decimal/hex/octal/binary/float/scientific), indentation (INDENT/DEDENT, tabs vs
spaces, CRLF normalization), comments, operators, soft keywords. Focus on `Lexer::stringLiteral`
and `Lexer::number` (the "intLiteral").

Method: deep read + adversarial inputs run against `./build-debug/ki`. READ-ONLY on the codebase.

Findings appended continuously below.

---

## Summary verdict

The lexer has been through five prior audit rounds and is genuinely hardened. All manufactured
adversarial inputs were handled without crash, OOB read, or UB:

- Numeric overflow: `99999999999999999999999999`, `0xFFF...`, `0b1...` (70 bits), 200k-digit literal
  all wrap to a well-defined two's-complement value (documented). `1e400` -> `inf`; `5e-324` decodes
  the subnormal via `parseDouble` (not `std::stod`, avoiding the underflow trap).
- Base-prefix with no digits (`0x`/`0o`/`0b`) and bad base digits (`0o8`) throw a clean
  `invalid numeric literal` (covered by `surf_lex_08/09/10`).
- All unterminated string forms (single/triple/raw/f) throw `unterminated ...` with correct spans;
  escapes (`\xHH`, bad `\q`, dangling `\`, one-digit `\x`) all report clean errors
  (`surf_lex_01-07`, `unterminated_*`, `bad_escape`).
- Embedded NUL byte in source is preserved as a real character (`atEnd()` vs `peek()=='\0'` split
  works); does NOT prematurely terminate a literal.
- CRLF and lone-CR normalization correct; tab/space ambiguity detection fires correctly.
- f-string same-quote-inside-braces gives a clean `unmatched '{'` error (documented limitation).

**No memory-safety, integer-overflow/UB, or mutable-global/static findings.** All lexer state is
per-instance (`src_`, `pos_`, `line_`, `col_`, `indent_`, `parenDepth_`); `normalizeNewlines` is a
pure static function. `KiritoVM` encapsulation is intact for this subsystem.

Only two LOW cleanliness/robustness notes and a minor coverage note follow.

## A01-1 — Dead `\r` branch in the main tokenize loop (post-normalization)
- **File:** src/kirito/lexer.hpp:78 (`if (c == ' ' || c == '\t' || c == '\r')`)
- **Severity:** LOW (dead code / DRY-cleanliness; not a bug)
- **Analysis:** `normalizeNewlines` (lines 56-64) rewrites **every** `\r` in the source to `\n`
  (CRLF collapses, lone CR becomes LF), so `src_` provably never contains a `\r` byte by the time
  `tokenize()` scans it. The `c == '\r'` disjunct on line 78 is therefore unreachable. Harmless, but
  it is dead defensive code that can mislead a reader into thinking a raw `\r` can survive into the
  token loop (it cannot). The comment block at lines 49-52 already documents that `\r` is eliminated
  up front, which makes the branch contradictory with its own stated invariant.
- **Triggerable:** No — cannot ever execute.
- **Proposed fix:** Drop the `|| c == '\r'` from line 78 (leave a one-line comment noting `\r` is
  normalized away), so the invariant "no `\r` reaches the scanner" is expressed once and enforced,
  not silently re-handled.

## A01-2 — `tokenize()` is not idempotent: positional state not reset
- **File:** src/kirito/lexer.hpp:67-71 (only `indent_` and the local `lineStart` are reset)
- **Severity:** LOW (latent robustness; not triggerable in current code)
- **Analysis:** `tokenize()` resets `indent_ = {{0,0}}` and the local `lineStart`, but it does NOT
  reset the scan cursor `pos_`, `line_`, `col_`, or `parenDepth_`. A second call to `tokenize()` on
  the same `Lexer` instance would start with `pos_ == src_.size()` and immediately fall through the
  `while (pos_ < src_.size())` loop, returning a degenerate `[EndOfFile]` (or, if a `parenDepth_`
  were left non-zero by malformed input, subtly wrong layout on a re-run). Every current call site
  constructs a fresh `Lexer` and calls `tokenize()` exactly once, so this is not reachable today —
  but the method reads as if it can be called repeatedly (it re-initializes `indent_`), which is a
  half-done reset that will bite a future caller.
- **Triggerable:** No (single-use contract, undocumented).
- **Proposed fix:** Either reset all scan state at the top of `tokenize()` (`pos_ = 0; line_ = 1;
  col_ = 1; parenDepth_ = 0;` alongside the existing `indent_` reset), making it genuinely
  re-runnable, or remove the partial `indent_` re-init and document `tokenize()` as single-use. The
  former is the idiomatic, surprise-free choice.

## A01-3 — Minor test-coverage note (not a defect)
- **Observation:** Error tests exhaustively cover base-prefix-with-no-digits (`0x`/`0o`/`0b`,
  surf_lex_08/09/10) but there is no explicit `.experr` for a base prefix followed by an
  **out-of-alphabet digit** (`0o8`, `0b2`, `0xG`). These all correctly throw the same
  `invalid numeric literal '0o'`/`'0b'`/`'0x'` (verified empirically: `0o8` -> error), because the
  scanner stops at the first non-base-digit and finds zero base digits. Coverage is *equivalent*
  via the empty-prefix tests, so this is a very-low-value gap; noting it only for completeness. No
  action strictly required.

## Doc-mismatch check
- Checked CLAUDE.md's lexer claims (universal newlines, `\xHH` = U+00HH as UTF-8, raw/f/rf spellings,
  base prefixes, `case`/`default`/`switch` soft-keyword handling, indentation two-measure rule)
  against the implementation and empirical behavior: **all accurate.** Leading-zero decimal (`0123`
  -> 123, `0128` -> 128) is accepted as decimal (not octal, not rejected); CLAUDE.md only documents
  `0o17` for octal and does not claim leading-zero decimals are rejected, so this is not a
  documented-behavior mismatch — just an undocumented leniency (consistent with the codebase's
  general leniency stance; no corruption/ambiguity). Not flagged as a bug.
